# Seata 分布式事务

## 是什么

Seata 是处理跨服务、跨数据库事务一致性问题的分布式事务框架，是国内 Spring Cloud 项目最常见的分布式事务方案。

## 同类组件与方案

| 方案 | 说明 |
| --- | --- |
| `Seata` | 分布式事务框架 |
| `RocketMQ Transaction Message` | 事务消息，详见 [RocketMQ](./rocketmq) |
| `本地消息表` | 最终一致性方案 |
| `TCC` | Try Confirm Cancel 模式 |
| `Saga` | 长事务补偿模式 |

## 重要原则：不要滥用

分布式事务复杂度高、对性能有影响，不能滥用。优先级建议：

```
优先本地事务
其次消息最终一致性
复杂场景再考虑 Seata
```

这里的「本地事务」指单库内的 [Spring 事务](/spring-core/transaction)。只有当一致性需求跨越了多个库或多个服务、本地事务管不到时，才需要往下考虑最终一致性和 Seata。

## 典型场景

一个下单流程可能涉及：

```
创建订单
扣减库存
扣减余额
生成支付记录
```

如果这些操作分布在多个服务和多个数据库中，就需要考虑分布式事务或最终一致性。如果都在同一个库，用本地事务即可，不需要 Seata。

## 选型建议

- 大多数场景优先用本地事务或基于 [消息队列](./) 的最终一致性。
- 确实需要强一致、跨库跨服务时，再引入 Seata。
