# 灰度发布与服务网格

把新版本逐步放量、按流量比例验证、出问题能快速回滚，是企业项目从"能发布"到"能安全发布"的关键。大型项目还会用服务网格统一治理服务间流量。

## 常见组件

| 组件 | 类别 | 主要作用 |
| --- | --- | --- |
| Istio | 服务网格 | 服务间流量治理、mTLS、可观测性。 |
| Linkerd | 服务网格 | 轻量级服务网格。 |
| Argo Rollouts | 灰度发布 | 蓝绿、金丝雀等渐进式发布。 |
| Gateway + Nacos | 简单灰度 | 基于网关和元数据的灰度路由。 |
| Kubernetes Ingress | 流量入口 | 集群流量入口。 |
| Envoy | 高性能代理 | 服务网格数据面代理。 |

## 大型项目的发布与治理需求

```
蓝绿发布
金丝雀发布
灰度发布
流量镜像
mTLS
服务间流量治理
```

## 选型建议

- 普通中小项目可以先不用服务网格，复杂度比较高。简单灰度用 [Gateway](/framework/gateway) + [Nacos](/framework/nacos) 元数据路由即可。
- 需要蓝绿/金丝雀发布：`Argo Rollouts`。
- 大型项目、服务间需要 mTLS 和精细流量治理：再考虑 `Istio`。
- 服务网格依赖 [Kubernetes](./kubernetes)，应在容器编排成熟后再引入。
