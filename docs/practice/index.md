# 工程实践

> 本栏目内容不来自《Spring 技术套件》原文档，是为补齐「Java 后端全技术」而新增的工程实践部分。

掌握了语言、框架和中间件之后，真正决定项目质量的往往是这些"看不见"的工程能力：构建怎么管、代码怎么组织、SQL 怎么写好、接口怎么定规范。这一层不引入新组件，而是讲清楚如何把已有技术用好。

## 本栏目内容

- [Maven 与依赖管理](./maven)：多模块、依赖仲裁、BOM、profile 与常用插件。
- [设计模式与架构](./design-pattern)：常用设计模式、分层架构、DDD 与整洁架构思想。
- [SQL 优化与索引](./sql-optimization)：索引原理、执行计划、事务隔离与慢查询排查。
- [API 设计规范](./api-design)：RESTful 风格、统一返回、全局异常、幂等与分页。

## 为什么单独成栏

| 问题 | 现象 | 这一层解决什么 |
| --- | --- | --- |
| 依赖混乱 | 版本冲突、jar 包冲突、打包失败 | 用 Maven BOM 和依赖仲裁统一管理 |
| 代码难维护 | 改一处崩一片、逻辑散落各处 | 用设计模式和分层架构约束结构 |
| 数据库慢 | 接口越来越慢、CPU 飙高 | 用索引和 SQL 优化解决根因 |
| 接口不统一 | 每个人返回格式不同、前端难对接 | 用统一规范约定返回、异常、分页 |

## 实践原则

```
先让代码能跑
再让代码可读
再让代码可测
最后才追求可扩展
```

不要一上来就套用复杂的设计模式或 DDD，过度设计和不设计同样有害。规范的价值在于团队协作时减少沟通成本，规模越大收益越明显。
