# SQL 优化与索引

> 本页内容为新增的工程实践部分，不来自原《Spring 技术套件》文档。

## 是什么

[数据存储](/storage/)栏目讲了"用 MySQL + MyBatis-Plus"，本页讲"怎么用好"——索引原理、SQL 优化、事务隔离和锁，是后端面试和实战的高频内容。

## 索引原理

MySQL（InnoDB）索引基于 B+ 树：

- 数据按主键有序组织成 B+ 树（聚簇索引），叶子节点存整行数据。
- 普通索引（二级索引）叶子节点存主键值，查非索引列需要"回表"。
- B+ 树高度通常 3~4 层，意味着一次查询最多几次磁盘 IO。

```
覆盖索引：查询列都在索引中，无需回表，性能最好
最左前缀：联合索引 (a,b,c) 能命中 a / a,b / a,b,c，但命中不了 b 或 c
```

## 索引失效的常见场景

```
对索引列做函数 / 运算：WHERE YEAR(create_time) = 2026
隐式类型转换：字符串列用数字查 WHERE phone = 138...
前导模糊查询：LIKE '%abc'
联合索引不满足最左前缀
OR 连接非索引列
```

## SQL 优化思路

排查慢查询的标准流程：

1. 开启慢查询日志，定位慢 SQL。
2. 用 `EXPLAIN` 看执行计划，重点关注 `type`、`key`、`rows`、`Extra`。
3. `type` 至少要到 `range`，避免 `ALL`（全表扫描）。
4. `Extra` 出现 `Using filesort` / `Using temporary` 通常需要优化。

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status = 1;
```

常见优化手段：

```
加合适的索引（联合索引、覆盖索引）
只查需要的列，避免 SELECT *
大分页用游标 / 延迟关联，避免 LIMIT 1000000, 20
拆分大事务、大 SQL
合理使用缓存，减少数据库压力
```

## 事务隔离级别

| 级别 | 脏读 | 不可重复读 | 幻读 |
| --- | --- | --- | --- |
| 读未提交 | 可能 | 可能 | 可能 |
| 读已提交 | 不会 | 可能 | 可能 |
| 可重复读（MySQL 默认） | 不会 | 不会 | 基本不会（间隙锁） |
| 串行化 | 不会 | 不会 | 不会 |

InnoDB 在可重复读级别通过 MVCC + 间隙锁，基本解决了幻读。

## 锁

```
行锁    InnoDB 默认，基于索引；查不到索引会退化成表锁
间隙锁  锁住索引区间，防止幻读
意向锁  表级，标记表中有行锁
乐观锁  靠版本号 / CAS，适合冲突少的场景（MyBatis-Plus @Version）
悲观锁  SELECT ... FOR UPDATE，适合冲突多的场景
```

## 选型建议

- 索引不是越多越好，写多读少的表要控制索引数量。
- 高并发更新场景优先考虑乐观锁，减少锁等待。
- 单表数据量过大（千万级以上）再考虑[分库分表](/storage/sharding)，不要过早优化。
- 复杂查询、聚合分析迁移到 [Elasticsearch](/storage/elasticsearch) 或分析型数据库，不要硬压在 MySQL 上。
