# 设计模式与架构

> 本页内容为新增的工程实践部分，不来自原《Spring 技术套件》文档。

## 是什么

设计模式是针对常见设计问题沉淀出来的可复用解决方案，重点不是背名字，而是识别变化点、降低耦合、提高扩展性。架构风格则决定系统整体代码如何组织，例如分层架构、DDD、整洁架构等。

在 Java 后端项目中，设计模式更多体现为：如何创建对象、如何扩展行为、如何解耦模块、如何组织流程、如何让框架能力在不侵入业务代码的前提下工作。

## 什么时候该用设计模式

设计模式适合用在“变化频繁、分支复杂、重复逻辑多、模块耦合高”的地方。简单 CRUD 不需要强行套模式，优先保证代码清晰、命名准确、分层合理。

| 场景 | 更适合的思路 |
| --- | --- |
| 大量 `if-else` 按类型选择逻辑 | 策略模式、工厂模式 |
| 多个处理步骤按顺序执行 | 责任链模式、模板方法模式 |
| 需要统一增强日志、事务、权限 | 代理模式、装饰器模式 |
| 对象构造参数多且组合复杂 | 建造者模式 |
| 新旧接口不兼容 | 适配器模式 |
| 事件通知多个后续动作 | 观察者模式、事件驱动 |

## 设计原则

设计原则是设计模式的基础。模式只是手段，原则才是判断代码是否可维护的依据。

### 单一职责原则

一个类、方法或模块只负责一类变化原因。比如导出报表时，不要让一个类同时负责查询数据、组装 Excel、上传文件和发送通知，可以拆成查询服务、导出器、文件服务和通知服务。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 避免一个类承担过多职责，导致修改互相影响。 |
| 典型场景 | Service 过大、工具类过大、方法过长。 |
| 注意点 | 拆分不是越细越好，要围绕变化原因拆。 |

### 开闭原则

对扩展开放，对修改关闭。新增功能优先通过新增实现类、配置、策略、插件或事件监听扩展，而不是频繁修改稳定代码。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 降低修改老代码带来的回归风险。 |
| 典型场景 | 支付渠道、优惠规则、消息发送、文件解析。 |
| Java 后端体现 | 接口多实现、策略注册表、Spring Bean 自动注入。 |

### 里氏替换原则

子类应该可以替换父类，并且不破坏原有程序行为。继承不能随意改变父类承诺，例如父类方法保证返回非空，子类不应返回空；父类支持某种输入，子类不应随意拒绝。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 避免继承层级破坏统一语义。 |
| 典型场景 | 基类模板、抽象服务、统一接口实现。 |
| 注意点 | 如果子类无法保持父类语义，优先用组合。 |

### 接口隔离原则

接口要小而专，不要让实现类被迫实现自己不需要的方法。一个大而全的接口容易产生空实现、异常实现和无效依赖。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 降低接口使用方和实现方的无关依赖。 |
| 典型场景 | 大服务接口、统一工具接口、多能力混合接口。 |
| 注意点 | 接口拆分要按调用方需求拆，而不是机械拆方法。 |

### 依赖倒置原则

高层模块依赖抽象，不直接依赖具体实现。业务代码应该依赖接口或抽象能力，具体实现交给 Spring 注入或工厂选择。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 降低业务逻辑和具体实现之间的耦合。 |
| 典型场景 | 支付、存储、消息、第三方接口对接。 |
| Java 后端体现 | `@Autowired` 注入接口、配置切换实现、SPI。 |

### 迪米特法则

也叫最少知道原则，一个对象尽量少了解其他对象内部结构，只和直接协作者交互。不要让业务代码穿透多层对象拿内部字段，避免链式依赖过深。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 降低对象之间的结构耦合。 |
| 典型场景 | 复杂聚合对象、远程接口返回对象、领域模型。 |
| 注意点 | 可以通过 Facade、DTO、领域方法封装内部结构。 |

### 组合复用原则

优先使用组合，而不是继承来复用代码。继承会把父子类强绑定，组合则可以在运行期替换依赖，更适合业务规则变化频繁的系统。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 避免继承层级过深、子类爆炸和父类修改影响面过大。 |
| 典型场景 | 策略组合、能力扩展、装饰增强。 |
| 注意点 | 能用接口 + 组合解决的问题，不要优先设计复杂继承树。 |

## 创建型模式（5种）

创建型模式关注“对象怎么创建”，目标是隐藏创建细节、控制创建过程、降低调用方和具体类之间的耦合。

### 单例模式

单例模式保证一个类在应用生命周期内只有一个实例，并提供统一访问入口。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 避免重复创建全局对象，统一管理共享资源。 |
| 典型场景 | 配置对象、连接池、线程池、运行时上下文。 |
| Java 后端体现 | Spring Bean 默认单例、`Runtime`、全局配置组件。 |
| 注意点 | 懒加载单例要考虑线程安全；双重检查锁需要 `volatile`。 |

### 工厂方法模式

工厂方法模式定义创建对象的统一接口，把具体对象创建延迟到具体工厂中。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 调用方不直接依赖具体类，新增产品时减少对调用方修改。 |
| 典型场景 | 日志输出、文件解析、消息发送、不同渠道客户端创建。 |
| Java 后端体现 | `FactoryBean`、日志框架创建 Appender、业务工厂类。 |
| 注意点 | 产品类型很多时类数量会增加，需要控制工厂层级。 |

### 抽象工厂模式

抽象工厂模式用于创建一组相关或相互依赖的产品族，而不是创建单个对象。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 保证同一产品族内多个对象成套创建和使用。 |
| 典型场景 | 多环境组件、多数据库方言、多品牌产品族。 |
| Java 后端体现 | `BeanFactory` 抽象、数据源工厂、跨环境组件工厂。 |
| 注意点 | 新增产品族容易，新增产品类型会影响所有工厂实现。 |

### 建造者模式

建造者模式把复杂对象的构建过程拆成多个步骤，调用方通过链式或分步方式设置参数，最后统一构建对象。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 避免构造方法参数过多、可选参数混乱、对象创建语义不清。 |
| 典型场景 | DTO、查询条件、配置对象、复杂请求参数。 |
| Java 后端体现 | Lombok `@Builder`、`StringBuilder`、`BeanDefinitionBuilder`、`SqlSessionFactoryBuilder`。 |
| 注意点 | 必填字段要在构建前校验，避免构建出半初始化对象。 |

### 原型模式

原型模式通过复制已有对象创建新对象，适合对象创建成本高或新对象和旧对象高度相似的场景。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 减少复杂对象重复初始化成本。 |
| 典型场景 | 模板配置复制、对象快照、批量创建相似对象。 |
| Java 后端体现 | Spring Bean 的 prototype 作用域、对象拷贝工具。 |
| 注意点 | 要区分浅拷贝和深拷贝；`clone()` 可读性和维护性一般。 |

## 结构型模式（7种）

结构型模式关注“对象之间怎么组合”，目标是在不破坏原有对象的前提下扩展能力、适配接口或组织复杂结构。

### 代理模式

代理模式为目标对象提供代理对象，由代理对象控制访问并增强功能。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 在不修改目标对象的前提下增加日志、权限、事务、缓存等能力。 |
| 典型场景 | AOP、RPC、权限控制、事务增强、延迟加载。 |
| Java 后端体现 | Spring AOP、JDK 动态代理、CGLIB、MyBatis Mapper 代理。 |
| 注意点 | JDK 动态代理依赖接口，CGLIB 通过子类增强，`final` 类或方法会受限制。 |

### 装饰器模式

装饰器模式通过包装目标对象动态叠加能力，装饰器和被装饰对象通常实现同一个接口。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 避免为了叠加功能产生大量子类。 |
| 典型场景 | 缓冲、压缩、加密、日志、链式增强。 |
| Java 后端体现 | JDK IO 流、缓存装饰器、MyBatis 缓存装饰器。 |
| 注意点 | 包装层过多会增加调用链复杂度，需要控制层数。 |

### 适配器模式

适配器模式把不兼容接口转换成调用方期望的接口，让旧系统、第三方组件或不同协议能够协同工作。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 解决接口不匹配和旧代码复用问题。 |
| 典型场景 | 第三方接口接入、老系统改造、协议转换。 |
| Java 后端体现 | `InputStreamReader`、Spring MVC `HandlerAdapter`、数组转集合适配。 |
| 注意点 | 适配器应只做转换，不应承载复杂业务规则。 |

### 组合模式

组合模式把对象组织成树形结构，让调用方可以用统一方式处理单个对象和组合对象。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 统一处理树形结构中的整体和部分。 |
| 典型场景 | 菜单树、组织架构、文件目录、规则树。 |
| Java 后端体现 | MyBatis 动态 SQL 的 `SqlNode`、权限菜单树、部门树。 |
| 注意点 | 树层级过深时要关注递归性能和循环引用。 |

### 享元模式

享元模式复用大量相似对象，把可共享的内部状态抽出来，减少内存占用。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 大量重复对象导致内存浪费。 |
| 典型场景 | 缓存池、连接池、线程池、字典对象。 |
| Java 后端体现 | 字符串常量池、`IntegerCache`、连接池思想。 |
| 注意点 | 要区分内部状态和外部状态，避免共享对象被错误修改。 |

### 桥接模式

桥接模式把抽象和实现分离，让两个维度可以独立扩展，避免多维组合导致类爆炸。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 多个维度同时变化时，继承层级膨胀。 |
| 典型场景 | 多渠道消息、多平台适配、多数据库驱动。 |
| Java 后端体现 | JDBC 驱动抽象、消息类型和发送渠道解耦。 |
| 注意点 | 适合两个或多个独立变化维度，不要为简单场景强行桥接。 |

### 外观模式

外观模式为复杂子系统提供统一入口，调用方不需要了解内部多个模块的协作细节。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 降低调用方理解和调用复杂子系统的成本。 |
| 典型场景 | 统一服务门面、复杂流程编排、第三方能力封装。 |
| Java 后端体现 | `ApplicationContext`、业务 Facade、网关服务。 |
| 注意点 | 外观类不要变成上帝类，复杂业务仍应拆到领域服务或应用服务中。 |

## 行为型模式（11种）

行为型模式关注“对象之间如何协作”，目标是组织流程、封装变化、解耦调用和处理复杂业务行为。

### 策略模式

策略模式把一组可替换的算法或业务规则封装成独立策略类，调用方只依赖统一接口，运行时根据条件选择具体实现。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 消除大量 `if-else`，让规则扩展更容易。 |
| 典型场景 | 支付方式、优惠规则、文件解析、消息发送渠道。 |
| Java 后端体现 | Spring 注入 `List<Strategy>` 或 `Map<String, Strategy>`。 |
| 注意点 | 策略过多时需要统一注册、命名、默认兜底和异常处理。 |

后端最常用的写法是让每个策略实现统一接口，再由 Spring 收集所有实现类：

```java
public interface PayStrategy {
    String type();
    void pay(Order order);
}

@Service
public class PayStrategyFactory {
    private final Map<String, PayStrategy> strategies;

    public PayStrategyFactory(List<PayStrategy> list) {
        this.strategies = list.stream()
            .collect(Collectors.toMap(PayStrategy::type, strategy -> strategy));
    }

    public void pay(String type, Order order) {
        PayStrategy strategy = strategies.get(type);
        if (strategy == null) {
            throw new IllegalArgumentException("unsupported pay type: " + type);
        }
        strategy.pay(order);
    }
}
```

新增一种支付方式只需要新增一个策略实现类，不需要改动调用方。

### 观察者模式

观察者模式定义一对多依赖，被观察对象状态变化后通知多个观察者，常用于事件驱动。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 解耦主流程和后续扩展动作。 |
| 典型场景 | 订单支付成功后发消息、送积分、发优惠券。 |
| Java 后端体现 | Spring `ApplicationEvent` / `ApplicationListener`、MQ 消费订阅。 |
| 注意点 | 异步通知要考虑失败重试、幂等和最终一致性。 |

### 模板方法模式

模板方法模式由父类定义固定流程，把可变步骤留给子类实现。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 统一流程骨架，同时允许局部步骤变化。 |
| 典型场景 | 文件解析、数据库操作、导入导出流程。 |
| Java 后端体现 | `JdbcTemplate`、`RestTemplate`、`HttpServlet`、MyBatis `BaseExecutor`。 |
| 注意点 | 模板方法依赖继承，变化维度过多时可考虑策略模式。 |

### 责任链模式

责任链模式把多个处理器串成链，请求沿链路依次处理，每个处理器决定是否继续传递。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 多个处理步骤解耦，避免主流程写死所有判断。 |
| 典型场景 | 参数校验、审批流、风控规则、过滤器链。 |
| Java 后端体现 | Servlet Filter、Spring MVC Interceptor、MyBatis 插件链。 |
| 注意点 | 要明确处理顺序、短路规则和异常传播方式。 |

### 命令模式

命令模式把请求封装成命令对象，让请求发起者和执行者解耦，并支持排队、撤销、重试等能力。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 请求调用和执行逻辑解耦。 |
| 典型场景 | 任务队列、异步命令、操作日志、重试任务。 |
| Java 后端体现 | `Runnable`、任务对象、消息命令、调度任务。 |
| 注意点 | 命令对象要保持参数完整，避免执行时依赖易变外部状态。 |

### 状态模式

状态模式把对象不同状态下的行为封装成独立状态类，对象行为随状态变化而变化。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 状态判断分支过多，状态流转规则散落。 |
| 典型场景 | 订单状态、工单状态、审批流、支付状态。 |
| Java 后端体现 | 状态机、订单状态处理器、审批节点处理器。 |
| 注意点 | 状态数量少时不必强行引入，复杂流程可考虑状态机框架。 |

### 迭代器模式

迭代器模式提供统一遍历接口，调用方不需要知道集合内部结构。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 屏蔽不同集合的内部遍历方式。 |
| 典型场景 | 集合遍历、分页遍历、流式读取。 |
| Java 后端体现 | JDK `Iterator`、`Iterable`、集合框架。 |
| 注意点 | 遍历时修改集合要使用迭代器自身的删除能力，避免并发修改异常。 |

### 中介者模式

中介者模式用一个中介对象封装多个对象之间的交互，避免对象之间互相直接依赖。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 多对象网状依赖复杂，难以维护。 |
| 典型场景 | 消息总线、聊天室、复杂 UI 组件协作。 |
| Java 后端体现 | 事件总线、流程协调器、应用服务编排。 |
| 注意点 | 中介者容易膨胀，复杂逻辑仍要拆分到具体领域服务。 |

### 备忘录模式

备忘录模式在不暴露对象内部实现的情况下保存对象状态，用于后续恢复。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 需要撤销、回滚或恢复历史状态。 |
| 典型场景 | 草稿、快照、撤销操作、配置版本。 |
| Java 后端体现 | 版本快照、审计记录、配置回滚。 |
| 注意点 | 状态对象大时要关注存储成本和清理策略。 |

### 访问者模式

访问者模式把作用于对象结构上的操作分离出来，使得可以在不修改对象结构的情况下新增操作。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 对稳定对象结构增加多种操作。 |
| 典型场景 | AST 语法树处理、规则检查、报表导出。 |
| Java 后端体现 | 编译器、表达式树、复杂规则引擎。 |
| 注意点 | 对象结构频繁变化时不适合，普通业务中使用较少。 |

### 解释器模式

解释器模式定义一套语法规则，并为每条规则提供解释执行逻辑。

| 关注点 | 说明 |
| --- | --- |
| 解决问题 | 需要解析和执行简单领域语言。 |
| 典型场景 | 表达式解析、规则表达式、条件过滤。 |
| Java 后端体现 | SpEL、规则引擎、SQL / DSL 解析思想。 |
| 注意点 | 复杂语法不适合手写解释器，应使用成熟解析器或规则引擎。 |

## 后端项目中最常用的模式

| 模式 | 后端常见用途 |
| --- | --- |
| 策略模式 | 替代复杂业务分支，适合多规则、多渠道、多算法。 |
| 代理模式 | AOP、事务、权限、RPC、缓存增强。 |
| 模板方法模式 | 抽象固定流程，把可变步骤交给子类。 |
| 责任链模式 | 过滤器、拦截器、审批链、风控规则链。 |

| 建造者模式 | 构建复杂参数对象、配置对象、请求对象。 |

## 分层架构

经典三层或四层是企业项目的默认组织方式：

```text
Controller   接收请求、参数校验、组装返回
    |
Service      业务逻辑、事务边界、领域编排
    |
Mapper/DAO   数据访问
    |
DB / MQ / Redis / Third API
```

配合的对象类型：

```text
DTO          接口传输对象，通常作为入参 / 出参
VO           展示层对象，面向前端页面或接口响应
Entity / PO  数据库实体，和表结构关系更近
Domain       领域对象，承载核心业务语义
```

用 MapStruct 或明确的转换方法在这些对象之间做转换，避免 Controller、Service、Mapper 之间互相污染模型。

## DDD 简述

领域驱动设计适合业务复杂的中大型系统，核心是按业务领域而非技术细节组织代码：

```text
接口层        接收请求、协议转换、参数校验
应用层        编排用例，不放核心业务规则
领域层        实体、值对象、领域服务、聚合根
基础设施层    数据库、MQ、缓存、外部接口实现
```

DDD 不是银弹。简单 CRUD 系统用清晰三层就够，强行 DDD 会增加抽象层和理解成本。只有当业务规则复杂、概念多、变化频繁，并且团队对领域模型有共识时，DDD 才有明显收益。

## 选型建议

- 大多数业务系统先保证分层清晰、职责单一、命名准确，再考虑设计模式。
- 用策略、工厂、责任链、模板方法替代大段 `if-else` 和重复流程。
- 框架扩展点优先使用接口、组合、事件和配置，不要直接改框架或侵入核心业务。
- 只有当变化点明确、复用价值高时才抽象；不确定的变化不要提前设计过深。
- 复杂业务可以逐步引入 DDD，但不要让简单 CRUD 被复杂分层拖慢。

## 常见误区

- **为了用模式而用模式**：设计模式是降低复杂度的工具，不是代码显得高级的装饰。
- **过度抽象**：只有一个实现类时不一定需要接口和工厂，除非它确实是稳定扩展点。
- **继承滥用**：继承层级过深会让行为难以预测，优先考虑组合。
- **模式命名替代业务命名**：类名只叫 `Strategy`、`Factory` 不够，应体现业务语义。
- **忽视测试**：引入模式后类变多，必须用单元测试覆盖策略选择、链路顺序和异常兜底。
