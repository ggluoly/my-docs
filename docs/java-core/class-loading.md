# 类加载机制

> 本页默认以 JDK 8 为技术基线。

类加载机制决定 `.class` 文件如何进入 JVM 并变成可运行的类型信息。理解它有助于分析 ClassNotFound、NoClassDefFoundError、热部署、SPI 和类冲突问题。

## 类加载过程

类加载过程通常分为：

```text
加载 -> 验证 -> 准备 -> 解析 -> 初始化
```

其中验证、准备、解析合称链接。

## 加载

加载阶段完成三件事：

- 通过类的全限定名获取二进制字节流。
- 将字节流转换为方法区中的运行时数据结构。
- 在堆中生成对应的 `Class` 对象，作为访问入口。

字节流来源不一定是本地 class 文件，也可能来自 jar 包、网络、动态生成字节码等。

## 验证

验证阶段确保字节码符合 JVM 规范，不会破坏虚拟机安全。

常见验证内容：

- 文件格式验证
- 元数据验证
- 字节码验证
- 符号引用验证

## 准备

准备阶段为类变量分配内存并设置默认初始值。

示例：

```java
private static int count = 10;
```

准备阶段 `count` 先是默认值 `0`，真正赋值为 `10` 发生在初始化阶段。

如果是编译期常量：

```java
private static final int SIZE = 10;
```

这类常量可能在准备阶段就被赋值。

## 解析

解析阶段把常量池中的符号引用转换为直接引用。

简单理解：

- 符号引用是名字层面的引用。
- 直接引用是可以直接定位到目标的引用。

## 初始化

初始化阶段执行类构造器 `<clinit>()`，完成静态变量赋值和静态代码块执行。

触发初始化的常见场景：

- `new` 创建对象。
- 访问或设置类的静态变量。
- 调用类的静态方法。
- 反射调用类。
- 初始化子类时先初始化父类。

## 类加载器

JDK 8 常见类加载器：

```text
Bootstrap ClassLoader
    ↑
Extension ClassLoader
    ↑
Application ClassLoader
```

说明：

- Bootstrap ClassLoader 加载 JDK 核心类库。
- Extension ClassLoader 加载扩展目录中的类库。
- Application ClassLoader 加载应用 classpath 下的类。

Java 9 之后类加载器体系随模块化有变化，这里按 JDK 8 基础与存量项目模型理解即可。

## 双亲委派模型

双亲委派模型指类加载请求先委派给父加载器，父加载器无法加载时，子加载器才尝试加载。

作用：

- 保证核心类库安全，例如不能随意替换 `java.lang.Object`。
- 避免类重复加载。
- 让基础类由更高层级加载器统一加载。

## 打破双亲委派

一些场景会打破或调整双亲委派：

- Tomcat 每个 Web 应用使用独立类加载器，隔离不同应用依赖。
- SPI 需要父加载器加载的接口反向发现子加载器中的实现。
- 热部署和插件系统需要动态加载、卸载类。

这些机制不是否定双亲委派，而是在特定场景下为了隔离性和扩展性做出的调整。

## 常见问题

| 问题 | 典型原因 |
| --- | --- |
| `ClassNotFoundException` | 运行期找不到类，常见于依赖缺失或 classpath 错误 |
| `NoClassDefFoundError` | 编译期存在，运行期缺失或初始化失败 |
| `ClassCastException` | 同名类被不同类加载器加载，JVM 认为不是同一个类 |
| `LinkageError` | 类链接阶段出现冲突或不兼容 |

## 实践建议

- 排查类冲突时先看依赖树和实际加载路径。
- 容器环境中注意同一个 jar 是否同时放在容器 lib 和应用 lib。
- 自定义类加载器要谨慎处理父加载器、缓存和资源关闭。
- 动态生成类或热部署场景要关注类加载器泄漏，避免 Metaspace OOM。
