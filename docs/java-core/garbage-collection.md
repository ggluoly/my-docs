# 垃圾回收

> 本页默认以 JDK 8 为技术基线。ZGC、Shenandoah 等后续版本能力放在 Java 版本特性模块中了解。

垃圾回收负责自动释放不再使用的对象内存。理解 GC 可以帮助定位 Full GC 频繁、接口抖动和内存泄漏问题。

## 对象存活判定

HotSpot 主流 GC 使用可达性分析，而不是简单引用计数。

从 GC Roots 出发，能够被直接或间接访问到的对象被认为存活；不可达对象可以被回收。

常见 GC Roots：

- 虚拟机栈中的局部变量引用
- 方法区中的静态变量引用
- 方法区中的常量引用
- JNI 本地方法引用
- 正在运行线程相关对象

循环引用不是问题，因为一组对象即使互相引用，只要整体不可从 GC Roots 到达，就可以被回收。

## 分代回收

JDK 8 常见堆分代：

```text
堆
├── 新生代
│   ├── Eden
│   ├── Survivor From
│   └── Survivor To
└── 老年代
```

经验假设：大多数对象朝生夕死，少数对象长期存活。

流程：

- 新对象通常分配在 Eden。
- Eden 不足时触发 Minor GC。
- Minor GC 后仍存活的对象进入 Survivor。
- 对象达到年龄阈值或 Survivor 放不下时晋升老年代。
- 老年代空间不足时可能触发 Full GC。

## Minor GC、Major GC、Full GC

| 类型 | 常见回收范围 | 特点 |
| --- | --- | --- |
| Minor GC | 新生代 | 频率较高，停顿通常较短 |
| Major GC | 老年代 | 不同收集器语义不完全一致 |
| Full GC | 整个堆或至少包含老年代，可能涉及元空间 | 代价高，停顿明显，应重点关注 |

面试和排查中最容易混淆的是：Minor GC 主要回收新生代，Full GC 通常覆盖老年代甚至整个堆。

## JDK 8 常见垃圾回收器

| 回收器 | 作用区域 | 特点 | 适用场景 |
| --- | --- | --- | --- |
| Serial | 新生代 / 老年代 | 单线程，简单 | 小堆、客户端程序 |
| ParNew | 新生代 | 多线程，常配 CMS | 老项目 CMS 组合 |
| Parallel Scavenge | 新生代 | 吞吐量优先 | 批处理、后台任务 |
| Parallel Old | 老年代 | 吞吐量优先 | 与 Parallel Scavenge 搭配 |
| CMS | 老年代 | 并发标记清除，低停顿 | JDK 8 老项目常见 |
| G1 | 整堆分区 | 可预测停顿，面向服务端 | JDK 8 后期和 JDK 9+ 常见 |

JDK 8 中生产项目常见组合：

- `Parallel Scavenge + Parallel Old`：吞吐量优先。
- `ParNew + CMS`：低停顿老项目常见。
- `G1`：更现代的服务端选择，但需要结合堆大小和停顿目标验证。

## CMS

CMS 目标是降低老年代 GC 停顿。

主要阶段：

```text
初始标记 -> 并发标记 -> 重新标记 -> 并发清除
```

优点：并发阶段可以和用户线程一起执行，停顿较短。

缺点：

- 使用标记清除算法，容易产生内存碎片。
- 对 CPU 敏感。
- 可能出现 concurrent mode failure，退化为更长停顿的 Full GC。

## G1

G1 将堆划分为多个 Region，不再严格按照连续的新生代和老年代物理空间管理。

特点：

- 面向服务端大堆场景。
- 可以设置停顿目标，例如 `-XX:MaxGCPauseMillis=200`。
- 优先回收垃圾收益高的 Region。
- JDK 9 后成为默认 GC，JDK 8 中需要显式开启。

JDK 8 开启 G1：

```text
-XX:+UseG1GC
```

## 常见问题与排查方向

| 现象 | 可能原因 | 方向 |
| --- | --- | --- |
| Minor GC 频繁 | 新生代太小、对象创建过快 | 调整新生代、减少临时对象 |
| Full GC 频繁 | 老年代压力大、内存泄漏 | 分析 GC 日志和堆转储 |
| GC 后内存降不下来 | 存在强引用链 | MAT 分析引用路径 |
| Metaspace OOM | 类加载过多、类加载器泄漏 | 排查动态代理、热部署、脚本引擎 |

## 实践建议

- 先通过 GC 日志和监控判断问题，再调整参数。
- 不要只靠增大堆解决内存泄漏。
- 低延迟接口重点关注 Full GC 次数和停顿时间。
- JDK 8 项目需要区分 CMS、G1 和 Parallel 的参数，不要直接套用 JDK 11+ 的 GC 日志参数。
