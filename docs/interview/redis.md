# Redis 面试题

本页整理 Redis 高频面试题，覆盖基础能力、数据结构、缓存问题、分布式锁、持久化、淘汰策略和性能排查。

## 基础与场景

### Redis 是什么？适合哪些场景？

Redis 是基于内存的高性能键值数据库，支持多种数据结构、持久化、复制、哨兵和集群。常见场景包括缓存热点数据、计数器、排行榜、分布式锁、会话存储、限流、消息队列和临时状态存储。

#### 常见追问

- Redis 为什么适合做缓存？
    Redis 适合做缓存，主要因为它是**基于内存读写，性能很高，访问延迟低，适合承接高并发读请求**。
    它支持丰富的数据结构，比如 String、Hash、List、Set、ZSet，可以比较方便地缓存不同类型的数据。
    Redis 支持过期时间，可以给缓存数据设置 TTL，自动淘汰失效数据。
    它也支持多种淘汰策略，比如内存不足时按 LRU、LFU 等策略清理数据，比较适合缓存场景。
    另外 Redis 支持持久化、主从复制、哨兵和集群，可以提升可用性和扩展性。
    所以总结来说，**Redis 适合做缓存是因为速度快、数据结构丰富、支持过期和淘汰机制，并且有较好的高可用和扩展能力**。
- 哪些数据不适合放 Redis？
    不适合放 Redis 的数据主要有几类。
    第一类是**强一致性要求特别高的数据**，比如余额、库存最终扣减结果这类核心数据，Redis 可以做加速，但不能只依赖 Redis 作为唯一数据源，最终还是要以数据库为准。
    第二类是**数据量特别大、访问频率又很低**的数据，放 Redis 会浪费内存，因为 Redis 主要成本在内存。
    第三类是**没有明确过期时间、又会不断增长的数据**，比如长期累积的日志、历史明细，不适合作为缓存长期放在 Redis。
    第四类是** value 特别大的数据**，比如大对象、大 JSON、图片、文件内容，这类数据会占用大量内存，也会影响网络传输和 Redis 单线程处理效率。
    第五类是**频繁变化但读收益不高的数据**，如果一直更新缓存，维护成本可能比查询数据库还高。
    所以 **Redis 更适合放热点数据、读多写少、允许短时间不一致、体积较小并且有明确过期策略**的数据。
- Redis 能不能替代数据库？
    一般不能用 Redis 替代数据库。
    Redis 的优势是内存读写快，适合做缓存、计数器、分布式锁、排行榜这类高性能场景，但它不是典型的关系型数据库。
    数据库更擅长持久化存储、复杂查询、事务一致性、约束校验、关联关系和数据恢复。Redis 虽然有 RDB、AOF 持久化，但持久化更多是为了故障恢复，不能完全等同于数据库的数据可靠性。
    如果把 Redis 当唯一数据库使用，一旦配置不当、内存淘汰、持久化丢失、主从切换异常，都可能造成数据风险。
    所以面试里可以这样答：**Redis 通常不能替代数据库，它更多是作为数据库前面的缓存或高性能组件使用；只有在数据结构简单、允许一定丢失或有特殊高性能需求的场景**，才可能作为主存储，但要额外考虑持久化、高可用和数据一致性。

### Redis 为什么快？

Redis 主要**基于内存读写，数据结构高效，单线程模型避免了大量线程切换和锁竞争，同时使用 IO 多路复用处理大量连接**。Redis 6 之后引入多线程 IO，但命令执行核心仍以单线程模型为主。

#### 常见追问

- Redis 单线程为什么还能高并发？
    Redis 单线程还能支持高并发，主要有几个原因。
    首先 Redis 的**数据主要在内存里操作**，读写速度非常快，不像数据库那样大量依赖磁盘 IO。
    其次 Redis 使用** IO 多路复用**模型，一个线程就可以同时监听和处理大量客户端连接，避免了每个连接一个线程带来的上下文切换开销。
    第三，Redis 的**命令执行大多比较简单**，单线程串行执行反而避免了加锁、线程竞争这些成本。
    另外 Redis 的**网络协议比较简单**，数据结构也做了很多优化，所以整体处理效率很高。
    需要注意的是，Redis 单线程主要指命令执行线程是单线程，不代表整个 Redis 只有一个线程，比如持久化、异步删除、集群同步这些可能会用后台线程。
    Redis 6 之后网络 IO 支持多线程，但命令执行核心流程仍然主要是单线程。
- IO 多路复用是什么？
    IO 多路复用就是**一个线程可以同时管理多个网络连接**。
    正常情况下，如果一个连接对应一个线程，连接很多时线程数量会很多，切换和资源开销都比较大。IO 多路复用是把多个连接注册到一个组件上，由它帮忙监听哪些连接已经准备好读写，线程只处理已经就绪的连接。
    在 Linux 里常见实现有 select、poll、epoll。Redis 在 Linux 下主要使用 epoll，它可以高效监听大量 socket 事件。
    所以 Redis 虽然命令执行是单线程，但通过 IO 多路复用，一个线程也能同时处理大量客户端连接，不需要为每个连接创建一个线程。
- Redis 6 多线程解决什么问题？
    Redis 6 的多线程主要解决的是**网络 IO 处理瓶颈**。
    Redis 之前命令执行是单线程，网络读写、协议解析、命令执行基本都在主线程里完成。当连接数很多、请求和响应数据量比较大时，主线程可能会耗在 socket 读写和协议处理上。
    Redis 6 引入可配置的多线程 IO 后，可以把网络数据读取、写回等工作交给多个线程处理，提高高并发场景下的吞吐量。需要配置 `io-threads` 才会启用；读请求是否也由 IO 线程处理还取决于 `io-threads-do-reads` 配置。
    但要注意，Redis 6 的多线程不是让命令并行执行，真正执行命令、操作数据结构的核心逻辑仍然主要是单线程，所以它没有改变 Redis 单线程串行执行命令的模型。
    所以面试里可以说：**Redis 6 多线程主要是优化网络 IO 性能，提升吞吐，不是为了解决命令执行并发问题**。

### Redis 有哪些功能？

Redis 常见功能包括**数据缓存、分布式锁、持久化、事务、发布订阅、消息队列、Lua 脚本、过期淘汰、主从复制、哨兵和集群**。面试时要说明 Redis 不只是缓存，而是高性能内存数据结构服务。

#### 常见追问

- Redis 事务是否支持回滚？
    Redis 事务**不支持传统意义上的回滚**。
    Redis 事务是通过 MULTI、EXEC、DISCARD、WATCH 实现的。MULTI 后命令会先入队，EXEC 时按顺序执行。
    如果是入队阶段就发生语法错误，EXEC 时整个事务不会执行；但如果是 EXEC 执行阶段某条命令出错，比如类型不匹配，其他命令仍然会继续执行，已经执行成功的命令不会回滚。
    DISCARD 只能在 EXEC 之前取消事务，不能回滚已经执行的结果。
    所以** Redis 事务保证的是命令按顺序串行执行，中间不会被其他客户端命令插入，但不保证像 MySQL 那样的原子回滚**。
- Lua 脚本有什么作用？
    Lua 脚本在 Redis 里主要作用是让一组操作以**不与其他客户端命令交错**的方式原子执行。
    因为 Redis 执行 Lua 脚本时，会把整个脚本当成一个整体执行，中间不会被其他客户端命令打断，所以适合做“先判断再操作”这种场景，比如扣库存、限流、分布式锁释放等。
    另外，Lua 脚本也可以减少网络交互次数。原本多次 Redis 命令调用，可以封装成一个脚本一次发送到 Redis 执行。
    需要注意，Lua 原子执行不是数据库事务：脚本运行中发生错误时，之前已成功执行的写命令不会自动回滚，因此应先校验参数和前置条件。**Lua 脚本执行期间会阻塞 Redis 主线程**，所以脚本逻辑不能太复杂，也不能执行太久。
- Redis Stream 适合什么场景？
    Redis Stream 适合做**消息队列**相关的场景。
    它支持消息持久化、消息 ID、消费组、ACK 确认机制，也支持一个消息被多个消费者组分别消费，所以比 List 更适合做具备消费确认能力的消息消费。
    常见场景包括异步任务处理、日志收集、订单状态流转、系统事件通知、削峰填谷等。
    消费组投递后、确认前的消息会进入 PEL；消费者宕机时，需要通过 `XPENDING` 配合 `XCLAIM` 或 `XAUTOCLAIM` 接管并重试。消费通常按至少一次处理，业务侧仍要保证幂等；不当的 `XTRIM` 或 `XDEL` 还可能使待处理消息正文不可用。
    不过 Redis Stream 更适合中小规模消息队列场景，如果对消息堆积能力、复杂路由、严格事务、超高可靠性要求很高，一般还是会选择 Kafka、RocketMQ 这类专业 MQ。

### Redis 和 Memcached 有什么区别？

Redis 支持 string、hash、list、set、zset 等多种结构，支持持久化、主从复制、Lua 和丰富命令；
Memcached 更偏简单 key-value 缓存，不支持复杂数据结构和持久化。现在 Java 后端项目通常更常用 Redis。

#### 常见追问

- Memcached 还有什么优势？
    Memcached 的优势主要是简单和高性能。
    它**只做纯内存 KV 缓存**，没有持久化、复杂数据结构、事务、Lua 这些功能，所以实现更轻量，运行开销也更小。
    另外 Memcached 是多线程模型，在多核机器上处理简单 KV 读写时吞吐量比较好；Redis 单线程命令执行为主，虽然性能也很高，但在纯缓存场景下 Memcached 仍然有优势。
    Memcached 还使用 slab 内存分配机制，对固定大小对象的缓存管理比较高效，适合存放简单字符串、页面片段、热点数据。
    不过它功能比较单一，不支持丰富数据结构、持久化、高可用和更复杂的缓存操作，所以现在实际项目里 Redis 用得更多。
- Redis value 最大可以多大？
    Redis 单个 **String 类型 value 最大是 512MB**。
    如果是 Hash、List、Set、ZSet 这类结构，里面每个元素本质上也受 Redis 字符串大小限制，单个元素最大也是 512MB，但整个集合可以包含很多元素。
    实际项目里一般不建议把 value 做得太大，因为大 key 会影响网络传输、内存分配、持久化、主从同步和删除性能，容易造成 Redis 阻塞。通常会控制 value 大小，必要时做拆分存储。
- 复杂数据结构会带来什么风险？
    主要风险是**大 key 和操作成本变高**。
    复杂数据结构一旦元素很多，读写、遍历、删除、过期、持久化、主从同步都会更慢，还会增加内存开销和网络传输压力。比如 Hash、Set、ZSet 数据量大时，内部维护成本也会上升，容易带来阻塞和性能抖动。整体上就是更容易影响 Redis 的稳定性，所以一般要控制单个 key 的规模，避免把太多数据放到一个 key 里。

## 数据结构与客户端

### Redis 常见数据类型有哪些？

常见数据类型包括 string、list、hash、set、sorted set(ZSet)。
string 可做缓存和计数器；list 可做队列；hash 适合存对象字段；set 适合去重和交并差；zset 适合排行榜和带权重排序。

#### 常见追问

- 点赞数用什么类型？
    如果只是点赞数，一般用 Redis 的 string 类型存整数，配合 INCR/DECR 做原子自增自减。
    如果还要记录谁点过赞，通常再配合 set 来存用户 ID。
- 排行榜用什么类型？
    排行榜一般用 sorted set，也就是 zset。
    它可以给每个成员设置一个 score，Redis 会按照 score 自动排序，适合做积分榜、热度榜这类场景。常用命令有 zadd、zincrby、zrange、zrevrange。
- 用户对象适合用 string 还是 hash？
    用户对象一般更适合用 hash。
    因为用户信息通常有多个字段，hash 可以按字段存取，更新局部字段也更方便。
    如果只是整对象序列化后整体读写，才更适合用 string。

### Redis Java 客户端有哪些？

常见 Java 客户端包括 Jedis、Lettuce 和 Redisson。
Jedis 是较早的同步阻塞客户端；Lettuce 基于 Netty，支持异步和响应式；Redisson 提供分布式锁、集合、队列等高级对象抽象。

#### 常见追问

- Spring Boot 默认常用哪个客户端？
    Spring Boot 2.x 开始，默认常用的是 Lettuce。
    它是基于 Netty 的，线程安全，支持异步和响应式，连接可以复用。
    Spring Boot 1.x 默认更多是 Jedis，Jedis 是阻塞式客户端，通常需要配合连接池使用。
- Redisson 分布式锁有什么优势？
    Redisson 分布式锁的优势主要是封装比较完整，使用起来更像 Java 的 Lock 接口。
    它支持自动续期，也就是 watchdog 机制，避免业务没执行完锁就过期。
    同时它支持可重入锁、公平锁、读写锁、联锁、红锁等多种锁类型。
    另外加锁和释放锁底层通过 Lua 脚本保证原子性，释放锁时也会校验是不是当前线程持有，避免误删别人的锁。
- Jedis 连接池为什么重要？
    Jedis 连接池重要是因为 **Jedis 连接不是线程安全的，通常不能多个线程共享同一个 Jedis 实例**。
    如果每次请求都新建和关闭连接，TCP 连接成本比较高，也会影响性能。
    连接池可以复用连接，控制最大连接数，避免并发高的时候把 Redis 或应用本身打垮。
    所以在多线程环境下使用 Jedis，一般都会配合 JedisPool。

### Jedis 和 Redisson 有什么区别？

Jedis 更像 Redis 命令客户端，直接执行 Redis 命令；Redisson 在 Redis 之上封装了分布式对象和并发工具，比如分布式锁、信号量、阻塞队列、延迟队列等。需要简单命令操作可以用 Jedis / Lettuce，需要分布式锁等高级能力常用 Redisson。

#### 常见追问

- Redisson 锁如何续期？
    Redisson 锁的续期是通过 watchdog 机制实现的。
    如果加锁时没有手动指定 leaseTime，默认锁过期时间是 30 秒，后台会启动定时任务定期检查锁是否还被当前线程持有。
    如果还持有，就会自动把锁的过期时间续回 30 秒，默认大概每 10 秒续期一次。
    等业务执行完主动 unlock，或者持锁线程宕机，续期停止，锁最终会自动过期释放。
- Lettuce 和 Jedis 有什么区别？
    Lettuce 和 Jedis 都是 Redis 的 Java 客户端。
    主要区别是 Jedis 基于阻塞 IO，实例本身**不是线程安全**的，多线程场景一般要配合 JedisPool 使用。
    Lettuce 基于 Netty，是**线程安全**的，一个连接可以被多个线程共享，并且支持同步、异步、响应式 API。
    在高并发场景下，Lettuce 的连接复用和异步能力更好一些；Jedis API 更简单直接，使用成本低。
    现在 Spring Boot 2.x 默认使用 Lettuce 作为 Redis 客户端。
- 为什么不要自己随便实现 Redis 锁？
    不建议自己随便实现 Redis 锁，主要是因为分布式锁细节比较多，写错很容易出现并发安全问题。
    比如加锁必须保证设置 key 和过期时间是原子操作，释放锁时要校验 value，不能把别的线程的锁删掉。
    另外还要考虑业务执行时间超过锁过期时间后的续期问题，以及 Redis 主从切换、网络抖动、客户端宕机等异常场景。
    如果这些场景处理不好，可能会出现锁失效、误删锁、死锁或者多个线程同时进入临界区。
    所以实际项目里一般优先使用 Redisson 这类成熟组件，它已经封装了加锁、解锁、自动续期等机制。

## 缓存问题

### 缓存穿透、击穿、雪崩是什么？

缓存穿透是查询不存在的数据绕过缓存打到数据库；
缓存击穿是热点 Key 过期瞬间大量请求打到数据库；
缓存雪崩是大量 Key 同时过期或缓存集群故障导致数据库压力骤增。

#### 常见解决方案

- 穿透：缓存空值、布隆过滤器。
- 击穿：互斥锁、逻辑过期、热点 Key 预热。
- 雪崩：过期时间加随机值、多级缓存、限流降级。

#### 什么是布隆过滤器？
    布隆过滤器是一种用来**判断一个元素“可能存在”或者“一定不存在”的数据结构**。
    它底层通常是一个位数组，加上多个哈希函数。添加元素时，会经过多个哈希计算，把对应位置标记为 1；查询元素时，也计算这些位置，如果有任意一个位置不是 1，就说明一定不存在；如果都为 1，只能说明可能存在。
    它的优点是空间占用小、查询速度快，适合做缓存穿透防护、黑名单判断、URL 去重这类场景。
    缺点是有误判率，也就是可能把不存在的数据判断成存在，但不会把存在的数据判断成不存在。

### 缓存和数据库一致性怎么保证？

常见方案是**更新数据库后删除缓存，并通过重试、消息队列或延迟双删增强可靠性**。强一致成本很高，多数业务追求最终一致性。读多写少场景要特别注意并发读写导致旧值回写缓存。

#### 常见追问

- 为什么通常是删除缓存而不是更新缓存？
    主要是为了**降低一致性问题和实现复杂度**。
    更新数据库和更新缓存不是一个原子操作，如果数据库更新成功、缓存更新失败，或者并发情况下旧请求后写入缓存，就可能导致缓存里长期存在脏数据。
    删除缓存的思路是让缓存失效，下次查询时再从数据库加载最新数据并回写缓存，这样更符合 Cache Aside 模式，也更简单可靠。
    另外有些缓存值可能不是单表字段，可能经过聚合、计算、转换，更新缓存的成本和出错概率更高，直接删除更稳。
    所以一般写操作是先更新数据库，再删除缓存；如果对一致性要求更高，还会配合延迟双删、消息队列重试或者设置较短过期时间。
- 延迟双删解决什么问题？
    延迟双删**主要解决并发读写下缓存和数据库短时间不一致的问题**。
    一个典型时序是：读请求在写请求更新数据库前读取旧值；写请求随后更新数据库并删除缓存；读请求最后才把先前读到的旧值写回缓存。这样数据库已是新值，缓存却被旧值覆盖。
    延迟双删是在数据库事务提交后，隔一小段时间再删除一次缓存，把这段时间内可能被并发读请求回写的旧缓存清掉。
    它不能做到强一致，只是降低脏数据在缓存中长期存在的概率。实际项目里一般还会配合缓存过期时间、删除失败重试、可靠消息或订阅 binlog 来保证最终一致性。
- Canal + MQ 如何做缓存一致性？
    Canal + MQ 做缓存一致性，本质是**通过监听数据库 binlog 来异步删除缓存，保证最终一致性**。
    流程一般是：业务先更新数据库，MySQL 产生 binlog，Canal 模拟从库订阅 binlog，解析出变更数据后发送到 MQ，缓存服务消费 MQ 消息，根据表名、主键或业务 key 删除对应缓存。
    这里通常还是删除缓存，而不是更新缓存，因为缓存值可能有复杂计算，删除后下次查询再回源数据库加载最新数据更安全。
    MQ 的作用主要是削峰、解耦和失败重试，但它本身不会自动避免消息丢失。
    实现时需要保证 Canal 位点提交和向 MQ 投递可恢复、可重试，Broker 启用合适的持久化和副本策略；消费侧按至少一次处理，删除缓存要保证幂等，失败要重试或进死信队列。消息可能乱序，可以用版本号、更新时间或者按 key 分区来降低影响；缓存本身也要设置过期时间兜底。
    这种方案不能保证强一致，但能做到比较可靠的最终一致性，适合对缓存一致性要求较高、又不想在业务代码里强耦合删除缓存逻辑的场景。

## 分布式锁

### Redis 分布式锁怎么实现？

常见实现是 `SET key value NX PX timeout`，加锁时设置唯一值和过期时间，释放锁时校验唯一值后删除，避免误删其他线程的锁。释放锁建议用 Lua 脚本保证校验和删除的原子性。生产环境可以使用 Redisson。

#### 常见追问

- 为什么要设置唯一 value？
    Redis 分布式锁设置唯一 value，主要是为了标识这把锁的持有者，避免误删别人的锁。
    因为锁一般会设置过期时间，如果线程 A 加锁后业务执行太久，锁过期了，线程 B 可能重新加锁成功。这个时候线程 A 执行完如果直接 delete key，就会把线程 B 的锁删掉，导致并发安全问题。
    所以加锁时 value 一般设置成 UUID、线程标识这类唯一值，释放锁时先判断 value 是不是自己设置的，如果是才删除。
    而且判断和删除必须保证原子性，通常用 Lua 脚本完成：先 get 判断 value，相等再 del。
    所以唯一 value 的核心作用就是**保证“谁加的锁，谁才能释放”，避免锁过期或并发场景下误释放**。
- 为什么释放锁要用 Lua？
    主要是为了保证“判断锁是不是自己的”和“删除锁”这两个操作的原子性。
    如果先 get 判断 value 相等，再单独 del 删除，中间锁可能刚好过期，然后被其他线程重新加锁成功，这时候再 del 就会误删别人的锁。
    Lua 脚本在 Redis 中执行是原子的，可以把判断 value 和删除 key 放到一个脚本里执行，确保只有 value 匹配时才删除。
    所以释放锁常用 Lua，本质是为了**避免并发场景下误释放锁**。
- Redisson watchdog 是什么？
    Redisson watchdog 是**锁续期机制**，用来**防止业务还没执行完，Redis 锁就自动过期了**。
    如果加锁时没有手动指定 leaseTime，Redisson 默认会给锁设置 30 秒过期时间，并启动 watchdog 后台任务，定期检查当前线程是否还持有锁。
    只要锁还被当前线程持有，就会自动把锁的过期时间续回 30 秒，默认大概每 10 秒续期一次。
    如果业务执行完释放锁，watchdog 就停止续期；如果服务宕机，续期任务也会停止，锁最终会在 30 秒后过期释放。
    所以它**主要解决的是业务执行时间不确定时，锁提前过期的问题**。

### Redis 分布式锁有什么缺陷？

常见问题包括锁过期导致业务没执行完锁已释放、线程误删别人锁、主从切换导致锁丢失、单点 Redis 故障等。解决思路包括唯一值校验、Lua 原子释放、合理过期时间、自动续期和 Redisson 等成熟实现。

#### 常见追问

- RedLock 是否一定可靠？
    RedLock 不是一定可靠的，它只是比单 Redis 实例锁提高了可用性和一定的容错能力。
    它的核心问题是依赖时间假设，比如 Redis 节点过期时间、客户端时钟、网络延迟、GC 停顿等。如果客户端拿到锁后发生长时间停顿，锁已经过期，其他客户端又拿到了锁，前一个客户端恢复后仍然可能继续操作共享资源。
    所以 RedLock 不能完全解决“锁过期后旧客户端继续执行”的问题。对强一致要求很高的场景，通常还需要 fencing token，也就是递增令牌，让下游资源只接受更大的 token，拒绝旧请求。
    如果只是防止重复执行、降低并发冲突，RedLock 可以用；但如果是金融扣款、库存强一致这类场景，不能只依赖 RedLock，更适合用数据库事务、唯一约束或带 fencing token 的方案。即使使用 ZooKeeper、etcd 等协调组件，资源端仍应校验递增令牌或版本，拒绝会话、租约失效后恢复的旧客户端请求。
- 分布式锁超时怎么设置？
    分布式锁超时时间一般不能随便写固定值，要根据业务最长执行时间来设置。
    锁的过期时间应该大于业务正常执行的最大耗时，再加一定冗余，比如业务最多 5 秒，可以设置 10 到 15 秒，避免业务没执行完锁就过期。
    但也不能设置太长，否则服务宕机后锁释放太慢，会影响其他请求获取锁。
    如果业务执行时间不确定，像 Redisson 可以不指定 leaseTime，使用 watchdog 自动续期，默认锁 30 秒，每 10 秒左右续期一次，业务执行完再主动释放。
    另外还要区分两个时间：一个是获取锁的等待时间，表示最多等多久拿锁；一个是锁持有的过期时间，表示拿到锁后多久自动释放。
    最终还需要结合业务做幂等、防重复提交，强一致场景不能只依赖超时时间兜底。
- 数据库锁和 Redis 锁怎么选？
    数据库锁和 Redis 锁主要看**一致性要求和性能要求**。
    如果是强一致场景，比如扣库存、扣余额、订单状态流转，更倾向用数据库锁，因为数据本身就在数据库里，可以结合事务、行锁、唯一索引、乐观锁保证一致性。
    如果是高并发下做防重复、限流、任务调度、缓存重建这类场景，可以用 Redis 分布式锁，因为性能更好，吞吐更高，对数据库压力小。
    数据库锁的优点是一致性强、和事务结合简单，缺点是性能压力大，锁持有时间长可能影响数据库。
    Redis 锁的优点是性能好、适合跨服务协调，缺点是要考虑锁超时、续期、误删、主从切换、业务幂等等问题。
    所以选择上，如果锁保护的是数据库核心数据，优先数据库方案；如果只是控制并发入口或避免重复执行，优先 Redis 锁。强一致场景不能只依赖 Redis 锁，最好还要有数据库约束或幂等兜底。

## 持久化、内存与淘汰

### RDB 和 AOF 有什么区别？

RDB 是定期快照，恢复快但可能丢失最近数据；
AOF 记录写命令，通常数据丢失窗口更小，但文件更大、恢复较慢。实际可靠性取决于 `appendfsync`：`always` 最可靠但开销大，常用的 `everysec` 宕机时仍可能丢失约一秒数据，`no` 的丢失窗口更大。
生产环境常结合使用。

#### 常见追问

- RDB 什么时候触发？
    RDB 主要在几种情况下触发：
    第一种是配置了 `save` 规则，比如 `save 900 1`，表示 900 秒内至少有 1 次写操作就触发一次 RDB 快照。
    第二种是手动执行 `SAVE` 或 `BGSAVE`，其中 `SAVE` 会阻塞 Redis 主线程，`BGSAVE` 会 fork 子进程异步生成 RDB，线上一般用 `BGSAVE`。
    第三种是主从全量复制时，主节点会生成 RDB 文件发送给从节点。
    第四种是 Redis 正常关闭时，如果开启了 RDB 持久化，也可能触发一次快照保存。
    面试里可以补一句：RDB 不是每次写操作都持久化，它是快照机制，所以性能好，但宕机时可能丢失最近一次快照之后的数据。
- AOF rewrite 是什么？
    AOF rewrite 是 AOF 文件重写机制，用来压缩 AOF 文件体积。
    Redis 运行时间长了，AOF 里会有很多历史写命令，比如同一个 key 被多次修改、删除，这些命令其实没必要都保留。
    rewrite 会根据当前内存中的最终数据状态，重新生成一份更小的 AOF 文件，而不是简单地压缩原文件。
    它一般通过 `BGREWRITEAOF` 触发，Redis 会 fork 子进程在后台重写，避免长时间阻塞主线程，但 fork 本身仍可能造成短暂停顿。
    Redis 7 的 multi-part AOF 由 base AOF、增量 AOF 和 manifest 组织；重写生成新的 base AOF，重写期间的新写入会进入新的增量 AOF，完成后由 manifest 切换到新的文件集合。旧版本单文件 AOF 则会在重写完成后补入重写期间的增量命令再替换旧文件。
    所以 **AOF rewrite 的作用就是减少磁盘占用，加快 Redis 重启恢复速度**。
- 两者同时开启时 Redis 如何恢复？
    RDB 和 AOF 同时开启时，Redis 启动恢复会**优先使用 AOF** 文件，而不是 RDB。
    原因是 AOF 记录的是写命令，通常数据更完整，丢失数据更少；RDB 是快照，可能会丢失上一次快照之后的数据。
    恢复流程大致是：Redis 启动时先判断是否开启 AOF，如果开启并且 AOF 文件存在，就加载 AOF；如果没有开启 AOF，或者 AOF 文件不存在，才会加载 RDB。
    如果 AOF 文件损坏，Redis 可能启动失败，需要用 `redis-check-aof` 修复后再启动。
    Redis 4.0 之后 AOF 支持混合持久化，AOF 文件前半部分可能是 RDB 格式，后半部分是增量 AOF 命令，但启动时仍然是按 AOF 文件优先恢复。

### Redis 内存如何优化？

可以减少大 key 和无意义 key，设置合理过期时间，使用 hash 聚合小对象字段，选择紧凑的数据结构，压缩 value，避免保存重复数据。很多小 key 会带来额外元数据开销，用 hash 管理对象字段有时能减少内存占用。

#### 常见追问

- 什么是 big key？
    Big key 指的是 value 很大，或者集合类型元素数量特别多的 key。
    比如一个 String 存了几 MB 甚至几十 MB 的内容，或者一个 List、Hash、Set、ZSet 里有几十万、上百万个元素，都可以认为是 big key。
    它的问题主要是会阻塞 Redis 主线程，比如删除、过期、迁移、持久化、网络传输时都可能耗时很长，导致 Redis 延迟升高。
    另外 big key 也会造成内存分布不均、主从同步变慢、集群数据倾斜等问题。
    排查时可以用 `redis-cli --bigkeys`，或者通过 `MEMORY USAGE key` 看单个 key 的内存占用。
    处理方式一般是拆分 key，比如把大 Hash 拆成多个小 Hash；删除时用 `UNLINK` 这种异步删除方式，避免直接 `DEL` 阻塞。
- 如何扫描大 key？
    扫描 big key 一般用这几种方式：
    1. 可以用 `redis-cli --bigkeys`，它内部是基于 `SCAN` 渐进式扫描，不会像 `KEYS` 一样一次性阻塞全库，但它主要统计每种类型里最大的 key。
    2. 如果要看 key 实际内存占用，可以结合 `SCAN` 和 `MEMORY USAGE key`，逐批扫描 key，然后对每个 key 计算内存大小。
    3. 对集合类型，还可以用 `STRLEN`、`HLEN`、`LLEN`、`SCARD`、`ZCARD` 看元素数量或字符串长度，判断是否过大。
    4. 生产环境不要用 `KEYS *` 扫描，因为它会阻塞 Redis 主线程。
    5. 如果是 Redis Cluster，需要对每个 master 节点分别执行扫描，因为 key 分散在不同节点上。
- hash 一定比 string 省内存吗？
    不一定。Hash 是否比 String 省内存，主要看数据规模和底层编码。
    如果是很多小字段，比如一个用户对象用一个 Hash 存多个属性，Hash 在元素较少、字段和值较小时会用压缩结构，能减少大量 key 的元数据开销，通常比多个 String 省内存。
    但如果 Hash 里的 field 或 value 比较大，或者元素数量超过阈值，Hash 会转成哈希表结构，这时内存开销会上升，不一定比 String 省。
    如果只是存一个简单值，用 String 反而更合适，没必要为了省内存强行用 Hash。
    所以结论是：**小对象、多字段、字段值较小的场景，Hash 通常更省；大 value、大 Hash 或单值场景，不一定省，甚至可能更耗内存**。

### Redis 淘汰策略有哪些？

常见淘汰策略包括 `volatile-lru`、`volatile-lfu`、`volatile-ttl`、`volatile-random`、`allkeys-lru`、`allkeys-lfu`、`allkeys-random` 和 `noeviction`。`volatile-*` 只淘汰设置了过期时间的 key，`allkeys-*` 会从所有 key 中淘汰；LFU 策略自 Redis 4.0 起提供。

#### 常见追问

- LRU 和 LFU 有什么区别？
    LRU 是 Least Recently Used，**按“最近是否使用过”淘汰**，优先淘汰最久没有被访问的 key。
    LFU 是 Least Frequently Used，**按“访问频率”淘汰**，优先淘汰访问次数最少的 key。
    核心区别是：LRU 看的是时间维度，最近访问过就认为更重要；LFU 看的是频率维度，访问次数多就认为更重要。
    比如一个 key 过去访问很多次，但最近一段时间没访问，LRU 可能会淘汰它，LFU 不一定会淘汰；反过来，一个 key 刚被访问一次，LRU 会认为它比较新，LFU 仍然认为它频率低。
    在 Redis 里 LRU 和 LFU 都不是完全精确实现，而是近似算法，通过采样来选择淘汰对象。LRU 适合热点变化比较快的场景，LFU 更适合热点相对稳定的场景。
- noeviction 会发生什么？
    `noeviction` 是 Redis 的**默认内存淘汰策略**。
    当 Redis 内存达到 `maxmemory` 限制后，不会淘汰任何 key。
    如果继续执行会增加内存的写命令，比如 `set`、`lpush`、`hset`，Redis 会直接返回 OOM 错误。
    读命令一般不受影响，删除、过期这类释放内存的命令也可以继续执行。
    所以这个策略适合不允许数据被自动淘汰的场景，但需要业务能处理写入失败，或者提前做好扩容和内存监控。
- 缓存场景通常选哪种策略？
    缓存场景通常选 `allkeys-lru`。
    因为缓存的特点是数据允许丢失，内存满了以后从所有 key 里淘汰最近最少使用的数据，比较符合热点缓存场景。
    如果热点数据比较稳定，也可以选 `allkeys-lfu`，按访问频率淘汰，效果可能比 LRU 更好。
    如果业务明确只有设置了过期时间的 key 才能被淘汰，可以选 `volatile-lru` 或 `volatile-ttl`，但要注意没有 TTL 的 key 不会被淘汰，可能导致 OOM。
    一般纯缓存不建议用 `noeviction`，否则内存满了写请求会直接失败。

### Redis 常见性能问题有哪些？

常见问题包括 big key、hot key、慢查询、持久化 fork 开销、AOF rewrite、主从复制延迟、网络带宽瓶颈和阻塞命令。主从节点最好部署在低延迟网络环境中，避免复制延迟扩大。

#### 常见追问

- RDB 快照为什么可能影响性能？
    RDB 快照可能影响性能，主要是因为 Redis 做 RDB 时会 `fork` 子进程。
    `fork` 本身需要复制父进程的页表，如果 Redis 内存很大，`fork` 过程可能变慢，导致主线程短暂阻塞。
    RDB 子进程写快照文件时会产生大量磁盘 IO，如果磁盘压力大，也会影响 Redis 的整体性能。
    另外 RDB 期间 Redis 继续处理写请求时，会触发写时复制，也就是 COW，修改过的数据页会被复制一份，可能导致额外内存占用和内存带宽消耗。
    所以 Redis 实例内存越大、写入越频繁、磁盘越慢，RDB 对性能的影响就越明显。
- 主从复制延迟怎么排查？
    主从复制延迟一般先看 `info replication`。
    重点看主库的 `master_repl_offset` 和从库的 `slave_repl_offset` 差值，差值越大说明复制积压越多；也可以看从库的 `master_link_status`、`master_last_io_seconds_ago` 判断连接是否正常。
    然后排查几个方向：
    一是网络问题，主从之间带宽不足、RTT 高、丢包都会导致延迟。
    二是主库写入压力太大，写命令太多或者有大 key，复制流量跟不上。
    三是从库处理能力不足，比如 CPU 高、执行慢查询、磁盘 IO 高，导致从库回放命令慢。
    四是主库或从库在做 RDB、AOF rewrite，`fork` 和磁盘 IO 可能影响复制。
    五是发生了全量同步，全量同步会传 RDB，数据量大时延迟会明显增加。
    实际排查会结合 `info replication`、`slowlog`、`bigkeys`、机器 CPU/内存/磁盘 IO、网络监控一起看。
- Redis 慢查询怎么查看？
    Redis 慢查询可以用 `slowlog` 查看。
    常用命令是：
    `slowlog get` 查看慢查询记录。
    `slowlog get 10` 查看最近 10 条。
    `slowlog len` 查看当前慢查询数量。
    `slowlog reset` 清空慢查询日志。
    慢查询阈值由 `slowlog-log-slower-than` 控制，单位是微秒，比如 `10000` 表示超过 10ms 记录。
    慢查询最多保存多少条由 `slowlog-max-len` 控制。
    需要注意，Redis 慢查询统计的是命令执行时间，不包括网络传输和排队等待时间。
