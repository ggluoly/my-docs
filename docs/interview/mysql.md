# MySQL 面试题

本页整理 MySQL 高频面试题，覆盖范式、事务、索引、锁、存储引擎和性能排查。回答时建议结合 InnoDB 当前主流特性，遇到旧版本行为要说明版本差异。

## 数据库基础

### 数据库三范式是什么？

第一范式要求字段具有原子性，不能再拆分；第二范式要求非主属性完全依赖主键，避免部分依赖；第三范式要求非主属性不依赖其他非主属性，避免传递依赖。实际业务中会在规范化和查询性能之间做权衡，必要时适度反范式。

### 如何获取 MySQL 版本？

可以执行 `select version();` 查看数据库版本，也可以通过 `show variables like 'version%';` 查看更详细的版本变量。排查兼容性问题时，版本信息很关键，例如 MySQL 5.7 和 8.0 在自增、窗口函数、JSON、索引优化等方面都有差异。

### char 和 varchar 有什么区别？

`char` 是定长字符串，适合长度固定的数据，比如状态码、MD5；`varchar` 是变长字符串，适合长度变化较大的数据。`char` 可能浪费空间但读取简单，`varchar` 更节省空间但需要额外记录长度。

#### 常见追问

- 手机号应该用 char 还是 varchar？
- varchar 长度是不是越大越好？
- 字符集会影响存储空间吗？

### 自增 ID 删除后重启再插入会怎样？

这个问题和存储引擎、MySQL 版本有关。MyISAM 会把自增值持久化到表文件；早期 InnoDB 的自增计数可能在重启后根据表中最大值重新计算；MySQL 8.0 起 InnoDB 自增计数持久化，重启后不会简单回退到最大值。面试时要说明不能脱离版本给固定答案。

## 事务与锁

### ACID 是什么？

ACID 分别是原子性、一致性、隔离性和持久性。原子性保证事务内操作要么都成功要么都失败；一致性保证数据从一个一致状态转到另一个一致状态；隔离性控制并发事务互相影响；持久性保证提交后的数据不会因为宕机丢失。

### 事务隔离级别有哪些？

事务隔离级别包括读未提交、读已提交、可重复读和串行化。MySQL InnoDB 默认可重复读，并通过 MVCC 和锁机制解决多数并发读写问题。

### MVCC 是什么？

MVCC 是多版本并发控制，通过保存数据版本和 Read View，让读写在多数场景下互不阻塞，提高并发性能。它是 InnoDB 实现一致性读的重要机制。

### 行锁和表锁有什么区别？

表锁锁住整张表，开销小但并发能力差；行锁锁住具体行，开销更大但并发能力强。InnoDB 支持行锁，MyISAM 主要是表锁。需要注意 InnoDB 行锁依赖索引，如果条件没有命中索引，可能导致锁范围扩大。

### 乐观锁和悲观锁有什么区别？

悲观锁认为并发冲突经常发生，读取或修改时直接加锁，例如 `select ... for update`；乐观锁认为冲突较少，提交时通过版本号、时间戳或状态条件校验是否被修改。高并发扣减库存常用版本号或状态机做乐观控制。

## 索引与存储引擎

### 为什么数据库索引常用 B+ 树？

B+ 树层级低、磁盘 IO 次数少，叶子节点按顺序连接，适合范围查询和排序。相比普通二叉树或哈希结构，B+ 树更适合磁盘存储和范围检索。

### 什么情况下索引会失效？

常见情况包括对索引列做函数或计算、隐式类型转换、最左前缀不满足、前置模糊匹配、OR 条件使用不当、低区分度字段不适合单独建索引等。

### 如何验证索引是否满足需求？

使用 `EXPLAIN` 查看执行计划，重点关注 `type`、`key`、`rows`、`Extra` 等字段，判断是否使用目标索引、扫描行数是否合理、是否出现临时表和文件排序。必要时结合慢查询日志和真实数据分布分析。

### MySQL 常用存储引擎有哪些？

常见存储引擎包括 InnoDB 和 MyISAM。InnoDB 支持事务、行锁、崩溃恢复和外键，是当前默认和主流选择；MyISAM 不支持事务，主要使用表锁，适合历史上的读多写少场景，现在新项目一般不推荐。

## 排查与优化

### 慢 SQL 怎么排查？

先通过慢查询日志定位 SQL，再用 `EXPLAIN` 查看执行计划，重点关注索引使用、扫描行数、排序、临时表和回表情况。最后结合业务场景优化索引、SQL 写法或数据结构。

### MySQL 问题排查常用手段有哪些？

常用手段包括 `show processlist` 查看当前连接和执行 SQL，慢查询日志定位慢 SQL，`EXPLAIN` 分析执行计划，监控 QPS、TPS、连接数、锁等待、Buffer Pool 命中率等指标。复杂问题还要结合业务发布、数据增长和硬件资源分析。

### MySQL 性能优化有哪些方向？

优化方向包括合理建索引、避免 `select *`、减少回表、优化分页、拆分大事务、控制 SQL 复杂度、选择合适字段类型、分库分表、读写分离和缓存热点数据。不要只看 SQL，还要结合表结构、数据量、业务访问模式和硬件资源。
