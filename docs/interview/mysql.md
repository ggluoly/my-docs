# MySQL 面试题

本页整理 MySQL 高频面试题，覆盖范式、事务、索引、锁、存储引擎和性能排查。回答时建议结合 InnoDB 当前主流特性，遇到旧版本行为要说明版本差异。

## 数据库基础

### 数据库三范式是什么？

第一范式要求字段具有原子性，不能再拆分；
第二范式要求非主属性完全依赖主键，避免部分依赖；
第三范式要求非主属性不依赖其他非主属性，避免传递依赖。
实际业务中会在规范化和查询性能之间做权衡，必要时适度反范式。

#### 常见追问

- 为什么实际项目会反范式设计？
  主要是为了性能和查询效率。
  范式化设计可以减少数据冗余，保证数据一致性，但表拆得越细，查询时 join 越多，复杂查询和高并发场景下性能压力会比较大。
  所以有些场景会适当冗余字段，比如订单表里冗余用户昵称、商品名称、商品价格，查询订单列表时就不用频繁关联用户表、商品表。
- 订单表为什么会冗余商品名称？
  一个是提高查询效率，订单列表、订单详情经常要展示商品名称，如果每次都关联商品表，查询成本会更高。
  另一个是保留交易快照。订单产生时的商品名称、价格、规格等信息应该代表下单那一刻的状态，后面商品改名、改价、下架，都不应该影响历史订单展示。
- 范式越高越好吗？
  不是范式越高越好。
  范式越高，数据冗余越少，一致性更好，但表会拆得更细，查询时关联更多，复杂度和性能成本也会上升。

### 如何获取 MySQL 版本？

可以执行 `select version();` 查看数据库版本，也可以通过 `show variables like 'version%';` 查看更详细的版本变量。排查兼容性问题时，版本信息很关键，例如 MySQL 5.7 和 8.0 在自增、窗口函数、JSON、索引优化等方面都有差异。

#### 常见追问

- MySQL 5.7 和 8.0 有哪些差异？
  MySQL 8.0 相比 5.7 主要差异有这些：
    第一，8.0 默认字符集从 latin1 变成了 utf8mb4，对中文和 emoji 支持更友好。
    第二，8.0 引入了窗口函数、CTE 递归查询，复杂 SQL 表达能力比 5.7 强很多。
    第三，8.0 数据字典做了重构，元数据统一存储到 InnoDB，DDL 原子性更好。
    第四，8.0 优化器能力更强，比如直方图、降序索引、隐藏索引等，对查询优化支持更好；其中 functional key parts（常被称为函数索引）从 MySQL 8.0.13 才支持，5.7 通常需要通过生成列再建索引实现类似效果。
    第五，8.0 权限体系增强，支持角色 role，账号权限管理更方便。
    第六，8.0 默认认证插件变成 caching_sha2_password，老客户端连接可能需要调整驱动或认证方式。
    第七，8.0 移除了 query cache，5.7 里虽然有但实际高并发下容易成为瓶颈。
    实际升级时，比较关注字符集、SQL 兼容性、认证插件、保留关键字变化，以及执行计划是否发生变化。
- 为什么线上排查要先确认版本？
  因为很多问题和具体版本强相关。
    比如 MySQL 5.7 和 8.0 在默认字符集、认证插件、优化器、SQL 特性、关键字、执行计划上都有差异，同一条 SQL 在不同版本下表现可能不一样。
    另外一些问题可能是某个版本的已知 bug，或者某个版本已经修复了，先确认版本可以快速判断是不是版本特性、兼容性问题或者已知缺陷。
    排查性能问题时也一样，不同版本的优化器策略、索引能力、统计信息机制可能不同，不能直接按另一个版本的经验下结论。
    所以线上排查一般先确认版本、配置和环境，避免方向判断错。
- 如何查看存储引擎？
    ```sql
    SHOW ENGINES;
    ```
    这个可以查看当前 MySQL 支持哪些存储引擎，以及默认存储引擎是哪一个。
    查看某张表使用的存储引擎：
    ```sql
    SHOW TABLE STATUS LIKE '表名';
    ```
    结果里的 `Engine` 字段就是该表的存储引擎。
    也可以用：
    ```sql
    SHOW CREATE TABLE 表名;
    ```
    建表语句里会显示 `ENGINE=InnoDB` 这类信息。
    如果想看默认存储引擎：
    ```sql
    SHOW VARIABLES LIKE 'default_storage_engine';
    ```
    MySQL 5.7 和 8.0 默认一般都是 InnoDB。

### char 和 varchar 有什么区别？

`char` 是定长字符串，适合长度固定的数据，比如状态码、MD5；`varchar` 是变长字符串，适合长度变化较大的数据。
`char` 可能浪费空间但读取简单，`varchar` 更节省空间但需要额外记录长度。

#### 常见追问

- 手机号应该用 char 还是 varchar？
  手机号一般建议用 `char(11)`，因为手机号长度固定是 11 位，用 `char` 比较符合定长字段的特点。
    不过如果考虑国际号码、带区号、扩展格式，建议用 `varchar`，比如 `varchar(20)`。
    手机号不建议用数字类型，因为它不是用来计算的，而且可能涉及前导 0、格式扩展等问题。
    所以国内固定 11 位手机号可以用 `char(11)`，如果业务有国际化或格式变化可能，就用 `varchar`。
- varchar 长度是不是越大越好？
  varchar 长度不是越大越好。
    虽然 varchar 是变长存储，实际只按内容长度占空间，但定义长度过大会影响内存临时表、排序、索引长度等场景，MySQL 可能会按最大长度评估资源。
    另外长度定义太大也会降低字段约束能力，容易存入不符合业务预期的数据。
    所以 varchar 长度应该按业务实际最大值设计，适当预留，不建议无脑写很大，比如直接 varchar(255) 或更大。
- 字符集会影响存储空间吗？
  会影响。字符集决定了一个字符最多占多少字节，所以会影响 char、varchar、text 这类字符类型的存储空间。
  比如在 MySQL 里，latin1 一个字符最多 1 个字节，utf8 最多 3 个字节，utf8mb4 最多 4 个字节。

### 自增 ID 删除后重启再插入会怎样？

这个问题和存储引擎、MySQL 版本有关。
**MyISAM** 会把自增值持久化到表文件；
早期 InnoDB 的自增计数可能在重启后根据表中最大值重新计算；MySQL 8.0 起 **InnoDB** 自增计数持久化，重启后不会简单回退到最大值。面试时要说明不能脱离版本给固定答案。

#### 常见追问

- 自增 ID 为什么可能不连续？
    常见原因有几个：
    1. 事务回滚：自增值分配出去后，即使事务回滚，ID 也不会退回。
    2. 插入失败：比如唯一键冲突、字段校验失败，也可能已经消耗了自增值。
    3. 批量插入：MySQL 可能一次预分配多个自增值，部分失败就会产生空洞。
    4. 删除数据：已经存在的记录被删除后，原来的 ID 不会复用。
    5. MySQL 重启或不同自增锁模式下，也可能导致自增值跳号。
    所以 `AUTO_INCREMENT` 生成的值通常按分配序列增长，但不保证连续，也不应作为跨版本、跨重启或按事务提交顺序严格单调的业务编号。唯一性应由 `PRIMARY KEY` 或 `UNIQUE` 约束保证；如果业务要求连续编号，需要单独设计业务编号。
- 分布式系统为什么不一定用数据库自增 ID？
  分布式系统不一定适合用数据库自增 ID，主要是因为它依赖单库生成。
    如果多个库、多个节点同时写入，自增 ID 很难保证全局唯一，除非集中到一个库生成，这样又会有单点问题和性能瓶颈。
    另外分库分表后，每个表都有自己的自增序列，容易产生 ID 冲突，需要额外设置步长或号段，维护成本比较高。
    数据库自增 ID 还会暴露数据增长趋势，比如通过 ID 大小大致猜到订单量。
    所以分布式系统里更常用**雪花算法、号段模式、UUID** 这类全局 ID 方案，根据业务在性能、趋势递增、可读性和全局唯一之间取舍。
- InnoDB 自增锁是什么？
  InnoDB 自增锁是 InnoDB 为了生成 `AUTO_INCREMENT` 值使用的一种特殊表级锁，主要作用是保证并发插入时自增 ID 分配的唯一性和有序性。
    它不是普通的行锁，因为自增值属于表级资源，多个事务同时插入同一张表时，需要协调自增值的分配。
    InnoDB 里和 `innodb_autoinc_lock_mode` 有关：
    1. `0`：传统模式，插入时基本都会持有自增锁到语句结束，并发能力较差。
    2. `1`：连续模式，普通 insert 会用轻量互斥量，批量 insert 这类不能提前确定行数的语句仍可能用自增锁，这是 MySQL 5.7 常见默认模式。
    3. `2`：交错模式，并发性能最好。并发的 INSERT-like 语句之间可能交错分配自增值；行数预先可知的简单多行 INSERT，同一条语句通常仍能获得连续值，而 `INSERT ... SELECT`、`LOAD DATA`、混合模式插入等则可能产生间隔。MySQL 8.0 默认是这个模式。
    所以自增锁**本质上是为了协调自增 ID 分配**，不是为了保证事务回滚后 ID 连续，自增 ID 分配出去后通常不会回退。

## 事务与锁

### ACID 是什么？

ACID 分别是原子性、一致性、隔离性和持久性。
原子性保证事务内操作要么都成功要么都失败；一致性保证数据从一个一致状态转到另一个一致状态；隔离性控制并发事务互相影响；持久性保证提交后的数据不会因为宕机丢失。

#### 常见追问

- 原子性靠什么实现？
    InnoDB 事务的原子性主要靠 undo log 实现。
    事务执行过程中，修改数据前会先记录对应的 undo log，里面保存回滚所需的信息。事务如果失败或者主动回滚，就根据 undo log 把数据恢复到修改前的状态。
    所以原子性保证的是事务里的操作要么全部成功，要么全部失败，不会只执行一部分。redo log 主要保证持久性，锁主要保证隔离性。
- 持久性和 redo log 有什么关系？
  持久性主要靠 redo log 保证。
    执行数据页修改时，InnoDB 会生成 redo 并写入 redo log buffer；事务提交时，再按 `innodb_flush_log_at_trx_commit` 的策略将相关 redo 写出或刷盘，而不是必须立刻把脏页刷到磁盘。只要 redo 按策略持久化，即使数据库宕机，重启后也可以根据 redo log 把已提交事务的修改恢复出来。
    所以 redo log 解决的是“已提交事务不能丢”的问题。数据页可以延迟刷盘，但相关 redo 必须先满足 WAL 要求；具体持久性强度取决于 `innodb_flush_log_at_trx_commit` 等参数配置。
- 隔离性和锁 / MVCC 有什么关系？
  隔离性主要靠锁和 MVCC 一起实现。
    锁主要解决并发写的问题，比如两个事务同时修改同一行时，通过行锁保证同一时间只有一个事务能改，避免写写冲突。
    MVCC 主要解决读写并发的问题，普通快照读不加锁，而是通过 undo log 版本链和 Read View 读取符合当前事务可见性的历史版本，这样读和写可以并发执行。
    在 InnoDB 里，不同隔离级别下表现不一样：
    1. 读已提交：每次快照读都会生成新的 Read View，所以能读到其他事务已经提交的数据。
    2. 可重复读：事务第一次快照读生成 Read View，后续复用，所以同一事务内多次读取结果一致。
    3. 当前读：比如 `select ... for update`、`update`、`delete`，读的是最新数据，需要加锁；在可重复读下还会用 next-key lock 防止幻读。
    所以简单说，锁保证写和当前读的隔离，MVCC 保证快照读的隔离。

### 事务隔离级别有哪些？

事务隔离级别包括读未提交、读已提交、可重复读和串行化。MySQL InnoDB 默认可重复读，并通过 MVCC 和锁机制解决多数并发读写问题。

#### 常见追问

- 脏读、不可重复读、幻读分别是什么？
  脏读是一个事务读到了另一个事务还没提交的数据，如果对方回滚了，那读到的数据就是无效的。
    不可重复读是一个事务内对同一行数据读了两次，期间另一个事务提交了修改，导致两次读取结果不一致，重点是“同一行数据被修改”。
    幻读是一个事务内按同一个条件查询两次，期间另一个事务插入或删除了符合条件的数据，导致第二次查询多出或少了记录，重点是“数据条数变化”。
    对应隔离级别上，读未提交可能有脏读；读已提交解决脏读，但可能有不可重复读和幻读；可重复读解决不可重复读，InnoDB 在可重复读下通过 MVCC 和 next-key lock 基本解决幻读；串行化隔离级别最强，但并发性能最低。
- MySQL 可重复读如何处理幻读？
  MySQL InnoDB 在可重复读下主要分两种情况处理幻读。
    普通快照读，比如普通 `select`，通过 MVCC 解决。事务第一次查询时生成 Read View，后续查询复用这个 Read View，所以其他事务新插入并提交的数据，对当前事务不可见，看起来就不会出现幻读。
    当前读，比如 `select ... for update`、`update`、`delete`，读的是最新数据，不能只靠 MVCC，所以 InnoDB 会通过 next-key lock 处理，也就是行锁加间隙锁，锁住命中的记录以及范围间隙，防止其他事务在这个范围内插入新数据。
    所以总结就是：**快照读靠 MVCC 避免幻读，当前读靠 next-key lock 防止幻读**。
- 隔离级别越高越好吗？
  不一定。隔离级别越高，一致性越强、并发异常越少，但通常锁范围更大、锁持有时间更长，并发性能更低，也更容易发生锁等待或死锁。
    实际要根据业务选择：
    - 读未提交：隔离性太弱，通常不使用。
    - 读已提交：并发性较好，能避免脏读，适合大多数普通业务。
    - 可重复读：能保证事务内重复读取结果一致；MySQL InnoDB 默认使用，适合需要较强一致性的业务。
    - 串行化：隔离性最强，但并发最低，只有对一致性要求极高且并发量较低的场景才考虑。
    一般不是追求最高隔离级别，而是在数据一致性和并发性能之间做权衡。

### MVCC 是什么？

MVCC 是多版本并发控制，通过保存数据版本和 Read View，让读写在多数场景下互不阻塞，提高并发性能。它是 InnoDB 实现一致性读的重要机制。

#### 常见追问

- Read View 包含哪些信息？
    InnoDB 的 Read View 主要包含以下信息：
    - `m_ids`：生成 Read View 时，当前活跃、尚未提交的事务 ID 集合。
    - `min_trx_id`：活跃事务中最小的事务 ID。
    - `max_trx_id`：生成 Read View 时，下一个将要分配的事务 ID，事务 ID 大于等于它的事务都不可见。
    - `creator_trx_id`：创建该 Read View 的事务 ID。
    通过这些信息结合数据行的事务 ID，判断某个版本对当前事务是否可见。
- 当前读和快照读有什么区别？
    **快照读**读取的是历史版本数据，基于 MVCC 和 Read View 实现，读取过程中通常不加锁。普通 `select` 属于快照读，在可重复读隔离级别下，同一事务内多次查询通常读取同一个一致性视图。
    **当前读**读取的是最新已提交的数据，并且会加锁或等待锁，用于保证后续更新的正确性。`select ... for update`、`select ... lock in share mode`、`update`、`delete` 都属于当前读。
    在 InnoDB 可重复读隔离级别下，快照读主要通过 MVCC 避免幻读；当前读主要通过 next-key lock，也就是记录锁加间隙锁，防止范围内插入新记录。
- MVCC 是否完全不需要锁？
    不是。MVCC 只是让普通快照读可以不加锁地读取历史版本数据，减少读写之间的阻塞，并不是完全不需要锁。
    在 InnoDB 中，普通 `select` 通常走快照读，通过 Read View 和 undo log 判断版本可见性，一般不加锁。
    但如果是当前读，比如 `select ... for update`、`update`、`delete`，还是需要加锁的，因为要读取最新数据并保证后续修改的安全。
    另外，写写之间也不能靠 MVCC 解决，多个事务更新同一行时仍然需要行锁来保证数据一致性。
    所以 **MVCC 的作用是提高并发能力，主要解决读写冲突，但不能完全替代锁**。

### 行锁和表锁有什么区别？

表锁锁住整张表，开销小但并发能力差；
行锁锁住具体行，开销更大但并发能力强。
InnoDB 支持行锁，MyISAM 主要是表锁。需要注意 InnoDB 行锁依赖索引，如果条件没有命中索引，可能导致锁范围扩大。

#### 常见追问

- 行锁一定只锁一行吗？
    不一定。InnoDB 里通常说的行锁是基于索引加锁的，不是简单理解为只锁某一行。
    如果是等值命中唯一索引，一般退化成记录锁，可能只锁一条记录。
    但如果是范围查询、非唯一索引查询，或者在可重复读隔离级别下防止幻读，可能会加 next-key lock，也就是记录锁加间隙锁，锁住一段范围。
    如果查询条件没有走索引，InnoDB 可能扫描大量记录并对扫描到的索引记录加锁，看起来就像锁了很多行，甚至影响范围很大。
    所以**行锁不代表一定只锁一行，实际锁多少取决于索引、查询条件、隔离级别和 SQL 类型**。
- 什么是间隙锁？
    间隙锁是 InnoDB 在索引记录之间的“间隙”上加的锁，它锁的不是某一条已经存在的记录，而是一个范围，主要目的是防止其他事务在这个范围内插入新数据，从而避免幻读。
    它通常出现在可重复读隔离级别下的当前读中，比如范围查询、`select ... for update`、`update`、`delete` 等场景。
    间隙锁只限制其他事务往这个间隙里插入数据，一般不影响读取已有记录，也不影响修改间隙外的数据。
    实际使用中，间隙锁经常和记录锁组合成 next-key lock，也就是锁住某条索引记录以及它前面的间隙。
- 为什么索引失效可能导致锁范围变大？
    因为 InnoDB 的行锁是加在索引上的，不是直接加在表记录上的。
    如果查询条件能走索引，InnoDB 可以定位到少量索引记录，只锁相关记录或范围。
    但如果索引失效，SQL 可能变成全表扫描或大范围扫描，InnoDB 在执行 `update`、`delete`、`select ... for update` 这类当前读时，会对扫描到的索引记录加锁。
    在可重复读隔离级别下，还可能加 next-key lock，也就是记录锁加间隙锁，导致锁住的范围更大。
    **所以索引失效后，并不是一定直接变表锁，而是因为扫描范围变大，加锁范围也随之变大，看起来就像锁了很多行，甚至接近锁表**。

### 乐观锁和悲观锁有什么区别？

悲观锁认为并发冲突经常发生，读取或修改时直接加锁，例如 `select ... for update`；
乐观锁认为冲突较少，提交时通过版本号、时间戳或状态条件校验是否被修改。高并发扣减库存常用版本号或状态机做乐观控制。

#### 常见追问

- 乐观锁更新失败怎么办？
  乐观锁更新失败一般说明数据已经被其他事务改过了，当前提交基于旧版本，不能直接覆盖。
  处理方式要看业务语义：库存扣减、计数器等可自动重试的场景，可以重新读取最新版本后做有限次数重试；表单编辑、价格修改等需要人工确认的场景，应返回冲突并提示用户刷新后再提交；异步任务可通过消息或延迟任务稍后重试。
  不能无上限直接重试，否则高冲突场景会持续消耗数据库和应用线程；应设置最大重试次数、退避策略和失败后的降级或补偿方案。
- 悲观锁会带来什么问题？
  悲观锁的主要问题是并发性能会下降，因为它在操作前就先加锁，其他事务或线程只能等待。
  所以悲观锁适合并发冲突比较高、必须强一致的场景，但要控制事务时间、加锁范围和访问顺序。
- 数据库锁和 Java 锁有什么区别？
    数据库锁和 Java 锁主要区别在锁的对象和作用范围不同。
    Java 锁锁的是 JVM 进程内的共享资源，比如对象、代码块、内存变量，主要解决多线程并发问题，常见的有 `synchronized`、`ReentrantLock`。
    数据库锁锁的是数据库里的数据资源，比如行、间隙、表，主要解决多个事务并发读写数据的一致性问题。
    Java 锁一般只在当前 JVM 内生效，多个服务实例之间不共享；数据库锁是数据库层面的，只要访问同一份数据，多个应用实例都会受影响。
    生命周期也不同，Java 锁通常随着代码块执行结束释放；数据库锁一般跟事务绑定，事务提交或回滚后释放。
    另外，Java 锁关注线程安全和内存可见性，数据库锁关注事务隔离性、数据一致性和并发控制。

## 索引与存储引擎

### 为什么数据库索引常用 B+ 树？

B+ 树层级低、磁盘 IO 次数少，叶子节点按顺序连接，适合范围查询和排序。相比普通二叉树或哈希结构，B+ 树更适合磁盘存储和范围检索。

#### 常见追问

- B 树和 B+ 树有什么区别？
    B 树和 B+ 树主要区别在数据存放位置和查询方式。
    B 树的非叶子节点和叶子节点都可以存放 key 和数据，所以查找时可能在非叶子节点就命中。
    B+ 树只有叶子节点存放完整数据，非叶子节点只存 key 和指针，所有查询最终都会走到叶子节点。
    B+ 树的叶子节点之间一般通过链表连接，所以范围查询更高效，适合数据库索引。
    另外，B+ 树非叶子节点不存数据，单个节点能存更多 key，树的高度通常更低，磁盘 IO 次数也更少。
    所以数据库索引，比如 MySQL InnoDB，通常更适合用 B+ 树。
- 为什么不用红黑树？
    数据库索引不用红黑树，主要是因为红黑树是二叉树，树高相对 B+ 树更高。
    数据库索引通常存储在磁盘上，查询性能主要受磁盘 IO 次数影响。红黑树每个节点只存少量 key，查一次可能要经过很多层节点，磁盘 IO 会比较多。
    B+ 树是多叉树，一个节点可以存很多 key，树高很低，通常 3 到 4 层就能定位大量数据，所以 IO 次数更少。
    另外，B+ 树的叶子节点有链表，范围查询、排序扫描更方便；红黑树做范围查询需要中序遍历，磁盘访问不够连续。
    所以红黑树适合内存里的有序结构，比如 JDK8 的 `TreeMap`，而数据库索引更适合用 B+ 树。
- 聚簇索引和非聚簇索引有什么区别？
    聚簇索引和非聚簇索引的核心区别是：数据行是不是和索引放在一起。
    聚簇索引的叶子节点存的是完整数据行，所以通过聚簇索引查到叶子节点就能拿到数据。InnoDB 里主键索引就是聚簇索引。
    非聚簇索引的叶子节点不存完整数据行，一般存的是索引列和主键值。通过非聚簇索引查到主键后，如果查询字段不在索引里，还需要再根据主键回到聚簇索引查一次，这个过程叫回表。
    所以聚簇索引查询主键范围数据效率高，因为数据本身按主键顺序组织；非聚簇索引适合按其他字段加速查询，但可能会有回表成本。
    另外，一张表通常只能有一个聚簇索引，因为数据只能按一种顺序存放；非聚簇索引可以有多个。

### 什么情况下索引会失效？

常见情况包括对索引列做函数或计算、隐式类型转换、最左前缀不满足、前置模糊匹配、OR 条件使用不当、低区分度字段不适合单独建索引等。

#### 常见追问

- `like '%abc'` 为什么难走索引？
    `like '%abc'` 难走索引，是因为 B+ 树索引是按字段值从左到右有序组织的，只能利用最左前缀来定位范围。
    如果是 `like 'abc%'`，可以定位到以 `abc` 开头的一段连续索引范围，所以可以走索引。
    但 `like '%abc'` 前面有通配符，开头不确定，数据库没法根据索引的有序性定位起始位置，只能逐个判断每个值是否以 `abc` 结尾，所以通常会全表扫描或全索引扫描。
    如果确实有后缀匹配需求，可以考虑反转字段建索引、全文索引，或者用搜索引擎来处理。
- 联合索引最左前缀是什么？
    联合索引最左前缀原则指的是：联合索引按索引字段从左到右依次排序，查询条件通常需要从最左边的字段开始连续使用，才能高效进行常规索引定位。
    比如有联合索引 `(a, b, c)`，条件里有 `a`，或者 `a、b`，或者 `a、b、c`，都可以利用索引。
    如果只查 `b`、只查 `c`，或者查 `b、c`，MySQL 5.7 通常难以利用这个联合索引进行前缀定位；MySQL 8.0.13+ 在特定条件下可能使用 Index Skip Scan，或者将索引用于覆盖扫描，最终应以 `EXPLAIN` 为准。
    另外，如果中间断了也会影响后面的字段使用，比如只查 `a、c`，通常只能用到 `a`，`c` 不能继续利用联合索引的有序性。
    还有一种情况是范围查询，比如 `a = ? and b > ? and c = ?`，一般索引能用到 `a、b`，但 `c` 往往不能继续用于索引定位。
- 隐式转换为什么危险？
    隐式转换危险主要有两个方面。
    第一是可能导致索引失效，比如字段是 `varchar`，SQL 里用数字去比较，数据库可能会把字段值转换成数字再比较，这样对列做了函数计算，B+ 树的有序性就用不上了，容易变成全表扫描。
    第二是可能导致结果不符合预期，比如字符串转数字时，`'123abc'` 可能会被转成 `123`，不同数据库的转换规则也不一样，容易产生隐藏 bug。
    所以写 SQL 时，参数类型要和字段类型保持一致，尤其是 Java 里传参时不要把字符串类型的字段用数字类型去绑定。

### 如何验证索引是否满足需求？

使用 `EXPLAIN` 查看执行计划，重点关注 `type`、`key`、`rows`、`Extra` 等字段，判断是否使用目标索引、扫描行数是否合理、是否出现临时表和文件排序。必要时结合慢查询日志和真实数据分布分析。

#### 常见追问

- `type` 中哪些值比较好？
    `type` 表示表的访问方式，不是脱离 SQL、数据分布和成本模型的统一性能排名。
    `const`、`eq_ref`、`ref` 通常选择性较好，`range` 是常见的范围访问方式；但实际优劣仍要结合扫描行数、返回行数、覆盖索引、回表成本和实际耗时判断。
    `const` 一般是主键或唯一索引等值查询，只匹配一行，性能很好。
    `eq_ref` 常见于 join 时，被驱动表通过主键或唯一索引关联，每次只匹配一行，也很好。
    `ref` 是普通非唯一索引等值查询，能命中索引，效果也不错。
    `range` 是索引范围查询，比如 `between、>、<、like 'abc%'`，也可以接受。
    `index` 表示全索引扫描；如果是覆盖索引且索引更窄，可能优于 `ALL`，但大量回表时也可能成本更高。
    `ALL` 是全表扫描；在小表、低选择性条件或本来就需要读取大部分数据时，可能是优化器的合理选择。
    所以面试里应说：出现 `index` 或 `ALL` 时，要结合数据量、过滤条件、`rows`、返回行数和实际耗时分析，不能为了消除 `ALL` 盲目加索引。
- `Using filesort` 一定很差吗？
    `Using filesort` 不一定很差，它只是表示排序没有完全利用索引顺序，需要 MySQL 额外做排序，不代表一定会落盘，也不代表一定慢。
    如果过滤后数据量很小，或者有 `limit`，即使出现 `Using filesort`，性能也可能完全可以接受。
    真正要判断好不好，要结合 `type`、`key`、`rows` 看扫描范围大不大，结合慢查询耗时、返回行数、数据分布来看。
    比如扫描几百行再排序问题不大，但如果扫描几十万、几百万行再排序，尤其还伴随 `Using temporary`，性能风险就比较高。
    优化时可以考虑让 `where` 和 `order by` 匹配联合索引，减少扫描行数，或者避免大结果集排序。
- `rows` 不准怎么办？
    `rows` 本身就是优化器基于统计信息估算出来的，不是精确值，不准很常见，尤其是数据分布不均、统计信息过旧、条件相关性强的时候。
    处理上首先可以执行 `ANALYZE TABLE` 更新表和索引统计信息，让优化器重新评估扫描行数。
    如果是数据倾斜比较严重，比如某些状态值特别集中，单纯统计信息可能还是不准，这时要结合真实数据分布看索引是否合适，必要时调整联合索引顺序，尽量把区分度高、过滤效果好的字段放到合适位置。
    如果优化器因为估算错误选错索引，可以先通过改写 SQL、补充更合适的索引来解决，最后才考虑 `FORCE INDEX` 这类手段。
    MySQL 8.0.18+ 还可以用 `EXPLAIN ANALYZE` 看实际执行行数和耗时，用来对比 `rows` 估算是否偏差很大；MySQL 5.7 和更早的 8.0 小版本需结合慢日志、`EXPLAIN` 和运行时指标判断。

### MySQL 常用存储引擎有哪些？

常见存储引擎包括 InnoDB 和 MyISAM。
InnoDB 支持事务、行锁、崩溃恢复和外键，是当前默认和主流选择；MyISAM 不支持事务，主要使用表锁，适合历史上的读多写少场景，现在新项目一般不推荐。

#### 常见追问

- InnoDB 为什么是默认选择？
    InnoDB 作为默认存储引擎，主要是因为它更适合大多数业务场景。
    它支持事务，满足 ACID；支持行级锁和 MVCC，并发性能比 MyISAM 这种表锁引擎更好；支持崩溃恢复，通过 redo log、undo log 能保证数据一致性和可靠性。
    另外 InnoDB 支持外键、聚簇索引、缓冲池，对高并发读写和数据安全要求比较高的业务更友好。
    所以一般业务表默认都会选 InnoDB，除非是非常特殊的只读、归档或全文检索场景，才会考虑其他存储引擎。
- MyISAM 为什么不支持事务？
    MyISAM 不支持事务主要是因为它的设计目标不是事务一致性，而是结构简单、读性能和存储效率。
    事务需要依赖 undo log 做回滚，redo log 做崩溃恢复，还需要配合锁机制、隔离级别等能力保证 ACID。但 MyISAM 没有这些事务日志机制，也不支持 MVCC，锁粒度主要是表级锁。
    所以 MyISAM 的写操作一旦执行，不能像 InnoDB 那样回滚；如果宕机，也只能做表修复，不能保证事务级别的一致性。
    因此涉及资金、订单、库存这类对一致性要求高的业务，一般不能用 MyISAM，要用 InnoDB。

## 排查与优化

### 慢 SQL 怎么排查？

先通过慢查询日志定位 SQL，再用 `EXPLAIN` 查看执行计划，重点关注索引使用、扫描行数、排序、临时表和回表情况。最后结合业务场景优化索引、SQL 写法或数据结构。

#### 常见追问

- 慢查询阈值怎么设置？
    慢查询阈值通过 `long_query_time` 设置，表示 SQL 执行时间超过多少秒会记录到慢查询日志。
    一般不建议直接用默认的 `10s`，这个值太大，很多有问题的 SQL 可能发现不了。实际设置要结合业务要求和数据库压力，比如核心接口要求比较高，可以设成 `0.5s` 或 `1s`；普通业务可以先设 `1s~2s`，再根据慢日志数量和系统负载调整。
    可以临时设置当前连接：
    `SET SESSION long_query_time = 1;`
    也可以设置全局默认值：
    `SET GLOBAL long_query_time = 1;`
    全局设置主要影响后续新建连接，当前已连接会话应单独设置 Session 值后再验证。
    也可以写到 MySQL 配置文件里长期生效：
    `slow_query_log = ON`
    `long_query_time = 1`
    `log_queries_not_using_indexes = ON`
    不过 `log_queries_not_using_indexes` 要谨慎开，可能产生大量日志。
    设置完以后重点不是只看阈值，而是持续分析慢查询日志，比如结合 `EXPLAIN`、索引、扫描行数、锁等待和数据量变化来优化；生产变更还要评估慢日志量和磁盘容量。
- 如何排查偶发慢 SQL？
    排查偶发慢 SQL，我一般会**先看慢查询日志**里这条 SQL 的执行时间、扫描行数、返回行数、是否有锁等待等信息，确认慢是执行计划问题、数据量问题，还是等待问题。
    **然后用 `EXPLAIN` 看执行计划**，重点看有没有走索引、索引是否合适、扫描行数是否异常、是否出现 filesort、temporary。
    如果平时快、偶发慢，还要重点**排查运行时因素**，比如当时是否有锁竞争、大事务未提交、后台任务、批量写入、DDL、主从延迟、Buffer Pool 命中率下降、磁盘 IO 抖动等。
    可以结合 `show processlist`、`show engine innodb status`、慢日志、监控指标一起看，判断当时是不是被锁住或者资源不够。
    如果是参数不同导致执行计划变化，还要**检查 SQL 是否存在隐式转换、索引选择不稳定、统计信息不准**，可以考虑更新统计信息或调整索引。
    最后根据原因处理，比如补充合适索引、拆分大事务、避免高峰跑批、优化 SQL 写法，必要时做限流或缓存。
- SQL 优化一定是加索引吗？
    不是，SQL 优化不一定是加索引。加索引只是常见手段，但不是唯一手段，而且索引也会增加写入成本和存储成本。
    SQL 优化一般会**先看执行计划**，判断慢的原因。比如是否全表扫描、是否扫描行数太多、是否有排序、临时表、回表过多、锁等待等。
    如果是查询条件缺少合适索引，可以考虑**加联合索引**；如果是 SQL 写法导致索引失效，比如函数、隐式转换、前置模糊匹配，就要先**改写 SQL**。
    有些场景要**优化表结构或业务逻辑**，比如减少返回字段、避免 `select *`、分页深度太大改成游标分页、大 SQL 拆小 SQL、冷热数据拆分、分库分表、加缓存等。
    所以 SQL 优化的核心不是盲目加索引，而是先定位瓶颈，再选择合适手段。

### MySQL 问题排查常用手段有哪些？

常用手段包括 `show processlist` 查看当前连接和执行 SQL，
慢查询日志定位慢 SQL，
`EXPLAIN` 分析执行计划，监控 QPS、TPS、连接数、锁等待、Buffer Pool 命中率等指标。
复杂问题还要结合业务发布、数据增长和硬件资源分析。

#### 常见追问

- `show processlist` 中 Sleep 很多怎么办？
    `show processlist` 里很多 `Sleep` 不一定是问题，`Sleep` 表示连接空闲，正在等客户端下一次请求。
    排查时先看数量和持续时间，如果没有接近 `max_connections`，CPU、内存、IO 也正常，一般不用急着处理。
    如果 `Sleep` 很多并且时间很长，要重点看是不是连接池配置不合理，比如最小空闲连接太大、最大连接数设置太高，或者应用代码连接没有正常关闭，导致连接泄漏。
    还要排查有没有空闲连接占着事务不提交，可以通过 `information_schema.innodb_trx` 结合线程 id 看，如果 `Sleep` 连接还持有事务或锁，这种就要重点处理。
    处理方式一般是：应用侧保证连接用完归还连接池，调整连接池的 `maxPoolSize`、`minIdle`、`idleTimeout`、`maxLifetime`；数据库侧可以适当调低 `wait_timeout`，让长时间空闲连接自动断开。
    如果线上已经影响连接数，可以临时 `kill` 掉长时间空闲且确认无事务的连接，但根因还是要从连接池配置和代码释放连接上解决。
- 如何排查锁等待？
    排查锁等待一般先看当前有没有被阻塞的 SQL，可以用 `show processlist` 看哪些线程处于 `Waiting for...`、`Locked` 或执行时间很长。
    然后查事务信息，MySQL 5.7 常用 `information_schema.innodb_trx`、`innodb_locks`、`innodb_lock_waits`，找出等待事务和阻塞事务；MySQL 8 里主要看 `performance_schema.data_locks` 和 `data_lock_waits`。
    定位到阻塞线程后，再看它执行的 SQL、事务开始时间、是否长时间未提交，判断是不是大事务、慢 SQL、业务代码没提交事务，或者更新顺序不一致导致互相等待。
    也可以配合 `show engine innodb status` 看最近的锁等待和死锁信息，里面能看到锁类型、表、索引和相关事务。
    处理上，如果确认阻塞事务异常，可以临时 `kill` 掉对应连接；根因上要缩短事务时间，避免事务里做耗时操作，SQL 尽量走索引，统一更新顺序，降低锁范围，必要时拆分大事务。
- Buffer Pool 命中率低说明什么？
    Buffer Pool 命中率低，一般说明很多数据页没有从内存命中，需要去磁盘读取，可能会导致磁盘 IO 增加、查询变慢。
    常见原因有几类：`innodb_buffer_pool_size` 太小，热点数据放不下；SQL 扫描数据量太大，比如全表扫描、大范围查询；索引不合理导致读了很多无效数据页；还有就是刚重启后 Buffer Pool 还没预热，命中率短时间偏低也正常。
    排查时不能只看命中率，要结合磁盘 IO、慢 SQL、执行计划、`Innodb_buffer_pool_read_requests` 和 `Innodb_buffer_pool_reads` 一起看。
    优化方向一般是：先优化慢 SQL 和索引，减少无效扫描；再评估是否需要调大 `innodb_buffer_pool_size`；如果是报表类大查询影响线上，可以考虑读写分离或拆到离线库。

### MySQL 性能优化有哪些方向？

优化方向包括合理建索引、避免 `select *`、减少回表、优化分页、拆分大事务、控制 SQL 复杂度、选择合适字段类型、分库分表、读写分离和缓存热点数据。不要只看 SQL，还要结合表结构、数据量、业务访问模式和硬件资源。

#### 常见追问
- 什么是回表？
    回表是 InnoDB 里通过普通二级索引查到主键值后，还需要再根据主键到聚簇索引里查完整数据的过程。
    因为 InnoDB 的数据行是存放在主键索引上的，二级索引叶子节点一般只保存索引字段和主键值。如果查询的字段在二级索引里已经都有，就不需要回表，这种叫覆盖索引。
    回表本身不是问题，但如果回表次数很多，会增加随机 IO 和查询成本。优化时一般会考虑让 SQL 走覆盖索引，或者减少返回字段，避免 `select *`。
- 为什么不推荐 `select *`？
    不推荐 `select *` 主要是因为会把不需要的字段也查出来，增加磁盘 IO、内存占用和网络传输开销。
    在 InnoDB 里，如果只查索引里的字段，可能走覆盖索引；但 `select *` 往往需要回表，查询成本更高。
    另外，表结构变更时，`select *` 可能导致返回字段变多，影响接口兼容性和程序稳定性。
    从可维护性上看，明确写字段也更清晰，能让 SQL 的数据依赖更明确。
    所以实际开发里一般按需查询字段，避免无意义的数据读取。
- 大表分页怎么优化？
    大表分页主要问题是深分页，比如 `limit 100000, 20`，MySQL 需要先扫描并丢弃前面 100000 条，再返回 20 条，offset 越大越慢。
    常见优化方式是改成基于游标的分页，也就是记住上一页最后一条的主键或排序字段，比如 `where id > lastId order by id limit 20`，这样可以利用索引往后查，性能比较稳定。
    如果业务必须跳页，可以用覆盖索引先查出主键，再回表查完整数据，也就是延迟关联，减少扫描宽行数据的成本。
    另外要保证 `where` 条件和 `order by` 字段有合适的联合索引，避免全表扫描和文件排序。
    实际业务里还可以限制最大翻页深度，深层数据用搜索条件筛选，或者把复杂查询放到 ES、缓存、离线报表里处理。
- 什么时候考虑分库分表？
    分库分表一般不是一开始就做，而是在单库单表经过索引优化、SQL 优化、缓存、读写分离、硬件扩容后，仍然扛不住时再考虑。
    常见场景有几个：单表数据量太大，比如达到千万级甚至更高，查询和 DDL 明显变慢；单库写入压力太高，主库 CPU、IO、连接数长期接近瓶颈；数据增长很快，后续容量明显不可控；或者业务天然可以按用户、租户、时间等维度拆分。
    分表主要解决单表数据量过大带来的查询和维护问题，分库主要解决单库容量和并发写入压力问题。
    但分库分表会带来分布式事务、跨库查询、分页排序、全局唯一 ID、扩容迁移等复杂度，所以只有在单库单表确实成为瓶颈，并且有明确拆分维度时才适合做。
