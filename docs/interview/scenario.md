# 场景题

## 如何设计接口幂等？

### 标准回答

接口幂等要求同一次业务请求执行一次和执行多次结果一致。常见方案包括唯一业务号、Token 防重复提交、数据库唯一约束、状态机校验、Redis 去重和 MQ 消费幂等。

## 如何防止重复提交？

### 标准回答

前端按钮置灰只能减少重复点击，不能作为最终保障。后端应使用请求 Token、业务唯一键或 Redis 锁，在服务端保证重复请求不会重复生效。

## 秒杀超卖怎么解决？

### 标准回答

核心是库存扣减必须具备原子性和并发控制。常见方案包括 Redis 预扣库存、Lua 脚本原子执行、消息队列异步下单、数据库乐观锁兜底和限流降级。

## 缓存一致性怎么处理？

### 标准回答

多数业务使用最终一致性。常见做法是先更新数据库再删除缓存，删除失败通过重试或消息队列补偿。强一致需要加锁或同步更新，但性能和复杂度更高。

## 热点 Key 怎么处理？

### 标准回答

可以通过本地缓存、多级缓存、热点 Key 拆分、预热、限流和异步刷新处理。核心是避免单个 Key 或单个实例承受过高流量。
