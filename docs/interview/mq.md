# 消息队列面试题

本页整理 MQ 通用问题、RabbitMQ、Kafka 和 Apache Pulsar 高频面试题。RabbitMQ 内容完整保留为传统消息中间件考点，Kafka 部分会说明 ZooKeeper 和 KRaft 的版本差异，Pulsar 部分重点关注多租户、订阅模式和存储分离。

## MQ 通用问题

### MQ 解决什么问题？

MQ 主要解决异步解耦、削峰填谷、事件驱动和最终一致性问题。它能降低服务之间的同步耦合，但也会引入消息丢失、重复消费、顺序性和一致性等新问题。

#### 常见追问

- 哪些场景不适合引入 MQ？
    不适合引入 MQ 的场景主要有几类。
    第一是**业务必须同步返回结果**，比如用户下单后马上要知道支付是否成功，这种强依赖实时结果的流程不适合强行异步化。
    第二是**强一致性要求特别高的场景**，消息队列一般是最终一致性，引入后要处理消息丢失、重复消费、事务一致性等问题。
    第三是**系统本身并发不高、调用链简单**，引入 MQ 反而会增加架构复杂度和运维成本。
    第四是**对延迟特别敏感的场景**，MQ 会多一层投递和消费链路，极端低延迟业务不一定合适。
    第五是**消费端没有幂等能力，或者业务无法接受重复消息**，这种场景直接上 MQ 风险比较大。
    第六是**消息顺序要求非常严格，而且吞吐量又很高的场景**，虽然 MQ 能做顺序消息，但会牺牲并发能力，设计成本也比较高。
    所以 MQ 更适合削峰、解耦、异步处理和最终一致性的场景，不适合为了用而用。
- MQ 如何削峰？
    MQ 削峰的核心是**把瞬时高并发请求先写入消息队列，让生产速度和消费速度解耦**。
    比如秒杀、活动报名这类场景，请求高峰时业务系统只做必要校验，然后把消息投递到 MQ，快速返回结果；后面的库存扣减、订单创建等操作由消费者按自身处理能力慢慢消费。
    这样数据库不会被瞬时流量直接打满，压力会被 MQ 缓冲下来。
    实现时一般会控制消费者并发数和消费速率，让下游系统按可承受的 QPS 处理。
    同时要注意几个问题：消息不能丢，需要开启持久化和确认机制；消费要做幂等，防止重复消费；队列积压要监控，避免积压过多导致延迟不可接受；如果是库存类场景，还要提前做库存校验或预扣，避免超卖。
- 引入 MQ 后系统复杂度体现在哪？
    引入 MQ 后复杂度主要体现在几个方面。
    首先是**链路变长了**，原来一次同步调用变成生产消息、Broker 存储、消费者消费，排查问题更复杂。
    其次是要**处理消息可靠性**，比如生产失败、消息丢失、确认机制、重试、死信队列这些问题。
    然后是**消费端要考虑幂等**。多数可靠投递方案会按至少一次语义设计，重复消费是常见情况；具体投递保证仍取决于中间件、配置和端到端处理方式。
    还要处理**消息顺序、消息积压、消费延迟、Broker 高可用和容量规划**。
    另外业务一致性也更复杂，比如本地事务成功了但消息发送失败，或者消息消费成功但业务处理失败，需要事务消息、补偿机制或最终一致性方案。
    所以 MQ 能带来解耦和削峰，但也会增加开发、运维和排障成本。
- 什么是幂等？
    幂等就是**同一个操作执行一次和执行多次，最终产生的结果是一样的**。
    在 MQ 场景里，因为消息可能重复投递，消费者就必须保证幂等。
    比如扣库存、创建订单、发优惠券这类操作，如果同一条消息重复消费，不能重复扣、重复建、重复发。
    常见做法是给业务操作加唯一标识，比如订单号、消息 ID，请求进来时先判断是否处理过；也可以用数据库唯一索引、状态机、Redis 去重等方式保证。
    简单说，幂等就是为了防止重复请求或重复消息导致数据被重复修改。

### 消息重复消费怎么办？

消费者必须保证幂等。常见做法包括业务唯一键、去重表、状态机校验、Redis 去重和数据库唯一约束。不要假设 MQ 只投递一次。

#### 常见追问

- 为什么 MQ 会重复投递？
    MQ 会重复投递，核心原因是**可靠投递方案通常采用“至少一次”语义**。
    比如消费者处理完业务后，还没来得及给 Broker 返回 ack 就宕机了，Broker 不确定这条消息是否处理成功，就会重新投递。
    还有可能是网络抖动导致 ack 丢失、消费者消费超时、Broker 重试机制触发、生产者发送失败后重试，都会造成同一条消息被再次投递。
    所以 MQ 很难同时做到高性能、高可用和绝对不重复，一般都是允许重复投递，由消费端通过幂等来保证最终结果正确。
- 消费端幂等放在哪里做？
    消费端幂等一般放在**消费者的业务处理入口做**，也就是收到消息后，真正执行业务逻辑之前先做去重判断。
    通常会基于业务唯一键或者消息唯一 ID 判断这条消息是否已经处理过，如果处理过就直接 ack，不再重复执行业务。
    更稳妥的做法是把幂等判断和业务更新放在同一个本地事务里，比如用数据库唯一索引、状态表、业务状态机来保证原子性。
    如果只是用 Redis 做去重，也要注意过期时间、原子操作以及和数据库更新之间的一致性问题。
    所以幂等不应该只依赖 MQ，本质上要落在消费端的业务层，并且尽量靠近核心数据更新的位置。
- 去重表如何设计？
    去重表一般按“**消息唯一标识 + 消费者维度**”来设计，因为同一条消息可能被不同消费者组消费。
    核心字段可以有：主键 id、message_id、consumer_group、biz_key、status、retry_count、create_time、update_time。
    其中 message_id 和 consumer_group 要建唯一索引，或者用 biz_key 和 consumer_group 建唯一索引，保证同一个消费者对同一条消息只能插入一次。
    消费流程一般是：收到消息后先插入去重表，插入成功说明第一次消费，然后执行业务逻辑；如果唯一索引冲突，说明已经处理过，就直接跳过并 ack。
    更稳妥的是把“插入去重表”和“业务数据更新”放在同一个本地事务里，事务成功后把状态改成 SUCCESS。
    status 可以设计成 PROCESSING、SUCCESS、FAILED，用来处理消费中宕机、失败重试等场景。
    如果数据量很大，去重表可以按时间分表，或者定期清理历史成功记录，避免表无限增长。

### 如何保证消息不丢失？

生产端确认发送成功，Broker 持久化和多副本保障，消费端处理成功后再提交确认。关键业务还可以加本地消息表和补偿任务。

#### 常见追问

- 生产者发送失败怎么处理？
    生产者发送失败不能简单认为消息一定没发出去，因为可能是 Broker 已经收到消息，但返回 ack 时网络异常了，所以要按“不确定结果”处理。
    一般会先开启生产端重试，设置合理的重试次数、超时时间和退避策略，避免瞬间把 Broker 打爆。
    更稳妥的做法是用本地消息表或者 outbox 机制，业务数据和待发送消息在同一个本地事务里落库，事务提交后再异步发送 MQ。
    如果发送成功，就把消息状态改成 SENT；如果发送失败，就由定时任务扫描未发送或发送失败的消息继续补偿发送。
    同时消息要带业务唯一键或 messageId，配合消费端幂等，解决生产者重试导致的重复消息问题。
    如果多次重试仍然失败，需要记录失败原因、告警，并支持人工补偿，不能让失败消息静默丢失。
- Broker 宕机如何保证消息不丢？
    Broker 宕机下要降低消息丢失风险，通用原则是：**生产端确认、Broker 持久化与副本、消费成功后确认、消费端幂等**。
    生产端需要确认发送结果；失败或超时按不确定结果处理并重试，关键业务最好配合本地消息表或 outbox 补偿。
    Broker 端应选择支持持久化和复制的队列或 Topic，并按产品的副本确认语义配置。Kafka 通常要结合 `acks=all`、`min.insync.replicas` 和副本因子；RabbitMQ 则要结合 publisher confirm、持久化队列或 quorum queue。
    消费端要在业务处理成功后再提交确认或 offset；否则消费者异常时会丢消息。处理成功但确认失败又可能造成重复，因此仍要做幂等。
    单 Broker 即使开启持久化，也无法完全避免机器、磁盘等故障带来的风险。完整方案需要按具体 MQ 的确认、刷盘和副本机制配置，而不是假设所有 MQ 都在相同阶段 ACK。
- 消费者先提交 offset 再处理会怎样？
    消费者先提交 offset 再处理消息，会有消息丢失风险。
    因为 offset 已经提交后，Broker 会认为这条消息已经消费完成，如果消费者还没处理业务就宕机，重启后会从新的 offset 继续消费，这条消息就不会再被拉取了。
    这种模式属于最多消费一次，适合能容忍丢消息的场景，但一般业务不推荐。
    更常见的做法是关闭自动提交，等业务处理成功后再手动提交 offset。
    不过这样也可能出现业务处理成功但提交 offset 失败的情况，消息会被重复消费，所以消费端还要做幂等控制。

### 顺序消息怎么实现？

把同一业务维度的消息发送到同一个队列或分区，并由单线程顺序消费。顺序性通常只能在局部业务维度上保证，而不是全局顺序。

#### 常见追问

- 全局顺序为什么代价高？
    全局顺序代价高，主要是因为**它会把并发能力压到很低**。
    正常情况下，消息可以按 key 分散到多个队列或分区里，不同队列可以并行发送、存储和消费。
    但如果要求全局顺序，通常所有消息只能进入同一个队列或同一个分区，并且只能由一个消费者线程按顺序处理。
    这样 Broker 的多分区能力、消费者集群的并发能力都用不上，吞吐量会明显下降。
    另外一条消息处理慢，后面的消息都要等待，容易出现阻塞放大。
    如果某条消息消费失败，为了保证顺序，后续消息也不能继续消费，只能等这条消息处理完或人工干预。
    所以实际业务一般不会追求全局顺序，而是按业务维度做局部顺序，比如同一个订单、同一个用户的消息保持顺序。
- Kafka 如何保证分区内有序？
    Kafka 保证分区内有序，主要是因为**同一个分区内部是追加写日志的**。
    生产者发送到同一个 partition 的消息，会按写入顺序追加到日志文件里，每条消息都有递增的 offset。
    消费者消费时，也是按 offset 从小到大拉取，所以在单个分区内天然是有序的。
    但这个有序只针对同一个 partition，不保证多个 partition 之间的全局顺序。
    如果业务要求某一类消息有序，一般会用相同的 key，让这些消息路由到同一个 partition。
    消费端也要注意，同一个 partition 不能被多个消费者实例同时消费；在同一个消费者内部，也不要把同一分区的消息拆到多个线程乱序处理。
    Broker 保证的是同一分区中已写入记录的顺序。对发送顺序要求严格时，生产端应启用幂等并避免同一业务 key 被并发乱序发送；增加 Topic 分区数会改变 key 到分区的映射，也可能破坏扩容前后的连续顺序语义。
- 消费失败会不会阻塞后续消息？
    会不会阻塞后续消息，**取决于消费端怎么处理失败消息**。
    Kafka 本身只是按 offset 拉取消息，不会因为某条消息业务处理失败就自动阻塞后面的消息。
    如果消费失败后不提交 offset，并且要求分区内顺序，那通常要停在这条消息上重试，这样同一个分区后续消息就会被阻塞。
    如果不要求严格顺序，可以把失败消息记录下来，或者发送到重试队列、死信队列，然后提交 offset，后续消息就可以继续消费。
    但如果直接跳过失败消息并提交 offset，就要接受这条消息丢失或后续补偿的成本。
    所以一般核心业务会结合手动提交、重试机制、死信队列和幂等处理来设计。

### 最终一致性怎么做？

通过本地事务 + 消息发送 + 消费端幂等 + 补偿任务实现最终一致性。复杂跨服务事务可以评估 Seata，但不要一上来就使用分布式事务框架。

#### 常见追问

- 本地消息表怎么设计？
    本地消息表一般是**为了保证业务操作和发消息在同一个本地事务里完成，避免业务成功但消息没发出去**。
    表里通常会有这些字段：消息 id、业务 id、业务类型、消息内容、状态、重试次数、下一次重试时间、创建时间、更新时间。
    业务执行时，先在同一个数据库事务里写业务数据和本地消息表，事务提交后由后台任务扫描待发送的消息，再投递到 MQ。
    发送成功后把消息状态改成已发送；发送失败就增加重试次数，并更新下一次重试时间。
    如果超过最大重试次数，可以改成失败状态，后续走告警或人工补偿。
    消费端也要做幂等，因为消息可能重复发送。一般可以用消息 id 或业务唯一键做去重，保证重复消费不会产生重复影响。
- 事务消息解决什么问题？
    事务消息主要解决“**本地事务和消息发送一致性**”的问题。
    也就是业务数据已经提交了，但消息发送失败；或者消息发出去了，但本地事务回滚了，导致下游拿到错误消息。
    事务消息的目标是协调本地事务结果与消息是否对消费者可见：本地事务成功时消息最终可投递，本地事务失败时消息不应对消费者可见。
    比如在下单、支付、扣库存这类场景里，不能只保证数据库操作成功，还要保证后续通知、库存变更这些异步流程能可靠触发。
    它一般保证的是最终一致性，不是强一致性；消费者是否最终处理成功仍依赖消费重试、死信、业务幂等和补偿机制。
    实现上可以用 MQ 的事务消息机制，也可以用本地消息表加定时补偿来实现。
- 补偿任务如何避免重复执行？
    补偿任务避免重复执行，一般不能只依赖定时任务本身，要从**任务调度**和**业务幂等**两层控制。
    首先补偿表里要有明确状态，比如待处理、处理中、成功、失败，执行前用条件更新抢占任务，比如只允许从待处理更新成处理中，更新成功的节点才能执行。
    如果是多实例部署，还需要分布式锁，或者用数据库乐观锁、唯一索引来保证同一条任务不会被多个节点同时处理。
    执行成功后再把状态改成成功，失败就记录重试次数和下次重试时间，避免一直高频重试。
    另外业务操作本身也要做幂等，比如用业务单号、消息 id 或补偿任务 id 做唯一约束，保证即使任务重复触发，也不会重复扣款、重复发货。
    所以核心是：**任务抢占防并发，状态流转防重复，业务幂等兜底**。

## RabbitMQ

### RabbitMQ 中有哪些重要角色？

RabbitMQ 重要角色包括生产者、消费者和 Broker。
生产者发送消息，消费者消费消息，Broker 负责接收、路由、存储和投递消息。Broker 内部还包含交换机、队列、绑定等概念。

#### 常见追问

- Broker 和 Queue 有什么区别？
    Broker 是消息中间件的服务节点，负责接收、存储、投递消息，也负责管理连接、路由、持久化等能力。
    Queue 是消息队列里的逻辑存储单元，用来保存某一类消息，消费者一般是从 Queue 里拉取或接收消息。
    简单说，Broker 是“服务器”或“消息代理节点”，Queue 是 Broker 里面承载消息的“队列”。
    一个 Broker 上可以有多个 Queue，一个 Queue 也可能根据中间件实现分布在多个 Broker 上。
    比如 RocketMQ 里，Broker 是物理服务进程，Topic 会拆成多个 MessageQueue 分布在 Broker 上；RabbitMQ 里，Broker 是 RabbitMQ 服务节点，Queue 是具体存放消息的队列。
- 生产者是否直接把消息发到队列？
    一般不是简单地直接发到某个队列，而是先发到 Broker，由 Broker 根据路由规则决定消息进入哪个队列。
    比如 RocketMQ 里，生产者发送到 Topic，客户端会根据 Topic 的路由信息选择一个 MessageQueue，再把消息发到对应的 Broker。
    RabbitMQ 里，生产者通常把消息发到 Exchange，Exchange 根据 routing key 和绑定关系再路由到 Queue。
    所以从结果看消息最终进入队列，但生产者通常面向的是 Topic 或 Exchange，而不是直接操作队列。
- 消费者如何确认消息？
    消费者确认消息一般是**通过 ACK 机制**。消费者拿到消息并且业务处理成功后，再向 Broker 返回确认，Broker 收到 ACK 后才认为这条消息消费成功，然后删除消息或者标记消费进度。
    如果业务处理失败，消费者可以返回失败、拒绝确认，或者不 ACK，这样 Broker 会根据机制重新投递。
    在 RocketMQ 里，消费者处理成功返回 `CONSUME_SUCCESS`，失败返回 `RECONSUME_LATER`，Broker 后续会重试投递。
    在 RabbitMQ 里，如果开启手动 ACK，消费者处理成功后调用 `basicAck`，失败可以 `basicNack` 或 `basicReject`，决定是否重新入队。
    所以关键点是：**业务成功后再 ACK，避免消息还没处理完就被确认导致丢失**。

### RabbitMQ 有哪些重要组件？

核心组件包括 ConnectionFactory、Connection、Channel、Exchange、Queue、RoutingKey 和 BindingKey。客户端通过连接和 Channel 与 Broker 通信，生产者把消息发到 Exchange，Exchange 根据绑定关系和路由键把消息路由到 Queue。

#### 常见追问

- 为什么需要 Channel？
    RabbitMQ 需要 Channel 主要是为了**复用 TCP 连接**，降低连接创建和维护的成本。
    如果每个生产者、消费者都单独建立 TCP 连接，开销会比较大，而且连接数太多也会影响 Broker 性能。
    Channel 是建立在 Connection 上的逻辑通信通道，一个 Connection 里可以创建多个 Channel，不同线程或业务可以用不同的 Channel 进行消息发送、消费、确认等操作。
    另外 RabbitMQ 的大部分操作，比如声明交换机、声明队列、发布消息、消费消息、ACK，都是通过 Channel 完成的。
    需要注意的是，**Channel 不是线程安全的**，所以通常一个线程使用一个 Channel，多个线程不要共享同一个 Channel。
- Exchange 和 Queue 是什么关系？
    Exchange 和 Queue 是 RabbitMQ 里的路由关系。生产者一般把消息发送到 Exchange，Exchange 根据类型、routing key 和绑定关系，把消息路由到一个或多个 Queue。消费者不是从 Exchange 消费，而是从 Queue 消费。
    Exchange 本身主要负责路由，不负责长期存储消息；Queue 才是真正存放消息的地方。
    Queue 通过 Binding 绑定到 Exchange 上，Binding 可以带 routing key。不同类型的 Exchange，比如 direct、fanout、topic、headers，路由规则不一样。
    所以它们的关系可以理解为：Exchange 负责分发消息，Queue 负责保存消息，Binding 负责建立两者之间的路由规则。
- RoutingKey 和 BindingKey 有什么区别？
    RoutingKey 是生产者发送消息时带上的路由键，表示这条消息希望按什么规则路由。
    BindingKey 是队列绑定到交换机时设置的绑定键，表示这个队列关心什么样的消息。
    Exchange 会根据自身类型，用消息的 RoutingKey 去匹配队列的 BindingKey，匹配成功才会把消息路由到对应 Queue。
    比如 direct exchange 要求 RoutingKey 和 BindingKey 完全匹配；topic exchange 支持通配符匹配；fanout exchange 会忽略这些 key，直接广播到绑定的队列。
    所以简单说：**RoutingKey 是消息上的 key，BindingKey 是绑定关系上的 key**。

### RabbitMQ vhost 有什么作用？

vhost 是 RabbitMQ 的虚拟主机，用来隔离交换机、队列、绑定和权限。不同业务或环境可以使用不同 vhost，避免资源命名冲突，也方便做权限控制。

#### 常见追问

- vhost 和数据库 schema 类似吗？
    类似，但不能完全等同。
    RabbitMQ 的 vhost 可以理解为逻辑隔离空间，里面有自己独立的 Exchange、Queue、Binding、权限等资源，不同 vhost 之间的资源互相不可见。
    这一点和数据库里的 schema 有点像，都是在同一个服务实例下做资源隔离和命名空间管理。
    但 vhost 不只是表结构命名空间，它还和 RabbitMQ 的权限控制、连接访问范围、消息资源隔离绑定得更紧。
    所以面试里可以说：**vhost 类似数据库 schema，用来做逻辑隔离，但在 RabbitMQ 里隔离粒度和权限控制更完整**。
- 用户权限如何绑定 vhost？
    用户权限是**按 vhost 维度授权的**，也就是用户创建后，还要给它指定能访问哪个 vhost，以及在这个 vhost 下对资源的权限。
    常用命令是：`rabbitmqctl set_permissions -p /my_vhost user ".*" ".*" ".*"`
    这里三个 `.*` 分别表示 configure、write、read 权限，也就是配置资源、写消息、读消息。
    如果只想限制权限，可以把正则写得更细，比如只允许访问某些 exchange 或 queue。
    也可以在 RabbitMQ Management 控制台里，进入用户详情页，给指定 vhost 配置权限。
    需要注意的是，用户的 tag 比如 administrator 主要影响管理后台权限，不等于自动拥有所有 vhost 的消息访问权限，vhost 权限还是要单独配置。
- 多租户场景如何隔离？
    多租户场景一般**优先用 vhost 做隔离**，每个租户分配一个独立 vhost。
    每个 vhost 下有自己独立的 Exchange、Queue、Binding，资源名称可以相同但互不影响。
    然后给每个租户创建独立用户，只授权访问自己的 vhost，不给其他 vhost 权限。
    权限上可以进一步通过 configure、write、read 三类权限控制用户能创建资源、发送消息、消费消息的范围。
    如果多个租户必须共用一个 vhost，就需要通过 Exchange、Queue 命名规范和权限正则来隔离，但这种隔离弱一些，维护成本也更高。
    所以面试里可以回答：**RabbitMQ 多租户隔离主要靠 vhost + 用户权限，必要时再结合资源命名规范、权限正则、配额和监控来控制资源使用**。

### RabbitMQ 消息是怎么发送的？

客户端先建立 TCP 连接并完成认证，然后创建 AMQP Channel，通过 Channel 把消息发布到 Exchange。Exchange 根据类型、RoutingKey 和 BindingKey 把消息路由到匹配队列，消费者从队列中拉取或接收推送消息。

#### 常见追问

- 为什么不每次发送都新建连接？
    不建议每次发送消息都新建连接，**因为 RabbitMQ 的 Connection 是比较重的资源**。
    每次新建连接都要经历 TCP 握手、认证、协议协商，开启心跳等过程，耗时比较高，会明显增加发送延迟。
    频繁创建和关闭连接还会占用 broker 的文件句柄、内存、网络资源，也可能导致客户端出现大量 `TIME_WAIT`，高并发下容易把连接数打满。
    正确做法一般是复用长连接，在连接上创建 Channel 发送消息。Channel 比 Connection 轻量很多。
    在 Java 客户端里通常会维护一个长期可用的 Connection，再按线程或通过 Channel 池复用 Channel。需要注意的是 Channel 一般不要被多个线程并发共享。
- Channel 是线程安全的吗？
    RabbitMQ Java Client 里的 Channel **不是线程安全的**，不建议多个线程并发共用同一个 Channel。
    如果多个线程同时在一个 Channel 上发布消息，可能会出现帧交错、确认回调混乱、协议异常等问题。
    比较常见的做法是一个线程使用一个 Channel，或者使用 Channel 池来复用，但同一时刻一个 Channel 只给一个线程使用。
    Connection 是相对重量级的，可以复用；Channel 是轻量级的，但也要按线程隔离使用。
    如果是 Spring AMQP，一般通过 `CachingConnectionFactory` 管理连接和 Channel 缓存，不需要手动频繁创建和关闭 Channel。
- 消息没有路由到队列会怎样？
    消息没有路由到任何队列时，**默认情况下 RabbitMQ 会直接丢弃这条消息，生产者不会感知到异常**。
    如果发送消息时设置了 `mandatory=true`，并且消息无法路由，RabbitMQ 会通过 `basic.return` 把消息退回给生产者，生产者可以通过 ReturnListener 或 Spring 的 ReturnsCallback 感知。
    如果 Exchange 配置了 Alternate Exchange，无法路由的消息会被转发到备份交换机，再由备份交换机路由到对应队列。
    因此不要把 Alternate Exchange 与 Return Callback 简化为固定先后关系：应按实际 Exchange、绑定和客户端回调配置验证不可路由消息的处理路径。
    还要区分不存在的 Exchange：这不是“未路由到队列”，发布会触发协议错误并关闭当前 Channel，不能依赖 Return Callback 兜底。
    所以面试里可以回答：**默认不可路由消息会丢弃；需要感知可使用 `mandatory` 和 Return Callback；需要兜底存储可配置 Alternate Exchange；Exchange 必须预先声明并确认存在**。

### RabbitMQ 如何保证发送稳定性？

可以使用事务或 publisher confirm 机制。事务模式简单但性能较差；confirm 模式由 Broker 异步确认发布处理结果，更适合高吞吐场景。关键消息还要结合队列类型、持久化、重试和补偿机制。

#### 常见追问

- confirm 和 return callback 有什么区别？
    Confirm Callback 和 Return Callback 关注的阶段不一样。
    Confirm Callback 是**生产者确认机制**，表示 Broker 已完成该发布的确认处理，不只是确认 Exchange 接收。持久化消息的确认时机还取决于队列类型、落盘和副本机制，例如 quorum queue 通常要等多数副本写入并同步后确认。
    Return Callback 关注的是消息到达 Exchange 之后，能不能根据路由键路由到队列。只有设置 `mandatory=true`，并且消息没有路由到任何队列时，才会触发 Return Callback。
    所以简单说：**Confirm 判断 Broker 对发布的确认结果，Return 判断消息是否因不可路由被退回**。两者需要结合处理，不能只依赖 Confirm 判断消息是否最终进入队列。
- 事务模式为什么性能差？
    RabbitMQ 的事务模式性能差，主要是因为它是**同步阻塞**的。
    开启事务后，生产者每发送一批消息都要执行 `tx.commit`，提交时必须等待 Broker 返回结果，期间线程不能继续高效发送后续消息。
    而且事务会增加 Broker 端的持久化、状态维护和网络往返开销，吞吐量会明显下降。
    相比之下，Confirm 机制是异步确认，生产者可以连续发送消息，再批量接收 ack，所以性能通常比事务模式好很多。
    面试里可以总结为：**事务模式只提供当前 Channel 内发布和确认操作的 Broker 侧同步事务语义，每次提交都要等待 Broker 确认，网络 RTT 和磁盘刷盘成本都会放大，所以不适合高吞吐场景，生产环境更常用 Confirm 机制**。它不能协调本地数据库事务，也不提供跨服务业务强一致。
- 生产者重试如何避免重复消息？
    生产者重试本身**不能绝对避免重复消息**，因为可能出现消息已经发送到 Broker，但确认响应丢失，生产者误以为失败又重发。
    一般做法是给每条消息生成全局唯一的 `messageId`，重试时保持同一个 `messageId` 不变，不要每次重试都生成新 ID。
    生产者侧可以配合消息表，先把消息落库，状态标记为待发送、已发送，收到 Confirm ack 后再更新状态，定时任务只重试未确认的消息。
    消费者侧必须做幂等处理，比如用 `messageId` 建唯一索引或 Redis 去重，已经处理过的消息直接跳过。
    所以面试里可以回答：**生产者重试只能降低消息丢失风险，不能保证不重复，真正避免重复消费要靠唯一消息 ID、消息状态表和消费者幂等共同保证**。

### RabbitMQ 如何避免消息丢失？

需要同时保证队列持久化、消息持久化、交换机持久化，以及消费者手动 ack。消息只有投递到持久化交换机并路由到持久化队列，且消息本身设置为持久化，Broker 重启后才更可能恢复。

#### 常见追问

- durable 队列是否能保证消息不丢？
    不能。`durable` 只表示队列元数据持久化，也就是 Broker 重启后队列还在，不代表队列里的消息一定不丢。
    要尽量保证消息不丢，通常还需要：消息本身设置为持久化，比如 `deliveryMode=2`；发送端开启 Confirm 并结合队列类型处理确认结果；Exchange 也要持久化；消费者处理完业务后再手动 ack。
    另外，即使队列和消息都持久化，如果消息刚到 Broker 还没刷盘就宕机，也可能丢失，所以严格来说 RabbitMQ 不能只靠 `durable` 队列保证消息不丢。
    面试里可以总结为：**`durable` 只能保证队列不丢，不能单独保证消息不丢，消息可靠性要结合队列持久化、消息持久化、Confirm 和手动 ack**。
- 消费者自动 ack 有什么风险？
    消费者自动 ack 的风险是**消息可能丢失**。
    自动 ack 是 Broker 把消息推给消费者后就认为消息已经消费成功，会立刻从队列中删除。
    如果消费者收到消息后还没来得及处理业务就宕机、线程异常或者服务重启，这条消息不会重新投递，就丢失了。
    所以在对可靠性有要求的场景，一般不用自动 ack，而是使用手动 ack，等业务逻辑真正处理成功后再确认；如果处理失败，可以 nack 或 reject，让消息重新入队或进入死信队列。
    面试里可以总结为：**自动 ack 性能稍好，但可靠性差，适合允许丢消息的场景；核心业务一般使用手动 ack**。
- RabbitMQ quorum queue 解决什么问题？
    quorum queue **主要解决队列高可用问题**，避免单个节点宕机导致队列不可用或已确认消息丢失。
    普通 classic queue 默认只存在于一个节点上；节点故障时，该队列可能不可用。quorum queue 基于 Raft 做多数派复制，适合需要复制和故障恢复的生产队列。
    RabbitMQ 4.0 已移除 classic queue mirroring，传统镜像队列只属于旧版本知识；当前需要高可用复制队列时应使用 quorum queue，并按副本数和故障域规划。
    它不能解决所有可靠性问题，例如生产者重试导致的重复消息、消费者业务失败和幂等问题，仍要配合 Confirm、手动 ack 和消费者幂等。
    面试里可以总结为：**quorum queue 解决 RabbitMQ 队列的高可用和节点故障下的消息可靠性问题，但端到端可靠性还需要生产、消费两端共同保证**。

### RabbitMQ 持久化有什么缺点？

持久化会增加磁盘 IO，降低吞吐量和写入性能。生产中需要在可靠性和性能之间权衡，可以使用 SSD、批量确认、合理队列设计和高可用队列降低影响。

#### 常见追问

- 所有消息都需要持久化吗？
    不是所有消息都需要持久化。
    消息持久化会提升可靠性，但也会增加磁盘 IO，性能会比非持久化消息差一些。
    如果是核心业务消息，比如订单、支付、库存扣减，一般需要持久化，并且配合 durable 队列、持久化 Exchange、Confirm 和手动 ack。
    如果是一些允许丢失的消息，比如日志、监控指标、临时通知、实时状态同步，就可以不持久化，优先考虑吞吐量和性能。
    面试里可以总结为：**是否持久化要看业务对可靠性的要求，核心消息持久化，允许丢失的高频消息可以不持久化**。
- 持久化和高可用是一回事吗？
    不是一回事。
    持久化解决的是消息落盘问题，主要防止 RabbitMQ 重启后消息丢失，比如 durable 队列、持久化消息、持久化交换机。
    高可用解决的是节点故障后的服务可用性问题，例如 quorum queue 让队列数据在多个节点之间复制，某个节点宕机后仍可继续提供服务。镜像队列只属于 RabbitMQ 4.0 之前的旧版方案。
    持久化不等于高可用，因为消息只落在单个节点磁盘上，这个节点宕机时队列仍然不可用。
    高可用也不完全等于不丢消息，还要配合生产者 Confirm、手动 ack、重试和幂等。
    面试里可以总结为：**持久化解决“重启后数据还在”，高可用解决“节点挂了服务还能用”**，两者通常要配合使用。
- 磁盘满了会发生什么？
    磁盘满了时，RabbitMQ 会**触发磁盘告警**，也就是 disk free limit。
    触发后 RabbitMQ 会**阻塞生产者连接，停止接收新的消息发布**，防止继续写入导致节点不可用或数据损坏。
    已经在队列里的消息一般还能被消费者继续消费，消费后释放磁盘空间；空间恢复到阈值以上后，生产者才会恢复发送。
    如果磁盘真的被写满，可能会导致**消息落盘失败、节点异常，甚至 RabbitMQ 服务不可用**。
    面试里可以总结为：**RabbitMQ 磁盘不足会触发流控，阻塞生产者写入，优先保护节点和已有消息，等磁盘空间恢复后再继续接收消息**。

### RabbitMQ 交换器类型有哪些？

常见类型包括 direct、fanout、topic 和 headers。
direct 按精确 routing key 路由；fanout 广播到所有绑定队列；topic 支持通配符模式匹配；headers 根据消息头匹配，使用较少。

#### 常见追问

- 日志广播适合哪种交换器？
    日志广播适合使用 fanout 交换器。
    fanout 交换器会把消息广播到所有绑定到它的队列，不关心 routing key，所以很适合日志这种“一条日志给多个消费者同时处理”的场景。
    比如一个消费者写文件，一个消费者做告警，一个消费者做实时分析，都可以各自绑定一个队列到同一个 fanout 交换器。
    如果日志需要按级别过滤，比如 error、warn、info，就更适合用 direct 或 topic 交换器。
- topic 中 `*` 和 `#` 有什么区别？
    在 RabbitMQ 的 topic 交换器里，routing key 和 binding key 都是用 `.` 分隔的多个单词。
    `*` 表示匹配一个单词，必须正好占一段。
    `#` 表示匹配零个或多个单词，可以占多段，也可以不占。
    比如 binding key 是 `log.*`，可以匹配 `log.info`，但不能匹配 `log.system.error`。
    如果是 `log.#`，可以匹配 `log`、`log.info`、`log.system.error`。
    面试里可以简单说：**`*` 匹配一段，`#` 匹配零段或多段，`#` 的范围更大**。
- direct 和 topic 如何选择？
    direct 和 topic 主要看路由规则复杂度。
    如果路由规则是精确匹配，比如按固定的业务类型、固定的日志级别分发消息，就选择 direct。它要求 routing key 和 binding key 完全一致，规则简单，性能也更直接。
    如果路由规则有层级或者需要模糊匹配，就选择 topic。topic 支持 `*` 和 `#` 通配符，比如按系统、模块、级别这种多维度路由会更灵活。
    面试里可以总结为：**能用 direct 精确匹配解决的就用 direct，规则有层级、需要通配匹配时再用 topic**。

### RabbitMQ 延迟队列怎么实现？

常见方式是 TTL + 死信交换机：消息过期后进入死信交换机，再路由到真正消费队列。也可以使用 `rabbitmq-delayed-message-exchange` 插件实现延迟消息。插件方式更直观，但需要额外安装和维护。

#### 常见追问

- TTL 设置在队列和消息上有什么区别？
    TTL 设置在队列上，是通过 `x-message-ttl` 给整个队列设置统一过期时间，进入这个队列的消息都按同一个时间过期，适合固定延迟时间的场景，比如 5 分钟延迟队列、30 分钟延迟队列。
    TTL 设置在消息上，是给每条消息单独设置 `expiration`，不同消息可以有不同的过期时间，灵活性更高。
    如果队列 TTL 和消息 TTL 同时设置，实际以较小的那个为准。
    需要注意的是，用消息 TTL 做延迟队列时，可能有队头阻塞问题：前面的消息延迟时间很长，后面的消息即使先过期，也可能要等前面的消息到期后才会进入死信交换机。
    所以实际项目里，如果延迟时间是固定几个档位，通常更倾向用队列 TTL + 死信交换机；如果延迟时间很灵活，可以考虑插件方式。
- 死信队列还能处理哪些场景？
    死信队列除了做延迟队列，还常用于异常消息兜底处理。
    常见触发死信的情况有三类：消息被消费者 `reject` 或 `nack`，并且 `requeue=false`；消息 TTL 过期；队列达到最大长度后，旧消息被挤出。
    实际场景里，死信队列可以用来保存消费失败的消息，方便后续人工排查、告警、补偿或者重新投递。
    比如订单、支付、库存这类关键业务，如果消费失败，不应该直接丢弃，可以进入死信队列做统一处理。
    也可以配合重试队列使用，失败后先延迟重试，超过最大重试次数后再进入死信队列，避免一直重复消费影响系统。
    面试里可以总结为：**死信队列主要用于处理无法正常消费的异常消息，起到隔离、兜底、补偿和监控告警的作用**。
- 延迟消息适合订单超时关闭吗？
    适合，但要看业务要求。
    订单超时关闭可以用 RabbitMQ 延迟消息实现，比如下单后发送一条 30 分钟延迟消息，消息到期后检查订单状态，如果还未支付就关闭订单。
    但消费时一定不能直接关闭，要先查库确认订单仍是待支付状态，因为用户可能已经支付成功，只是延迟消息还没消费到。
    如果对时间精度要求不是特别高，延迟消息比较合适；如果要求非常精确，或者订单量特别大，也可以考虑定时任务扫描、时间轮、Redis 过期事件等方案。
    实际项目里一般会做兜底，比如延迟消息负责及时关闭，定时任务负责补偿扫描，避免消息丢失或消费失败导致订单一直不关闭。

### RabbitMQ 集群有什么作用？

RabbitMQ 集群用于提升可用性和容量。普通集群会同步元数据，但队列消息默认不一定完整复制到所有节点；当前需要高可用复制的消息队列应使用 quorum queue。设计集群时要区分元数据高可用和消息高可用。

#### 常见追问

- RabbitMQ 集群节点会完整复制所有消息吗？
    不会，RabbitMQ 普通集群不会把所有消息完整复制到每个节点。
    普通集群主要同步的是**交换机、队列、绑定关系、用户权限这些元数据**，队列里的消息默认只存放在队列所在的节点上。
    客户端连到任意节点都可以发送和消费，是因为集群内部会帮忙路由到实际队列所在节点，不代表消息在每个节点都有一份。
    如果队列所在节点宕机，这个队列就不可用；如果消息是持久化的，节点恢复后还能继续使用，否则可能丢失。
    RabbitMQ 4.0 之前可使用镜像队列或 quorum queue；当前镜像队列已移除，需要消息高可用时应使用 quorum queue。消息会复制到多个副本节点，但不会默认复制到所有节点，而是按副本数配置复制。
- 镜像队列有什么问题？
    镜像队列是 RabbitMQ 旧版本的高可用方案，性能和稳定性成本较高：它会复制完整队列副本，消息越多，同步、网络和磁盘开销越大；一个队列仍由主节点负责读写，扩容也不一定提升单队列性能。
    主节点切换会带来短暂不可用，扩缩容、重启时还可能触发大量同步，对集群压力较大。
    RabbitMQ 4.0 已移除 classic queue mirroring，因此新集群不应再设计镜像队列；需要高可用复制时应使用基于 Raft 的 quorum queue。

### RabbitMQ 节点类型有哪些？

旧版 RabbitMQ 曾区分磁盘节点和内存节点：磁盘节点持久化元数据，内存节点主要将元数据保存在内存中。
RAM node 已在 RabbitMQ 4.0 移除。当前生产集群应使用磁盘节点，并通过 quorum queue 的副本和故障域规划队列数据高可用。

#### 常见追问

- 唯一磁盘节点宕机会怎样？
    这是旧版混合磁盘节点和内存节点集群的历史问题。唯一磁盘节点宕机后，剩余内存节点可能短暂保留已有元数据，但元数据变更和集群恢复会受影响；内存节点重启后尤其依赖磁盘节点恢复元数据。
    RabbitMQ 4.0 已移除 RAM node，当前生产集群不应采用这种拓扑。应使用磁盘节点，并通过 quorum queue 副本、节点故障域和容量冗余避免关键队列形成单点。
- 为什么生产不建议只有一个磁盘节点？
    对旧版混合节点集群而言，唯一磁盘节点会成为元数据持久化和恢复的单点，因此不建议只部署一个。
    对当前 RabbitMQ 4.x 集群而言，RAM node 已移除，生产节点应使用磁盘节点。关键是按队列类型规划高可用：普通 classic queue 不具备复制能力；需要容错的队列应采用 quorum queue，并将副本分散到不同节点或故障域。
- 节点类型和消息持久化是什么关系？
    在旧版混合节点集群中，节点类型与消息持久化没有直接等价关系：节点类型主要影响元数据存储，消息是否可在重启后恢复取决于队列和消息的持久化属性。
    当前 RabbitMQ 4.x 已无 RAM node。对 durable classic queue，消息还应标记为 persistent；quorum queue 的消息始终持久化。无论哪种队列，端到端可靠性仍需生产者 Confirm、消费者手动 ack 和业务幂等。

### RabbitMQ 集群搭建要注意什么？

要确保节点之间网络互通，Erlang cookie 一致，节点名和主机名解析正确。当前生产集群使用磁盘节点；关键队列采用 quorum queue，并将副本分散到不同节点或故障域。旧版“先停内存节点、最后停磁盘节点”的操作仅适用于已移除 RAM node 的历史拓扑，不适用于 RabbitMQ 4.x。

#### 常见追问

- Erlang cookie 不一致会怎样？
    Erlang cookie 不一致时，RabbitMQ 节点之间无法通过 Erlang 分布式通信完成认证，所以节点不能加入集群，已经在集群里的节点如果重启或重新连接，也可能无法重新加入。
    表现上通常是 `rabbitmqctl` 访问节点失败、节点 join_cluster 失败，日志里可能看到 authentication failed、Invalid challenge reply 这类错误。
    因为 RabbitMQ 集群通信依赖 Erlang cookie 做节点间身份校验，所有集群节点的 cookie 必须一致，而且执行 `rabbitmqctl` 的用户环境下 cookie 也要和目标节点一致，否则命令行工具也连不上节点。
- 为什么节点名解析很重要？
    RabbitMQ 集群底层依赖 Erlang 分布式通信，节点之间不是简单用 IP 识别，而是用节点名通信，比如 `rabbit@hostname`。
    所以每个节点的主机名必须能被其他节点正确解析到对应 IP，否则节点之间就无法建立连接，可能导致加入集群失败、节点重启后回不来、集群分区或者 `rabbitmqctl` 命令连接失败。
    如果用短节点名，所有节点就要能解析短主机名；如果用长节点名，就要统一使用 FQDN，比如 `rabbit@node1.example.com`，并配置 `RABBITMQ_USE_LONGNAME=true`。
    生产里一般会通过 DNS 或 `/etc/hosts` 保证解析稳定，避免主机名变化或解析不一致导致集群通信异常。
- 集群扩容如何迁移队列？
    RabbitMQ 扩容后，已有队列一般不会自动迁移到新节点，新节点默认更多是承接后续新建队列。
    如果是普通 classic queue，迁移通常要停流量或灰度切流，先把消息消费完，再在目标节点重新声明队列，或者通过删除重建让队列重新分布，这种方式对业务影响比较大。
    镜像队列扩容方式仅适用于 RabbitMQ 4.0 之前的旧集群：可调整 HA policy，把新节点加入镜像范围，等消息同步完成后再做主节点重平衡或逐步下线旧节点。
    如果是 quorum queue，扩容后可以用 `rabbitmq-queues grow` 或 `add_member` 把新节点加入副本，再用 `rabbitmq-queues rebalance` 做 leader 均衡。
    实际生产里一般不会直接“搬队列”，而是先加节点、调策略、同步副本、确认状态正常，再逐步切流或做 rebalance，避免消息丢失和业务中断。

## Kafka

### Kafka 是什么？

Kafka 是分布式事件流平台，具备高吞吐、低延迟、持久化和可扩展能力，常用于日志采集、行为埋点、异步解耦、削峰和数据管道。Kafka 的核心抽象包括 Topic、Partition、Producer、Consumer、Consumer Group 和 Broker。

#### 常见追问

- Kafka 为什么吞吐高？
    Kafka 吞吐高主要是因为它的设计**大量利用了顺序读写和批量处理**。
    消息写入是追加到日志文件末尾，磁盘顺序写性能很高，避免了随机 IO。
    Producer 和 Broker 都支持批量发送、批量写入，减少网络请求次数和磁盘刷写次数。
    Kafka 还利用了 Page Cache，很多读写都走操作系统缓存，降低了直接磁盘 IO 开销。
    消费时可以使用零拷贝，比如通过 `sendfile` 把数据从 Page Cache 直接发送到网卡，减少用户态和内核态的数据拷贝。
    另外 Kafka 通过分区机制把数据分散到多个 Broker 上，并行写入和消费，所以整体吞吐能力可以横向扩展。
- Topic 和 Partition 有什么关系？
    Topic 是 Kafka 里消息的逻辑分类，业务上一般按主题来区分消息类型。
    Partition 是 Topic 的物理分片，一个 Topic 可以有多个 Partition。
    消息最终不是直接存到 Topic 上，而是写入某个 Partition，每个 Partition 内部是有序追加的日志。
    Kafka 通过 Partition 实现并行读写和横向扩展，不同 Partition 可以分布在不同 Broker 上。
    顺序性只在单个 Partition 内保证，不能保证整个 Topic 全局有序。
    Consumer Group 消费时，一个 Partition 同一时刻只能被组内一个 Consumer 消费，但一个 Consumer 可以消费多个 Partition。
- Consumer Group 解决什么问题？
    Consumer Group **主要解决 Kafka 消费端的负载均衡和容错问题**。
    同一个 Consumer Group 里的多个 Consumer 可以共同消费一个 Topic，不同 Partition 会分配给不同 Consumer，这样可以并行消费，提高吞吐。
    同一个 Partition 在一个 Consumer Group 内同一时刻只能被一个 Consumer 消费，所以也能避免组内重复消费。
    如果某个 Consumer 挂了，Kafka 会触发 rebalance，把它负责的 Partition 分配给其他 Consumer，保证消费继续进行。
    另外，不同 Consumer Group 之间互不影响，可以各自独立消费同一份消息，比如一个组做订单处理，另一个组做数据统计。

### Kafka 可以脱离 ZooKeeper 吗？

早期 Kafka 依赖 ZooKeeper 管理元数据和 Controller 选举；KRaft 模式由 Kafka 内置 Raft 管理元数据，不依赖 ZooKeeper。Kafka 3.x 的存量集群可能仍使用 ZooKeeper，但新建集群应直接使用 KRaft；面向 Kafka 4.2 的升级必须先完成 KRaft 迁移，因为 ZooKeeper 模式已移除。

#### 常见追问

- KRaft 解决什么问题？
    KRaft **主要解决 Kafka 对 ZooKeeper 的依赖问题**。
    早期 Kafka 的元数据，比如 Broker、Topic、Partition、副本信息，都是通过 ZooKeeper 管理的，这会让架构更复杂，运维成本也更高。
    KRaft 模式下，Kafka 自己通过内置的 Raft 协议管理集群元数据，不再需要 ZooKeeper。
    这样可以简化部署和运维，也减少 Kafka 和 ZooKeeper 之间的协调开销。
    同时元数据管理由 Kafka Controller Quorum 负责，Controller 切换更快，集群扩展能力和稳定性也更好。
    需要注意，KRaft 是新版本的标准架构；早期 Kafka 版本仍依赖 ZooKeeper。ZooKeeper 集群迁移到 KRaft 需要经过兼容桥接版本，Kafka 3.9 是官方文档所述的最后桥接版本，不能直接将旧 ZooKeeper 集群升级到 Kafka 4.2。
- ZooKeeper 在旧 Kafka 中负责什么？
    旧版 Kafka 中，ZooKeeper **主要负责集群元数据管理和协调**。
    比如 Broker 上下线注册、Controller 选举、Topic 和 Partition 的元数据信息、Partition 副本分配和 ISR 状态等。
    Kafka 的 Controller 也是通过 ZooKeeper 选举出来的，Controller 负责分区 Leader 选举、副本状态变更这些集群管理操作。
    如果 Broker 宕机，ZooKeeper 会感知节点变化，然后通知 Controller 进行对应的分区迁移和 Leader 重新选举。
    早期版本里 Consumer Offset 也存过 ZooKeeper，后来主要存到 Kafka 内部的 `__consumer_offsets` 主题里。
    KRaft 模式之后，这些元数据管理能力由 Kafka 自己的 Controller Quorum 接管，不再依赖 ZooKeeper。
- Kafka 为什么要去 ZooKeeper？
    Kafka 去掉 ZooKeeper，**主要是为了简化架构和提升元数据管理能力**。
    以前 Kafka 依赖 ZooKeeper 管理 Broker、Topic、Partition、副本、Controller 选举等元数据，等于一套 Kafka 集群还要维护一套 ZooKeeper 集群，部署和运维更复杂。
    而且 Kafka 和 ZooKeeper 之间存在双系统协调，Controller 故障切换、元数据同步这些操作会有额外开销。
    KRaft 模式下，Kafka 自己通过 Raft 协议维护元数据，由 Controller Quorum 负责选主和元数据复制，不再依赖 ZooKeeper。
    这样集群架构更简单，Controller 切换更快，元数据传播更高效，也更适合大规模 Topic 和 Partition 场景。
    ZooKeeper 模式已在 Kafka 4.2 移除；当前 Kafka 后续版本以 KRaft 模式为准。

### Kafka 数据保留策略有哪些？

Kafka 数据保留可以按时间和大小控制，例如保留 7 天或保留到日志段总大小达到阈值。只要任一条件满足，就可能触发清理。Kafka 还支持基于 key 的日志压缩 compaction，用于保留最新状态。

#### 常见追问

- 同时设置 7 天和 10G 怎么清理？
    Kafka 同时设置保留 7 天和 10G 时，不是同时满足才清理，而是**任意一个条件满足就可能清理**。
    也就是数据超过 7 天会被清理，或者某个分区日志大小超过 10G，也会从最老的日志段开始清理。
    需要注意，Kafka 是按 Log Segment 粒度删除的，不是一条消息一条消息删，而且当前正在写入的 active segment 不会被删除。
    所以**实际清理可能不会刚好卡在 7 天或 10G，而是等后台定时检查时，删除符合条件的旧 segment**。
    如果是 `10G` 这个限制，要注意它通常是针对每个分区的日志大小限制，不是整个 Topic 总大小。
- Kafka 删除消息是立刻删除吗？
    不是立刻删除。Kafka 的消息清理是**后台线程定期检查**的，满足 `retention.ms` 或 `retention.bytes` 后，按 Log Segment 粒度删除，不会精确删除某一条消息。
    而且当前正在写入的 active segment 一般不会被删除，要等它滚动成普通 segment 后才可能被清理。
    符合清理条件的 segment 通常会先被标记或改名为待删除文件，再经过一定延迟后由系统真正删除，所以磁盘空间释放也不是实时的。
    如果是日志压缩 `log.compaction` 场景，也不是立刻清理旧值，而是后台 Cleaner 异步合并清理。
- compaction 和 delete 有什么区别？
    `delete` 是按时间或大小删除旧的 Log Segment，主要受 `retention.ms`、`retention.bytes` 控制，超过条件后旧消息会被整段删除，适合普通消息日志保留场景。
    `compaction` 是日志压缩，不是简单按时间删除，而是按消息的 key 保留最新 value，同一个 key 的旧 value 会被后台 Cleaner 清理掉，适合保留最新状态的场景，比如配置、用户状态、数据库变更日志。
    简单说，**`delete` 关注“旧数据要不要删”，可能把一段数据全部删掉；`compaction` 关注“同一个 key 只保留最新版本”，不会因为时间到了就直接删完整段**。
    Kafka 的 `cleanup.policy` 可以设置为 `delete`、`compact`，也可以同时设置 `compact,delete`，表示既做压缩，也受保留时间或大小限制。

### Kafka 变慢可能是什么原因？

常见原因包括磁盘 IO 瓶颈、网络带宽瓶颈、CPU 压力、分区过多、消费者处理慢、ISR 抖动、批量参数不合理和消息过大。排查时要结合 Broker 指标、Consumer Lag、磁盘和网络监控。

#### 常见追问

- Consumer Lag 变大怎么排查？
    Consumer Lag 变大一般先确认是持续增长还是瞬时抖动。
    如果持续增长，说明消费速度跟不上生产速度，可以从几个方向排查：
    第一，看生产端流量是否突然上涨，比如消息量、单条消息大小是否变大。
    第二，看消费者是否有异常，比如消费线程阻塞、频繁 GC、CPU 或内存打满、消费逻辑变慢。
    第三，看下游依赖是否变慢，比如数据库、Redis、HTTP 接口响应变慢，导致 poll 到消息后处理耗时增加。
    第四，看分区和消费者数量是否匹配，如果分区数太少，消费者再多也提升不了并发；如果某些分区 Lag 特别高，可能是消息 key 分布不均导致热点分区。
    第五，看是否频繁 Rebalance，比如消费者重启、心跳超时、`max.poll.interval.ms` 设置不合理，都会导致消费暂停。
    第六，看 Offset 提交是否正常，如果处理成功但 Offset 没提交，也会看到 Lag 异常。
    最后再看 Kafka Broker、磁盘、网络是否有瓶颈。
    处理上可以优化消费逻辑、提升并发、增加分区、扩容消费者、优化下游性能，必要时调整 `max.poll.records`、`max.poll.interval.ms` 等参数。
- 分区越多越好吗？
    不是，Kafka 分区不是越多越好。
    分区多的好处是可以提升并行度，增加消费者并发能力，也能提升 Topic 的吞吐能力。
    但分区太多也有成本，比如 Broker 需要维护更多日志文件和索引文件，文件句柄、内存、磁盘 IO 压力都会增加。
    分区过多还会让 Leader 选举、Controller 管理、故障恢复变慢，Rebalance 的耗时也可能变长。
    另外单个 Topic 分区太多，也可能导致消息顺序范围变小，因为 Kafka 只保证单分区内有序。
    所以分区数一般要根据吞吐量、消费者并发数、Broker 数量和未来扩展预留来定，不是越多越好。
- Kafka 为什么依赖顺序写磁盘？
    Kafka 依赖顺序写磁盘，**主要是为了提高写入性能**。
    机械磁盘随机写很慢，主要耗时在磁头寻道和旋转延迟；顺序写可以避免大量随机寻址，吞吐量接近内存级别。即使是 SSD，顺序写也通常比随机写更稳定。
    Kafka 的消息是追加写到日志文件里的，不会在原位置修改数据，这样可以充分利用磁盘顺序 IO。
    另外 Kafka 还利用操作系统的 Page Cache，写消息时先写入 Page Cache，再由操作系统异步刷盘，所以写入延迟比较低。
    消费时也是按 offset 顺序读取，配合 Page Cache 和零拷贝机制，可以减少磁盘 IO 和数据拷贝成本。
    所以 Kafka 虽然基于磁盘存储，但通过顺序写、Page Cache 和零拷贝，仍然能做到很高的吞吐。

### Kafka 集群节点数量怎么考虑？

Kafka 节点数量不是越多越好，要结合吞吐、分区数、副本因子、磁盘容量和故障容忍度。Kafka Broker 本身节点数没有必须奇数的要求，但要保证副本合理分布和故障后仍有足够容量。KRaft 的 Controller Quorum 需要多数派，生产通常使用 3 或 5 个 Controller，遵循 `2N + 1` 规划；关键生产环境建议 Controller 与 Broker 分离部署，`broker,controller` 混部更适合开发或轻量场景。

#### 常见追问

- 副本因子通常设置多少？
    Kafka 副本因子通常设置为 3。
    这样可以在可用性、可靠性和存储成本之间取得比较好的平衡。
    副本因子为 3 时，一个 Leader 副本加两个 Follower 副本。要保护已确认消息，通常还要使用 `acks=all`、设置 `min.insync.replicas=2`，并避免不干净 Leader 选举；这种常见组合可在一个副本故障时继续安全生产。
    如果是测试环境或者对数据可靠性要求不高，可以设置为 1 或 2；如果是核心业务、跨机架部署，通常还是建议至少 3，并让副本分散在不同故障域。
    但副本因子越高，磁盘占用、网络复制开销和 Broker 压力也越大，所以不是越高越好。
- 分区数如何规划？
    分区数一般从几个因素来规划：**吞吐量、消费者并发数、Broker 数量和未来扩展空间**。
    首先要看吞吐量，如果单个分区的读写能力不够，就需要增加分区来提升并行度。
    其次看消费者并发数，同一个消费者组里，一个分区同一时间只能被一个消费者消费，所以分区数至少要能支撑目标消费并发，但也没必要远大于消费者数量。
    还要考虑 Broker 数量，分区和副本要尽量均匀分布在各个 Broker 上，避免某些 Broker 压力过大。
    生产环境一般会适当预留一些分区，避免后续扩容频繁调整；但分区太多会增加文件句柄、内存、Controller 管理、Leader 选举和 Rebalance 成本。
    所以分区数不是固定值，通常根据压测结果和业务峰值来定，在满足吞吐和并发的基础上，控制在一个合理范围内。
- Broker 扩容后数据会自动均衡吗？
    Broker 扩容后，已有数据一般不会自动均衡到新 Broker 上。
    Kafka 只会在新建 Topic 或新增分区时，可能把副本分配到新的 Broker 上；已有分区的副本不会因为加了 Broker 就自动迁移。
    如果要让新 Broker 分担已有数据和流量，需要做分区副本重分配，比如使用 `kafka-reassign-partitions.sh`，或者用 Cruise Control 这类工具自动规划迁移。
    迁移过程中要注意控制限流，避免大量数据复制影响线上读写。
    另外，副本迁移解决的是数据分布问题，Leader 分布如果不均衡，还需要做 Leader 负载均衡，避免请求压力集中在少数 Broker 上。

## Apache Pulsar

### Pulsar 是什么？

Apache Pulsar 是分布式消息与流式数据平台，支持发布订阅、持久化消息、多租户、跨集群复制和多种订阅模式。它既可以作为消息队列，也可以承载流式数据场景。

#### 常见追问

- Pulsar 适合什么场景？
    Pulsar 比较**适合对消息可靠性、扩展性和多租户隔离要求比较高的场景**。
    它的架构是**计算和存储分离**，Broker 主要负责服务接入，数据存到 BookKeeper，所以在大规模 Topic、分区扩展、存储扩容方面比较灵活。
    如果业务里有大量 Topic、多租户、多团队共用一套消息集群，Pulsar 的命名空间、权限、限流和隔离能力会比较有优势。
    另外它同时支持消息队列和流式消费模型，也支持延迟消息、定时消息、消息重试、死信队列这类能力，适合订单、通知、IoT、日志采集、事件驱动这类场景。
    如果只是普通高吞吐日志流处理，Kafka 生态更成熟；如果特别看重多租户、存储弹性和队列能力，Pulsar 会更合适。
- Pulsar 和传统 MQ 有什么区别？
    Pulsar 和传统 MQ 的主要区别在**架构和使用模型上**。
    传统 **MQ 一般是 Broker 本身负责消息存储和转发**，架构相对简单；**Pulsar 是计算和存储分离**，Broker 负责接入和路由，消息存储在 BookKeeper，扩容 Broker 和扩容存储可以分开做。
    在消费模型上，传统 MQ 更偏队列模型；Pulsar 同时支持队列消费和发布订阅，也支持类似 Kafka 的流式消费。
    在扩展性上，Pulsar 对大量 Topic、多租户、跨地域复制支持比较好，有租户、命名空间、权限、限流这些隔离能力。
    在消息能力上，Pulsar 原生支持延迟消息、重试队列、死信队列、消息回溯等，功能比较完整。
    简单说，**传统 MQ 更适合架构简单、业务队列场景；Pulsar 更适合大规模、多租户、弹性存储和同时需要队列加流式能力的场景**。
- Pulsar 的 Topic 格式是什么？
    Pulsar 的 Topic 标准格式是：`persistent://tenant/namespace/topic`。
    其中 `persistent` 表示持久化 Topic，`tenant` 是租户，`namespace` 是命名空间，`topic` 是具体的 Topic 名称。
    如果是不持久化 Topic，格式是：`non-persistent://tenant/namespace/topic`。
    比如：`persistent://public/default/order-topic`。
    Pulsar 也支持简写形式，比如只写 `order-topic`，默认会解析到 `persistent://public/default/order-topic`，但生产环境一般建议使用完整 Topic 名称，避免环境和命名空间混淆。

### Pulsar 和 Kafka 有什么区别？

Kafka 以分区日志模型为核心，生态成熟，适合高吞吐日志、埋点和实时计算。
Pulsar 强调云原生、多租户、消息与流统一，并采用 Broker 与 BookKeeper 存储分离架构，计算层和存储层可以独立扩展。

#### 常见追问

- Kafka 的优势是什么？
Kafka 的优势**主要在生态成熟度和使用普及度上**。
Kafka 出现更早，社区、文档、案例、运维经验都更成熟，很多公司已经把 Kafka 作为**日志采集、数据同步、实时计算**的标准组件。
Kafka 和大数据生态结合更好，比如 Flink、Spark、Kafka Connect、Debezium、ClickHouse 这些组件对 Kafka 的支持都比较完善。
从性能上看，Kafka 的顺序写磁盘和零拷贝机制很成熟，在高吞吐日志流、事件流场景下表现很稳定。
运维上 Kafka 架构相对更简单，核心就是 Broker 集群加存储；当前新版本以 KRaft 管理元数据，整体使用门槛和排查经验更普遍。
所以如果是**日志采集、实时数仓、流式计算、事件总线**这类场景，Kafka 通常更稳妥；如果更看重多租户、存储计算分离、大量 Topic 和队列能力，Pulsar 会更有优势。
- 什么场景优先选 Pulsar？
    优先选 Pulsar 的场景主要有这些：
    如果**业务需要同时支持消息队列和发布订阅**，Pulsar 会比较合适，它的消费模式更灵活。
    如果 Topic 数量很多，或者需要大量小 Topic，Pulsar 更适合，因为它的存储和计算分离，对大量 Topic 的支持更好。
    如果是多租户场景，比如不同业务线、不同团队共用一套消息平台，Pulsar 有 tenant、namespace、权限、限流这些隔离能力。
    如果需要跨机房、跨地域复制，Pulsar 原生支持 geo-replication，做多活或者异地容灾比较方便。
    如果业务很依赖延迟消息、重试队列、死信队列、消息回溯这些能力，Pulsar 原生支持会更完整。
    如果希望 Broker 无状态、扩容更灵活，或者存储容量和计算能力需要分别扩展，Pulsar 的架构会更有优势。

### Pulsar 的订阅模式有哪些？

Pulsar 常见订阅模式包括 `Exclusive`、`Failover`、`Shared` 和 `Key_Shared`。
`Exclusive` 只允许一个消费者；`Failover` 支持主备切换；`Shared` 支持多个消费者并发消费；`Key_Shared` 可以让相同 key 的消息进入同一个消费者，兼顾并发和局部有序。

#### 常见追问

- `Shared` 为什么不保证全局顺序？
    Shared 不保证全局顺序，**核心原因是它会把同一个订阅下的消息分发给多个消费者并行处理**。
    Broker 可能按顺序把消息投递出去，比如 msg1 给 consumerA，msg2 给 consumerB，但两个消费者的处理速度不同，consumerB 可能先处理完 msg2，这样业务看到的完成顺序就不是 msg1、msg2。
    另外 Shared 模式下 ack 是各个消费者独立提交的，如果某个消费者处理慢、失败或者触发重投递，后面的消息也可能先被消费成功，所以无法保证全局有序。
    Shared 更关注的是并发消费和提升吞吐，不适合要求严格顺序的场景。
    如果要全局有序，一般用 Exclusive 或 Failover，并且避免多分区；如果只要求相同 key 内有序，可以用 Key_Shared。
- `Failover` 和 `Exclusive` 有什么区别？
    Exclusive 和 Failover 都不是多个消费者同时消费同一条消息，区别主要在是否允许备消费者存在。
    Exclusive 模式下，一个订阅只能有一个消费者连接，其他消费者再连接会失败，所以它适合单消费者消费、严格顺序处理的场景。
    Failover 模式下，一个订阅可以有多个消费者连接，但同一时间只有一个消费者是 active，其他消费者是 standby；active 挂了以后，Broker 会自动切换到其他消费者继续消费。
    所以 Exclusive 更像“只能有一个人消费”，Failover 更像“主备消费”。
    顺序性上，非分区 Topic 下两者都能保证消费顺序；如果是分区 Topic，一般只能保证单个分区内有序，不能保证全局有序。
- `Key_Shared` 适合哪些业务？
    `Key_Shared` 适合既需要并发消费，又要求同一业务标识下消息有序的场景，例如：
    - 同一订单的状态变更按顺序处理
    - 同一用户的事件按顺序处理
    - 同一账户的交易或余额变更
    - 同一设备的上报消息处理
    它会把相同 key 的消息分配给同一个消费者，不同 key 可以由多个消费者并行处理，因此实现局部有序并提升吞吐。
    使用时生产者必须设置稳定的消息 key；在 `allowOutOfOrderDelivery=false` 的默认有序语义下，同一 key 才能持续由单个消费者按顺序处理。开启批处理时，Key-Based Batching 是保持 key 粒度批处理的稳妥选择；不同 Pulsar 版本对批内 key 提取和分发的实现存在差异，不应简单断言普通 batching 必然破坏路由。它不保证不同 key 之间的全局顺序。

### Pulsar 为什么适合多租户？

Pulsar 原生提供 Tenant 和 Namespace 模型，可以按租户、业务线或环境隔离 Topic、权限、配额和策略。平台型系统中，不同团队可以共享同一套集群，同时保持资源和权限边界。

#### 常见追问

- Tenant 和 Namespace 分别解决什么问题？
    **Tenant** 主要解决多租户隔离问题，是 Pulsar 里最高层的逻辑隔离单位，通常对应一个组织、业务线或团队。它可以配置哪些集群可用、租户级权限，避免不同团队的资源和权限混在一起。
    **Namespace** 主要解决 Topic 的分组管理问题，属于某个 Tenant，通常对应一个具体业务模块或环境，比如订单、支付、日志。很多策略都是配置在 Namespace 级别的，比如消息保留、TTL、订阅权限、限流、复制策略、Schema 策略等。
    简单说，Tenant 偏组织和资源隔离，Namespace 偏 Topic 的统一治理和策略管理。Topic 的完整层级一般是 `persistent://tenant/namespace/topic`。
- 多租户和普通 Topic 命名规范有什么区别？
    多租户命名不是单独的一种 Topic 类型，而是要求 Topic 放在明确的 `tenant/namespace` 下面。
    Pulsar 完整 Topic 一般是：`persistent://tenant/namespace/topic`。
    其中 `tenant` 通常按团队、业务线、组织来命名，解决资源和权限隔离；`namespace` 通常按环境、应用、模块来命名，方便统一配置保留策略、TTL、权限、限流等；`topic` 才是具体的消息主题名。
    普通写法如果只写一个 Topic 名，可能会落到默认的 `public/default` 下，比如 `persistent://public/default/topic`，这种更适合测试或简单场景，生产上不建议这么用。
    所以区别是：**多租户命名强调显式的组织隔离和策略管理，普通 Topic 名更偏具体消息通道本身**。
- 多业务共用集群要注意什么？
    多业务共用 Pulsar 集群时，重点是**做好隔离和治理**。
    首先要按业务划分 `Tenant` 和 `Namespace`，不要都放在 `public/default`，这样权限、策略和资源才好管理。
    其次要配置认证和授权，避免业务之间互相读写 Topic。
    然后要在 Namespace 级别设置合理的保留时间、TTL、积压限制、生产消费限流，防止某个业务消息堆积或流量突增影响整个集群。
    还要注意 Topic、订阅名、Schema 的命名规范，避免冲突，也方便排查问题。
    对于核心业务，可以考虑单独的 Namespace，甚至做 BookKeeper 的隔离策略或独立集群，避免被低优先级业务拖慢。
    监控上要按租户、Namespace、Topic 维度看生产速率、消费延迟、Backlog、存储占用、失败率，提前发现某个业务影响集群稳定性。
    简单说，**共用集群可以，但不能共用一套默认配置，要做到权限隔离、资源限制、策略独立和监控可见**。

### Pulsar 的 Broker 和存储分离有什么好处？

Pulsar Broker 负责客户端接入、消息路由和调度，BookKeeper 负责持久化存储。
两者分离后，计算和存储可以独立扩容，Broker 故障后也更容易由其他 Broker 接管 Topic 服务。

#### 常见追问

- BookKeeper 在 Pulsar 中负责什么？
    BookKeeper 在 Pulsar 中**主要负责消息的持久化存储**。
    Pulsar Broker 本身更偏计算和服务入口，负责接收生产者消息、分发给消费者、处理订阅等逻辑；真正的消息数据会写到 BookKeeper 的 Bookie 节点里。
    BookKeeper 提供的是分布式日志存储能力，消息会按 Ledger 追加写入，并通过多副本机制保证数据可靠性。
    这样 Broker 是无状态的，Broker 挂了以后，其他 Broker 仍然可以从 BookKeeper 读取数据继续服务。
    简单说，BookKeeper 负责“可靠存消息”，Broker 负责“接入和转发消息”。
- 存储分离会带来哪些复杂度？
    Pulsar 存储计算分离的好处是 Broker 可以做得比较无状态，扩缩容更灵活，但也会带来一些复杂度。
    首先是**组件更多**，除了 Broker，还要维护 BookKeeper、ZooKeeper 或元数据服务，整体运维和故障排查会更复杂。
    其次是**链路更长**，消息写入要经过 Broker 再落到 Bookie，读历史消息也要从 BookKeeper 拉取，所以网络、磁盘、Bookie 状态都会影响延迟。
    然后是**容量规划更复杂**，Broker 看的是连接数、Topic 数、吞吐；Bookie 看的是磁盘、IO、Ledger 数量和副本分布，两边都要单独评估。
    再就是**故障场景更多**，比如 Broker 正常但 Bookie 写入慢，或者某些 Bookie 下线导致 Ledger 恢复、重复制，都会影响生产和消费。
    另外，**多租户共用时还要考虑 BookKeeper 层面的资源隔离**，否则某个业务大量堆积消息，可能拖慢整个存储层。
    所以存储计算分离提升了弹性和可靠性，但代价是**架构链路、运维、监控和故障定位都会更复杂**。
- Kafka 和 Pulsar 的架构差异体现在哪里？
    Kafka 和 Pulsar 的架构差异**主要体现在存储模型和组件职责上**。
    Kafka 是存储和计算相对耦合的架构，Broker 既负责客户端读写，也负责本地分区日志存储，Partition 分布在哪个 Broker 上，数据也就主要存在哪个 Broker 上。
    Pulsar 是存储计算分离架构，Broker 主要负责接入、路由、协议处理和消费分发，消息数据持久化在 BookKeeper 的 Bookie 节点上，所以 Broker 更偏无状态。
    在扩展方式上，Kafka 扩容 Broker 后通常还要做分区迁移和副本重平衡；Pulsar 扩 Broker 主要提升接入和计算能力，扩 Bookie 主要提升存储和 IO 能力，扩展维度更细。
    在 Topic 模型上，Kafka 的 Topic 由 Partition 组成，分区是并行和顺序的核心；Pulsar 也有分区 Topic，但底层存储是 Ledger，消息会写入 BookKeeper。
    在消费模型上，Kafka 主要通过 Consumer Group 实现发布订阅和负载均衡；Pulsar 支持更多订阅模式，比如 Exclusive、Failover、Shared、Key_Shared，模型更灵活。
    简单说，**Kafka 架构更简单直接，Broker 自己管计算和存储；Pulsar 架构更分层，弹性和多租户能力更强，但组件和运维复杂度也更高**。

### Pulsar 的 Key_Shared 订阅解决什么问题？

`Key_Shared` 订阅可以让相同 key 的消息分发给同一个消费者，从而在 `allowOutOfOrderDelivery=false` 时保证同一业务维度的顺序，同时允许不同 key 的消息并发消费。它适合订单、用户、设备等需要局部有序的业务。

#### 常见追问

- 局部有序和全局有序有什么区别？
    局部有序指的是按 key 有序，也就是相同 key 或 ordering key 的消息会分发给同一个 Consumer，同一个 key 内部保持顺序消费。
    全局有序指的是整个 Topic 或 Partition 上所有消息都严格按写入顺序消费，不区分 key，这种要求下通常就不能让多个 Consumer 并行消费。
    所以 Key_Shared 在 `allowOutOfOrderDelivery=false` 时保证的是局部有序，不保证全局有序。不同 key 的消息可能被分发到不同 Consumer 并行处理，它们之间的先后顺序是不保证的。
    如果业务要求同一个订单、同一个用户的事件有序，用 Key_Shared 比较合适；如果要求所有消息绝对有序，就要用单分区加单 Consumer，或者 Exclusive、Failover 这类模式，但吞吐会受限制。
    另外 Key_Shared 要注意消息必须带 key。生产端批量发送时，Key-Based Batching 是保持 key 粒度批处理的稳妥选择；实际顺序语义还应结合 Pulsar 版本、批处理实现与 `allowOutOfOrderDelivery` 配置验证。
- key 设计不合理会有什么问题？
    Key_Shared 里 key 设计不合理，主要会影响顺序性、并发度和负载均衡。
    如果 key 太集中，比如大量消息都用同一个 key，那这些消息会一直分到同一个 Consumer，导致热点 Consumer，整体并发能力上不去。
    如果 key 太分散或者没有业务含义，虽然并发高了，但同一业务实体的消息可能不能按预期聚合处理，业务上的顺序性就不好保证。
    如果同一个业务对象的 key 不稳定，比如有时用订单号、有时用用户号，那相关消息可能被分到不同 Consumer，导致同一对象的处理顺序被打乱。
    另外，如果 key 分布不均，还可能造成部分 Consumer 很忙、部分 Consumer 很空，出现消费延迟不均衡。
    所以 Key_Shared 的 key **一般要选择真正需要保证顺序的业务维度，同时保证分布尽量均匀**，比如订单维度、用户维度这类稳定标识。
- `Key_Shared` 和 Kafka 分区有序有什么区别？
    Key_Shared 和 Kafka 分区有序的**核心区别是有序粒度不一样**。
    Kafka 是分区有序，消息只要进入同一个 Partition，就按 Partition 内的写入顺序消费；key 主要是用来决定消息路由到哪个 Partition。同一个 Consumer Group 里，一个 Partition 同一时刻只能被一个 Consumer 消费，所以并发度受 Partition 数量限制。
    Pulsar 的 Key_Shared 是 key 维度有序：在 `allowOutOfOrderDelivery=false` 时，相同 key 的消息会分给同一个 Consumer 并保持该 key 内顺序；不同 key 可以分给不同 Consumer 并行处理，它不强调不同 key 之间的顺序。
    所以 Kafka 是“Partition 作为顺序和并发的基本单位”，一个分区里的多个 key 也会被串行消费；Pulsar Key_Shared 是“key 作为顺序单位”，不同 key 之间可以更细粒度并发。
    扩容上也有差别，Kafka 增加 Consumer 超过 Partition 数没有意义；Pulsar Key_Shared 可以按 key 的分布在多个 Consumer 间分摊，但如果 key 热点明显，也会出现单个 Consumer 压力过大的问题。
