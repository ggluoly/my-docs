# 并发编程面试题

本页整理 Java 并发面试高频题，重点覆盖线程基础、线程通信、线程池、锁、`ThreadLocal`、AQS、CAS 和原子类。回答时先说明结论，再补充底层机制和工程取舍。

## 线程基础

### 创建线程有哪几种方式？

直接创建并启动线程的常见方式包括继承 `Thread`、将 `Runnable` 交给 `Thread`、将 `Callable` 包装为 `FutureTask` 后交给 `Thread`。Java 21 还可以通过 `Thread.ofVirtual().start(...)` 创建虚拟线程。
工程中通常不直接管理线程，而是向线程池提交任务，由线程池按需创建或复用工作线程；虚拟线程场景可使用每任务一个虚拟线程的执行器。

#### 常见追问

- 为什么生产环境不推荐直接 `new Thread()`？
  频繁创建和销毁线程成本高，会消耗 CPU 和内存资源。
  线程数量不可控，如果请求量一大，每个请求都 new Thread()，可能导致线程过多，甚至 OOM。
  缺少统一管理，像线程命名、异常处理、监控、超时控制、拒绝策略都不好做。
  任务执行结果也不好获取，不像线程池可以配合 Future。
  线程复用能力差，执行完就销毁，资源利用率低。
- 线程池如何复用线程？
  线程池复用线程的核心是：线程执行完一个任务后不会立即销毁，而是继续从任务队列里取下一个任务执行。
- 线程池里面的线程什么情况下才会销毁？
  线程池中的工作线程一般在以下情况会销毁：
  1. 线程池处于 `STOP` 状态，例如调用 `shutdownNow()` 后，线程池会尝试中断工作线程；任务响应中断并结束后，线程才会退出。
  2. 线程池调用 `shutdown()` 后进入 `SHUTDOWN` 状态，不再接收新任务；已提交任务和队列任务执行完毕后，空闲工作线程会退出，最终线程池进入 `TERMINATED`。
  3. 非核心线程空闲时间超过 `keepAliveTime` 会被回收。
  4. 调用 `allowCoreThreadTimeOut(true)` 后，核心线程空闲超过 `keepAliveTime` 也会被回收；默认核心线程不会因空闲而销毁。
  5. 工作线程执行任务时发生未捕获的运行时异常或 `Error`，该线程会退出；如果线程池仍需要维持线程数，可能会创建新线程补充。
  ***·注意·***，`shutdownNow()` 只是发出中断请求，任务不响应中断、死循环或处于不可中断阻塞时，线程不一定能立即销毁。

### Runnable 和 Callable 有什么区别？

Runnable 的 run() 没有返回值，不能直接声明受检异常；Callable 的 call() 有返回值，并且可以声明抛出异常。
Executor.execute() 只接收 Runnable，因此 Callable 不能直接传给 execute()。最常见的用法是通过 submit(Callable) 提交，并获得 Future。
但 submit() 不是 Callable 的唯一执行方式：可以将 Callable 包装成 FutureTask，再交给 execute() 或 new Thread(futureTask) 执行；ExecutorService.invokeAll() 和 invokeAny() 也可以直接接收 Callable 集合。

#### 常见追问

- `Future` 如何获取执行结果？
  Future 获取执行结果主要是调用 get() 方法。
  比如：
    `Future<String> future = executorService.submit(callable)`;
    `String result = future.get()`;
  这里 get() 会阻塞当前线程，直到任务执行完成并返回结果。
  如果不想一直阻塞，可以用带超时时间的：
    `future.get(3, TimeUnit.SECONDS)`;
  如果任务执行过程中抛异常，调用 get() 时会抛出 `ExecutionException`，真正的异常在 `getCause()` 里。
  如果任务被取消了，`get()` 会抛 `CancellationException`。
  一般配合 `isDone()` 判断是否完成，或者用超时 `get()`，避免主线程无限等待。
- `FutureTask` 的作用是什么？
  主要是把 Callable 或 Runnable 包装成一个既可以执行、又可以获取结果的任务。同时实现了 Runnable 和 Future 接口，所以有两个能力：
  可以作为 Runnable 被线程执行，比如 new Thread(futureTask).start()。
  可以作为 Future 获取执行结果，比如调用 futureTask.get()。
  **FutureTask 的作用就是：把异步任务和结果获取能力封装在一起。**
- 异步任务异常怎么处理？
  异步任务的异常不能只依赖日志，因为异常可能被线程池吞掉，关键是按提交方式处理。
  如果用 submit() 提交任务，通过 future.get() 获取，捕获 ExecutionException，再通过 getCause() 拿到真实异常：
  如果用 execute() 提交 Runnable，通过自定义线程工厂统一设置异常处理器。
  如果是 CompletableFuture，使用 exceptionally() 做兜底，或者用 handle() 同时处理正常结果和异常。
  在 Spring 的 @Async 中，返回 Future 或 CompletableFuture 时同样通过结果对象处理；如果返回 void，需要配置 AsyncUncaughtExceptionHandler。
  实际项目里会统一记录任务参数、链路标识和完整异常；对于可重试任务再结合重试机制或消息队列补偿，避免异常只被记录但业务没有恢复。

### run 和 start 有什么区别？

`start()` 会启动一个新线程，由 JVM 调度后执行 `run()`；直接调用 `run()` 只是普通方法调用，不会创建新线程。一个线程对象只能调用一次 `start()`，重复调用会抛出 `IllegalThreadStateException`。

#### 常见追问

- 为什么线程启动只能调用一次？
  因为它只能经历一次线程生命周期。首次调用 `start()` 时，线程状态从 `NEW` 变为 `RUNNABLE`，JVM 为它建立执行实体并进行调度；执行结束后状态变为 `TERMINATED`。再次调用 `start()` 时，该对象已不处于 `NEW` 状态，会抛出 `IllegalThreadStateException`。
  JDK 8 中，`Thread` 对应的平台线程，通常由 JVM 创建并管理底层原生线程。Java 21 的虚拟线程则由 JVM 调度到载体线程上运行，但同一个虚拟线程对象同样只能调用一次 `start()`。
  需要重复执行相同逻辑时，应创建新的 `Thread` 对象，实际开发中更常用线程池提交任务。`run()` 可以多次调用，但只是普通方法调用，不会创建新线程。
- 线程启动后什么时候执行由谁决定？
  以 **JDK 8** 的平台线程为例，调用 `start()` 后线程进入 `RUNNABLE` 状态，但不代表立即获得 CPU 执行。具体何时执行由操作系统线程调度器决定，JVM 无法精确控制。
  因此不能依赖多个线程调用 `start()` 的先后顺序来推断实际执行顺序；有顺序要求时，应使用 `join()`、`CountDownLatch`、`Semaphore`、`CyclicBarrier` 或锁等同步机制协调。
  **Java 21** 的虚拟线程中，虚拟线程由 JVM 调度器调度到载体线程上运行，载体线程最终仍由操作系统调度；同样不能依赖线程启动顺序或实际执行顺序。

### Java 线程有哪些状态？

Java 线程状态包括 `NEW`、`RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING`、`TERMINATED`。其中 `RUNNABLE` 包含就绪和运行中，是否真正占用 CPU 由操作系统调度决定。

#### 常见追问

- `BLOCKED` 和 `WAITING` 有什么区别？
  BLOCKED 是线程在等待获取 synchronized 的监视器锁，比如其他线程还没有释放锁；拿到锁后才会继续执行。此时调用 interrupt() 不会让它立即退出锁竞争。
  WAITING 是线程主动进入无限期等待，通常由 Object.wait()、Thread.join()、LockSupport.park() 触发；需要其他线程通过 notify()、unpark()、目标线程结束，或中断来唤醒。
  简单说，**BLOCKED 是“等锁”，WAITING 是“等通知或条件成立”**。
- `sleep()` 后线程进入什么状态？
  会进入 TIMED_WAITING 状态。等待时间结束后，线程回到 RUNNABLE 状态，等待操作系统再次调度执行。sleep() 不会释放已经持有的 synchronized 监视器锁；如果需要释放锁等待条件，应使用 wait()。
- 如何查看线程状态？
  代码中调用 `thread.getState()`，可以得到 Java 线程状态，例如 RUNNABLE、BLOCKED、WAITING、TIMED_WAITING。
  线上排查常用 `jstack <pid> `导出线程快照，查看每个线程的状态、调用栈以及是否在等锁。
  使用 `jcmd <pid> Thread.print`，效果和 jstack 类似。
  图形化工具可以用 JConsole、VisualVM 或 Arthas 的 thread 命令。 需要注意，jstack 里显示的 RUNNABLE 不一定代表线程正在占用 CPU，也可能处于可运行但等待系统调度的状态。

### 守护线程是什么？

守护线程是为用户线程提供后台服务的线程，例如 GC 线程。当 JVM 中只剩守护线程时，JVM 会退出，守护线程也会随之结束。业务关键任务不应该依赖守护线程保证最终执行完成。

#### 常见追问

- 如何设置守护线程？
  对平台线程，可以通过 `Thread#setDaemon(true)` 设置，并且必须在线程启动前调用：
  ```
  Thread thread = new Thread(task);
  thread.setDaemon(true);
  thread.start();
  ```
  守护线程会随所有用户线程结束而自动退出，常用于垃圾回收、监控或后台任务。平台线程已经调用 `start()` 后再设置会抛出 `IllegalThreadStateException`。
  Java 21 的虚拟线程固定为守护线程：调用 `setDaemon(true)` 没有效果，不能将其设为非守护线程，调用 `setDaemon(false)` 会抛出 `IllegalArgumentException`。
- 守护线程适合做什么？
  适合执行不需要保证完成的后台辅助任务，例如：
    日志刷新、监控采集、定时清理缓存。
    心跳检测、连接保活等后台维护任务。
    JVM 的垃圾回收线程也是守护线程。 核心原则是：任务可以随 JVM 退出直接终止，不能用于订单落库、文件写入、消息消费等必须可靠完成的业务。
- 为什么不能把业务落库任务放到守护线程？
  因为当所有非守护线程结束后，JVM 会直接退出，守护线程不会等待任务执行完成。
  如果把落库放在守护线程，可能出现数据还没提交、事务没完成、连接没释放，进程就退出了，导致数据丢失或状态不一致。
  所以核心业务任务要使用用户线程或线程池，并配合事务、重试和优雅停机保证执行完成。

## 线程通信

### sleep 和 wait 有什么区别？

`sleep()` 是 `Thread` 的静态方法，让当前线程暂停一段时间，不释放已持有的锁；`wait()` 是 `Object` 的方法，当前线程必须持有等待目标对象的监视器才能调用。`wait()` 会释放该对象的监视器，并在被 `notify()`、`notifyAll()`、中断，或带超时参数的等待到期后重新竞争该监视器。

#### 常见追问

- 为什么 `wait()` 必须在同步代码块里调用？
  更准确地说，调用 `wait()` 时当前线程必须持有该 `wait` 目标对象的监视器；通常写在 `synchronized (obj)` 中，也可以在锁定该对象的同步方法中调用。
  `wait()` 会原子地释放目标对象的监视器并进入该对象的等待集合，之后必须重新获取该监视器才能返回；如果没有持有目标对象的监视器，会抛出 `IllegalMonitorStateException`。
- `wait()` 被唤醒后会立刻执行吗？
  不会立刻执行。notify() 或 notifyAll() 只是让等待线程从 WAITING 状态进入锁竞争状态，它还需要重新获取对应对象的监视器锁。只有拿到锁后，wait() 才会返回并继续执行同步代码；如果锁仍被其他线程持有，就会继续阻塞。
- `sleep()` 会释放锁吗？
  不会。Thread.sleep() 只是让当前线程进入 TIMED_WAITING 状态，线程仍然持有已经获取的 synchronized 锁。
  因此其他线程无法进入同一个对象的同步代码块，直到该线程睡眠结束并退出同步块释放锁。

### notify 和 notifyAll 有什么区别？

`notify()` 会从等待该对象监视器的等待集合中唤醒任意一个线程，具体选择顺序不保证随机、公平或 FIFO。
`notifyAll()` 会唤醒该对象监视器等待集合中的所有线程。
两者都必须在持有该对象监视器的同步块或同步方法中调用，否则会抛出 `IllegalMonitorStateException`。
被唤醒的线程不会立即继续执行，需要先从等待状态转为可竞争状态，待当前线程释放对象锁后重新竞争锁。
存在多个等待条件时，通常使用 `notifyAll()` 更安全，避免 `notify()` 唤醒不满足执行条件的线程，而真正可执行的线程一直无法被唤醒。

#### 常见追问

- 为什么推荐用 while 判断等待条件？
  因为 wait() 可能发生虚假唤醒，而且被 notify() 唤醒后，线程重新获得锁之前，等待条件可能已经被其他线程修改。
  使用 while 可以在线程每次被唤醒并重新拿到锁后，再次判断条件；条件不满足就继续 wait()，避免错误执行。
  if 只判断一次，发生虚假唤醒或条件被抢占时，线程可能直接向下执行，造成并发问题。
- 什么是虚假唤醒？
  指线程调用 wait() 后，在没有收到对应 notify() / notifyAll() 通知、等待条件也未满足的情况下，仍然可能从等待状态返回。
  这是 JVM 和操作系统层面允许的行为，所以不能假设 wait() 返回就代表条件成立。通常写成：
 ```
  synchronized (lock) {
      while (!condition) {
        lock.wait();
    }
    // 条件成立后执行
  }
 ```
- `Condition` 和 `wait/notify` 有什么区别？
  wait/notify 是 Object 的监视器机制，只能配合 synchronized 使用；每个对象通常只有一组等待队列，通知粒度较粗
  Condition 是 Lock 体系的条件变量，通过 lock.newCondition() 创建，使用时必须先持有对应的 Lock。它提供 await()、signal()、signalAll()，分别对应 wait()、notify()、notifyAll()。
  主要区别是 Condition 可以为同一把锁创建多个条件队列，例如生产者等待“队列未满”，消费者等待“队列非空”，可以精准唤醒对应线程，避免 notifyAll() 带来的无效唤醒。Lock 和 Condition 支持更丰富的能力，比如可中断等待、超时等待、非阻塞获取锁；synchronized 加 wait/notify 相对更基础。


### wait、notify 和 notifyAll 为什么定义在 Object 中？

`wait()`、`notify()` 和 `notifyAll()` 都依赖对象监视器锁，线程是在某个对象的等待队列上等待和唤醒的，因此它们定义在 `Object` 中，而不是 `Thread` 中。这样任意对象都可以作为同步锁和线程协作载体。

#### 常见追问

- `notify()` 唤醒的是线程还是对象？
  notify() 唤醒的是在该对象监视器等待队列中等待的一个线程，不是唤醒对象。
  被选中的线程不会立刻执行，而是从 wait 状态进入锁竞争队列，等当前线程退出 synchronized、释放对象锁后，它才有机会重新获取锁并继续执行。
- `Condition` 为什么可以支持多个等待队列？
  因为 Condition 不是绑定在对象监视器上的固定等待队列，而是 Lock 显式创建的条件变量。一个 Lock 可以通过多次 newCondition() 创建多个 Condition 实例，每个实例内部各自维护独立的等待队列。
  线程调用某个 Condition.await() 时，会进入该 Condition 对应的队列；调用该 Condition 的 signal() 时，也只会唤醒这个队列中的线程。因此可以把不同业务条件隔离开，例如“队列为空”和“队列已满”分别使用不同的等待队列。

## 线程池

### 线程池核心参数有哪些？

线程池核心参数包括核心线程数、最大线程数、空闲线程存活时间、任务队列、线程工厂和拒绝策略。任务执行流程是核心线程、队列、非核心线程、拒绝策略。

#### 常见追问

- CPU 密集型和 IO 密集型线程池怎么设置？
  **CPU 密集型**任务主要受 CPU 核数限制，核心线程数通常设置为 CPU 核数 + 1，最大线程数一般与核心线程数接近，队列不宜过大，避免大量任务排队导致延迟升高。例如 8 核机器可先设为 8~9 个线程，再结合压测和上下文切换情况调整。
  **IO 密集型**任务大部分时间在等待网络、磁盘或数据库响应，线程数可以高于 CPU 核数。常用估算公式是：线程数 = CPU 核数 × 目标 CPU 利用率 × (1 + 等待时间 / 计算时间)；经验上可从 CPU 核数 × 2 到 CPU 核数 × 4 起步。最大线程数需结合下游容量控制，防止线程过多压垮数据库或远程服务。
  队列容量不能脱离线程数单独设置：无界队列会让最大线程数失效，还可能造成内存风险；生产中更倾向于有界队列，并配置合理的拒绝策略、监控活跃线程数、队列积压、任务耗时和拒绝次数，通过压测最终确定参数。
- 拒绝策略有哪些？
  Java `ThreadPoolExecutor` 内置四种拒绝策略：
    1. `AbortPolicy`：默认策略，直接抛出 `RejectedExecutionException`，让调用方感知任务提交失败。
    2. `CallerRunsPolicy`：由提交任务的线程自己执行任务，可以形成反压，降低任务提交速度。
    3. `DiscardPolicy`：直接丢弃当前任务，不抛异常，适合允许少量任务丢失的场景。
    4. `DiscardOldestPolicy`：移除工作队列的队首任务后，再尝试提交当前任务；对于 FIFO 队列，队首通常就是最早入队的任务。
  生产中一般优先考虑 `CallerRunsPolicy` 或自定义拒绝策略。自定义策略通常会记录日志、埋点告警，或者把任务转移到消息队列、降级处理，但要避免在拒绝策略里做长时间阻塞操作。
- 不同业务为什么要隔离线程池？
  不同业务隔离线程池，核心是避免资源争抢和故障扩散。
  隔离后，每类业务有独立的线程数、队列和拒绝策略，可以按业务特点分别调优：CPU 密集、IO 密集、核心链路、非核心异步任务的配置都不同。
  同时隔离便于监控和治理，可以明确看到哪个业务队列积压、拒绝任务或耗时异常，并单独限流、降级和扩容。

### 为什么不推荐 Executors？

`Executors` 的部分工厂方法会创建无界队列或无界线程数的线程池，在高并发或任务堆积时容易 OOM。生产环境应显式配置与场景匹配的执行器：通用业务任务通常使用带有界队列和拒绝策略的 `ThreadPoolExecutor`，定时任务使用 `ScheduledThreadPoolExecutor`，可拆分计算可考虑 `ForkJoinPool`。

#### 常见追问

- `newFixedThreadPool` 的风险是什么？
  主要风险是使用了无界 `LinkedBlockingQueue`。
  当任务提交速度超过消费速度时，任务会持续堆积，线程数又固定，最终可能导致堆内存不断增长，触发频繁 Full GC 甚至 OOM。
  另外，因为队列无界，线程池通常只会维持固定的核心线程数，`maximumPoolSize` 实际不会起作用，也无法通过扩容线程应对突发流量。
  生产环境一般不直接使用 `Executors.newFixedThreadPool`，而是显式创建 `ThreadPoolExecutor`，配置有界队列、线程工厂、拒绝策略和监控指标。
- `newCachedThreadPool` 的风险是什么？
  风险主要是线程数无上限。
  它的核心线程数是 `0`，最大线程数是 `Integer.MAX_VALUE`，并使用 `SynchronousQueue`，任务不会排队，提交一个任务通常就需要立即创建或复用一个线程。
  在突发流量、下游超时或任务阻塞时，线程会快速膨胀，导致大量线程争抢 CPU、上下文切换频繁、内存被线程栈耗尽，最终可能触发 OOM 或导致服务不可用。
  它更适合任务量可控、执行时间很短的场景。生产环境一般显式限制最大线程数，并配合超时、有界队列、拒绝策略和监控。
- 线程池队列为什么要有界？
  主要是为了给系统设置明确的容量上限。
  如果队列无界，任务生产速度高于消费速度时，任务会持续堆积，占用越来越多的堆内存，最终可能 Full GC 频繁甚至 OOM；同时最大线程数通常也不会生效，无法利用扩容线程处理短时流量。
  有界队列满了以后，线程池会继续创建非核心线程直到 `maximumPoolSize`；再满时触发拒绝策略。这样可以尽早暴露系统处理能力不足的问题，便于采取限流、降级、快速失败或转存等措施，避免请求无限积压导致整体服务失控。

- [线程池](/java-core/thread-pool)

### 线程池有哪些创建方式？

  常见有两类创建方式：
  1. 通过 `Executors` 工厂方法：
  `newFixedThreadPool`：固定线程数，适合任务量较稳定的场景。
  `newCachedThreadPool`：按需创建线程，适合短时间、轻量级异步任务，但要防止线程无限增长。
  `newSingleThreadExecutor`：单线程顺序执行任务，保证任务串行。
  `newScheduledThreadPool`：支持延迟执行和周期执行任务。
  `newWorkStealingPool`：基于 `ForkJoinPool`，适合 CPU 密集型、可拆分任务。
  2. 直接创建 `ThreadPoolExecutor`：
  对通用业务任务，生产环境更推荐显式创建 `ThreadPoolExecutor`，明确配置核心线程数、最大线程数、有界队列、线程工厂、空闲回收时间和拒绝策略，避免不合适的默认参数带来的 OOM 风险。

#### 常见追问

- 定时任务线程池底层怎么实现？
  定时任务线程池通常是 `ScheduledThreadPoolExecutor`，底层使用延迟队列 `DelayedWorkQueue` 保存任务，队列按任务下一次执行时间排序。
  工作线程从队列中取任务时，会先判断队首任务是否到期；未到期就通过 `awaitNanos` 等待，到期后执行。它使用 Leader-Follower 模式，通常只让一个线程等待最近到期的任务，其他线程等待唤醒，减少无效竞争。
  一次性延迟任务执行后结束。周期任务分为两种：`scheduleAtFixedRate` 按固定频率计算下一次计划时间；若单次执行超过周期，后续执行会延迟并在前一次结束后尽快开始，但同一个周期任务不会并发重叠执行。`scheduleWithFixedDelay` 则在本次执行完成后，再等待固定延迟执行下一次。
  周期任务若未捕获异常而异常结束，后续执行会被抑制，因此任务内部应处理可预期异常并记录监控。
  它的队列本质上是无界的，因此生产中仍需要控制任务提交速率、执行耗时和异常处理，避免任务堆积。
- `ForkJoinPool` 适合什么场景？
  适合 CPU 密集型、可递归拆分的并行计算场景，比如大数组排序、批量数据聚合、矩阵计算、文件目录遍历等。
  它会把大任务拆成多个 `ForkJoinTask`，通过工作窃取机制让空闲线程处理其他线程队列中的任务，提高 CPU 利用率。
  不适合大量阻塞型任务，比如网络 IO、数据库调用、长时间锁等待。因为线程被阻塞后会降低并行度，这类任务更适合普通的 `ThreadPoolExecutor`。

### 线程池有哪些状态？

线程池状态包括 `RUNNING`、`SHUTDOWN`、`STOP`、`TIDYING`、`TERMINATED`。
`RUNNING` 接收并处理任务；`SHUTDOWN` 不再接收新任务但处理队列任务；`STOP` 不接收新任务也不处理队列任务，并尝试中断运行中任务。

#### 常见追问

- `shutdown()` 和 `shutdownNow()` 有什么区别？
  `shutdown()` 会发起有序关闭并立即返回，不会阻塞等待线程池终止。它不再接收新任务，但正在运行的任务和已进入队列的任务会继续执行；调用方需要等待结束时，应调用 `awaitTermination()`。
  `shutdownNow()` 会尝试立即关闭并立即返回：不再接收新任务，尝试通过中断停止正在执行的任务，返回队列中尚未开始执行的任务列表。正在执行的任务是否结束取决于其是否正确响应中断；调用方同样可通过 `awaitTermination()` 等待线程池终止。
- 线程池如何优雅关闭？
  我通常按“两阶段关闭”处理：
  1. 先调用 `shutdown()`，拒绝新任务，但让正在执行和队列中的任务正常完成。
  2. 调用 `awaitTermination(timeout, unit)` 等待线程池结束。
  3. 如果超时，调用 `shutdownNow()`，中断运行中的任务，并获取队列里未执行的任务做记录或补偿。
  4. 再次 `awaitTermination()`；若仍未结束，记录告警日志，排查任务是否正确响应中断。
  5. 捕获 `InterruptedException` 时，要先调用 `shutdownNow()`，再通过 `Thread.currentThread().interrupt()` 恢复中断标记。
  任务代码也要配合中断：阻塞操作正确处理 `InterruptedException`，循环任务定期检查 `Thread.currentThread().isInterrupted()`。
- 线程池状态存在哪里？
  存储在 ThreadPoolExecutor 的 ctl 字段中，它是一个 AtomicInteger。
    ctl 将两个信息压缩在一个 int 里：
    1. 高 3 位存线程池运行状态：RUNNING、SHUTDOWN、STOP、TIDYING、TERMINATED。
    2. 低 29 位存当前工作线程数 workerCount。
   这样设计可以通过 CAS 原子地同时协调线程池状态和线程数量变化，减少并发场景下的锁竞争。

### submit 和 execute 有什么区别？

`execute()` 只能提交 `Runnable`，没有返回值；
`submit()` 可以提交 `Runnable` 或 `Callable`，会返回 `Future`。
异常处理也不同：
  `execute()` 的异常通常由线程的异常处理器处理，
  `submit()` 的异常会封装在 `Future` 中，需要调用 `get()` 才会抛出。

#### 常见追问

- 为什么 `submit()` 后异常看起来被吞了？
  `submit()` 会把任务包装成 `FutureTask`，任务执行时抛出的异常会被 `FutureTask` 捕获并保存，而不是直接交给线程的 `UncaughtExceptionHandler`，所以控制台看起来像被吞了。
  需要通过 `Future.get()` 获取结果；任务异常时会抛出 `ExecutionException`，其 `getCause()` 就是原始异常。
  如果希望异常直接打印，可以用 `execute()` 提交任务，或者对 `Future.get()` 做统一的异常处理。
- `Future.get()` 会阻塞吗？
  会阻塞，但不是一定会一直阻塞。
  当任务尚未完成时，`Future.get()` 会阻塞当前调用线程，直到任务完成、被取消或当前线程被中断。
  任务正常完成则返回结果；任务被取消抛出 `CancellationException`；任务执行异常抛出 `ExecutionException`；调用线程被中断抛出 `InterruptedException`。
  可以使用 `get(timeout, unit)` 限制最长等待时间，超时会抛出 `TimeoutException`。
- 如何设计异步任务超时？
  一般分四层设计：
  1. 调用方等待超时：使用 `Future.get(timeout, unit)`，超时后返回降级结果或抛出超时异常。
  2. 任务取消：超时后调用 `future.cancel(true)` 尝试中断执行线程。取消是协作式的，任务代码需要正确响应中断，例如捕获 `InterruptedException` 后恢复中断标记并结束执行；死循环、吞掉中断或不可中断阻塞的任务不一定会停止。
  3. 底层资源超时：HTTP、数据库、RPC 等客户端独立设置连接超时、读写超时或请求超时，保证底层 I/O 能及时返回，不能只依赖上层 `Future` 超时。
  4. 线程池隔离和治理：不同优先级或不同下游依赖使用独立线程池，避免超时任务堆积耗尽核心线程；同时增加任务状态、超时监控、限流、熔断和幂等设计。
  `CompletableFuture` 在 Java 9+ 可使用 `orTimeout()` 或 `completeOnTimeout()`，但它们通常只是让 `CompletableFuture` 以超时异常或默认值完成，不会自动停止底层已经提交并执行的任务。要停止任务，仍需要持有对应的 `Future` 并执行协作式取消，或依赖底层客户端自身的超时和取消能力。

## 锁与同步

### volatile 能保证原子性吗？

不能。`volatile` 保证可见性和有序性，但不保证复合操作的原子性。比如 `i++` 包含读取、计算、写回三个步骤，多线程下仍然会丢失更新。

#### 常见追问

- `volatile` 适合什么场景？
  volatile 适合“一写多读、只需要保证可见性和有序性、不依赖复合操作原子性”的场景。
- 什么是 happens-before？
  happens-before 是 Java 内存模型中的可见性和有序性规则：如果操作 A happens-before 操作 B，那么 A 的执行结果对 B 一定可见，并且 A 的执行顺序排在 B 之前。
  常见规则包括：同一线程内代码顺序；对同一 `volatile` 变量的写先于后续对该变量的读；对同一对象监视器的解锁先于后续对该监视器的加锁；`Thread.start()` 先于新线程中的操作；线程中的操作先于其他线程 `join()` 成功返回。
  它不等同于实际执行时间上的先后，而是用来约束多线程下内存可见性和指令重排序。

### synchronized 底层怎么实现？

`synchronized` 底层依赖对象监视器（monitor）。同步代码块在字节码中通过 `monitorenter` 和 `monitorexit` 获取、释放监视器；同步方法则带有 `ACC_SYNCHRONIZED` 标志，由 JVM 在调用时完成监视器管理。
JVM 会针对同步做优化，但具体实现随 JDK 和 JVM 版本变化。偏向锁、轻量级锁等是 JDK 8 HotSpot 常见的教学模型，不应当作所有版本的固定实现。

#### 常见追问

- 对象头里存了哪些锁信息？
  以 JDK 8 HotSpot 的经典实现为例，锁相关信息主要存放在对象头的 `Mark Word` 中，并会随锁状态变化。
  无锁时可存储对象哈希码、GC 分代年龄和锁标志位；偏向锁时会记录偏向线程 ID、时间戳和锁标志位；轻量级锁时可指向线程栈中的锁记录（Lock Record）；重量级锁时可指向 `ObjectMonitor` 监视器对象。
  `Klass Word` 用于指向类元数据，不保存锁信息。偏向锁已在较新 JDK 中移除，对象头布局和轻量级锁的具体表示也会随 JVM 版本演进，不能依赖具体位布局编写业务代码。
- 同步方法和同步代码块字节码有什么区别？
  同步方法在方法表的访问标志中带有 `ACC_SYNCHRONIZED`，字节码指令里通常看不到 `monitorenter` 和 `monitorexit`；JVM 在调用方法时自动完成加锁和释放锁。
  同步代码块会在字节码中显式生成 `monitorenter` 和 `monitorexit` 指令，并额外生成异常处理逻辑，确保发生异常时也能执行 `monitorexit` 释放锁。
  两者锁的对象取决于写法：实例同步方法锁的是当前 `this`，静态同步方法锁的是当前类的 `Class` 对象；同步代码块则可以指定任意对象作为锁。
  实际使用中，同步代码块的锁范围更精细，通常更容易降低锁竞争。
- 为什么异常退出也能释放锁？
  因为编译器会为 `synchronized` 代码块生成异常处理表：执行 `monitorenter` 后，如果代码块内抛出异常，控制流会进入异常处理逻辑，执行 `monitorexit` 释放监视器锁，然后再将异常继续抛出。
  本质上等价于在同步代码块外包了一层 `try-finally`，`finally` 中释放锁。
  对于同步方法，方法带有 `ACC_SYNCHRONIZED` 标志，JVM 会在方法正常返回或异常退出时自动释放对应的监视器锁。

### synchronized 锁升级过程是什么？

以 JDK 8 HotSpot 为例，`synchronized` 的锁状态通常可理解为：无锁、偏向锁、轻量级锁、重量级锁。这是 HotSpot 为降低无竞争或低竞争同步开销采用的实现模型，不应理解为所有 JVM 或所有 JDK 版本都固定遵循该路径。
1. 无锁：对象未被同步代码使用。
2. 偏向锁：没有竞争时，锁偏向第一个获取它的线程，后续该线程再次进入同步块时，通常不需要 CAS 加锁操作。
3. 轻量级锁：出现其他线程竞争时，偏向锁可能撤销并升级；线程通过 CAS 尝试获取锁，适合短时间、低竞争场景。
4. 重量级锁：轻量级锁竞争持续或 CAS 获取失败后，锁可能膨胀为重量级锁，线程会进入阻塞和唤醒流程，底层依赖操作系统互斥量进行调度。
在一次持续的持锁和竞争过程中，已膨胀的锁通常不会直接回退为轻量级锁；但对象完全空闲后，HotSpot 可能通过 monitor deflation 回收重量级监视器。这属于 JVM 内部资源回收，不是 Java 语言规范保证的锁降级。较新 JDK 中偏向锁已被废弃并移除，因此不能直接套用 JDK 8 的完整升级过程。

#### 常见追问
- 锁能降级吗？
  以 JDK 8 HotSpot 为例，需要区分“持锁期间的锁降级”和“空闲监视器的回收”。`synchronized` 锁一旦在竞争中膨胀为重量级锁，在本次持锁和竞争过程中不会再降回轻量级锁或无锁状态。
  但重量级锁对应的 `ObjectMonitor` 在后续完全空闲时，HotSpot 可以在安全点进行 monitor deflation，回收监视器并将对象恢复为普通的无锁状态。这属于 JVM 内部资源回收和状态缩减，不等同于线程持锁时的锁降级，也不是 Java 语言规范保证的行为。
  `ReentrantReadWriteLock` 支持业务语义上的写锁降级：线程持有写锁时先获取读锁，再释放写锁，之后继续持有读锁读取数据；不能从读锁直接升级为写锁。
- 自旋锁解决什么问题？
  解决的是线程获取锁失败后，立即阻塞、唤醒带来的上下文切换开销。
  它会让线程在用户态循环执行 CAS 尝试获取锁，预期持锁线程很快释放锁时，比挂起线程再恢复更高效。
  适合锁持有时间短、竞争不激烈、CPU 核数较多的场景。
  缺点是会占用 CPU；如果锁持有时间长或竞争激烈，空转收益很低，通常会在一定次数后升级为阻塞等待。
- 重量级锁为什么开销大？
  主要是竞争失败的线程会进入阻塞和唤醒流程，需要操作系统参与线程的挂起、恢复和调度，会产生用户态与内核态切换、上下文切换等成本。
 另外，线程被唤醒后也不一定立刻拿到锁，竞争激烈时会反复阻塞和唤醒，吞吐量会下降。
 所以它更适合锁竞争激烈、临界区执行时间较长的场景，避免线程长期自旋浪费 CPU。

### synchronized 和 ReentrantLock 有什么区别？

`synchronized` 是 JVM 内置锁，使用简单，自动释放；`ReentrantLock` 是 JUC 显式锁，支持可中断、可超时、公平锁和多个条件队列，但需要手动释放。

#### 常见追问

- `ReentrantLock` 为什么要放在 finally 中释放？
  核心是保证锁一定被释放。
  因为获取锁后，业务代码可能抛异常、提前 return，如果没有在 finally 里调用 unlock()，锁会一直被当前线程持有，其他线程就会永久阻塞，甚至造成死锁。
- 公平锁和非公平锁有什么区别？
  `ReentrantLock(true)` 在竞争时通常优先让同步队列中等待更久的线程获取锁，能降低线程长期饥饿的概率；但它不保证严格 FIFO，也不保证操作系统调度公平。公平策略需要更多排队和调度协调，吞吐量通常低于非公平锁。
  非公平锁允许新来的线程直接尝试获取刚释放的锁，可能抢在已排队线程之前成功。它通常吞吐量更高，但等待线程可能长期得不到锁。
  `ReentrantLock` 默认是非公平锁，可以通过 `new ReentrantLock(true)` 创建公平锁；无参 `tryLock()` 不遵守公平策略，仍可能插队成功。
- `tryLock()` 适合什么场景？
  适合不希望线程无限等待锁的场景。它会立即尝试获取锁，失败后可以执行降级逻辑、返回失败或处理其他任务，避免线程长时间阻塞。
  带超时的 `tryLock(timeout, unit)` 适合需要控制等待时长的场景，避免多个资源互相等待导致死锁。
  常见应用包括高并发下的快速失败、缓存更新、任务去重，以及需要同时获取多把锁的业务。
- `Lock` 使用不当有什么风险？
  主要风险有：
  1. 死锁：多线程以不同顺序获取多把锁，彼此等待。
  2. 锁未释放：没有在 `finally` 中调用 `unlock()`，异常时可能永久占有锁。
  3. 活锁：线程不断重试或互相让出资源，但始终无法完成业务。
  4. 线程饥饿：非公平锁下，部分线程可能长期抢不到锁。
  5. 性能下降：锁粒度过大或锁竞争激烈，会导致阻塞、上下文切换增多，吞吐量下降。
  6. 条件变量误用：`await()`、`signal()` 未在持锁状态下调用，或条件判断不用 `while`，可能造成异常唤醒或逻辑错误。
- 为什么 `synchronized` 越来越常用？
  主要是因为 JDK 对 `synchronized` 的优化已经比较成熟了，比如锁升级、轻量级锁、自旋等机制，使它在多数普通同步场景下性能足够好。
  语法简单，异常时会自动释放监视器锁，不像 `Lock` 必须在 `finally` 中手动 `unlock()`，代码更不容易出错。

### synchronized 和 volatile 有什么区别？

`volatile` 只保证变量可见性和有序性，不保证复合操作原子性；`synchronized` 同时保证互斥、可见性和原子性。
读多写少且只需要状态标记时可以用 `volatile`，涉及临界区和复合操作时应使用锁或原子类。

#### 常见追问

- `volatile` 为什么禁止指令重排？
  `volatile` 通过内存屏障保证可见性和有序性，避免编译器或 CPU 将关键读写操作跨越 `volatile` 变量重排。
  例如，写线程先写普通变量，再写 `volatile` 标志位；读线程先读到标志位，再读取普通变量，就能保证读取到的是最新数据。
  它不是禁止所有指令重排，而是限制与 `volatile` 读写相关的重排，以建立 happens-before 关系。
- 单例双重检查为什么需要 `volatile`？
  主要防止 new Singleton() 的指令重排。对象创建可分为分配内存、初始化对象、赋值引用；如果“赋值引用”先于“初始化对象”，其他线程可能读到非空实例，但对象还没初始化完成。volatile 禁止这种重排，并保证实例创建完成后对其他线程可见。
- `AtomicInteger` 和 `volatile int` 有什么区别？
  `volatile int` 只保证可见性和有序性，适合状态标识，例如开关、停止信号；`count++` 仍然不是原子操作。
`AtomicInteger` 基于 CAS 实现原子更新，支持 `incrementAndGet()`、`compareAndSet()` 等读改写操作，适合并发计数和无锁更新。
可以理解为：`AtomicInteger` 内部的值也是 `volatile` 可见的，但额外提供了原子操作能力。

### 什么是死锁？怎么预防？

死锁是多个线程互相持有对方需要的锁，导致所有线程都无法继续执行。
预防方式包括固定加锁顺序、减少锁粒度、避免锁嵌套、使用 `tryLock()` 超时退出、缩短持锁时间和使用并发工具类替代手写锁。

#### 常见追问

- 死锁的四个必要条件是什么？
  **互斥**条件：资源同一时刻只能被一个线程占用。
  **请求与保持**条件：线程持有资源的同时继续请求其他资源。
  **不可剥夺**条件：已获得的资源不能被强制释放，只能主动释放。
  **循环等待**条件：多个线程形成首尾相接的资源等待环。
- 线上如何排查死锁？
  先通过监控确认接口超时、线程池活跃但无吞吐等现象，并保留故障现场。
  使用 `jps -l` 找到 Java 进程，再执行 `jstack -l <pid>`，或使用 `jcmd <pid> Thread.print -l` 导出线程栈。
  在线程栈末尾重点看 `Found one Java-level deadlock`，它会给出死锁线程、持有的锁和等待的锁。
  如果没有直接识别，就根据线程状态为 `BLOCKED`、`waiting to lock`、`locked` 的信息，手工串出锁依赖链路。
  必要时连续导出 2～3 次线程栈，确认线程长期不变化，再结合业务日志定位对应代码。
  应急上通常先重启实例恢复服务；后续统一锁顺序、缩小锁粒度，避免锁嵌套，并补充线程栈和死锁告警。
- 数据库死锁和 Java 死锁有什么区别？
  数据库死锁发生在多个事务互相等待行锁、表锁等数据库资源；Java 死锁发生在线程互相等待 `synchronized`、`ReentrantLock` 等 JVM 内锁。
  数据库通常能检测到死锁环，主动回滚其中一个事务并返回死锁异常；Java JVM 只能识别部分锁死锁，业务通常需要通过线程栈排查。
  数据库死锁影响的是事务执行，应用可以捕获异常后按策略重试；Java 死锁会导致相关线程永久阻塞，通常需要重启实例恢复。
  数据库侧主要通过缩短事务、统一更新顺序、建立合适索引来减少；Java 侧通过统一加锁顺序、避免嵌套锁和使用超时锁来避免。
- 避免嵌套锁能不能完全解决死锁？
  不能完全解决。避免嵌套锁能显著降低死锁概率，但死锁的根本是循环等待。
  即使没有明显的嵌套锁，多个线程在不同调用链中以相反顺序获取锁，仍可能形成死锁。
  例如线程 A 先拿锁 1 再拿锁 2，线程 B 先拿锁 2 再拿锁 1，就会死锁。
  采用统一锁获取顺序、缩小锁粒度，并对 `ReentrantLock` 使用 `tryLock` 超时机制兜底的方式避免死锁。

## ThreadLocal 与原子类

### ThreadLocal 是什么？适合什么场景？

`ThreadLocal` 为每个线程维护独立的变量槽位。它不会自动深拷贝 value；只有各线程存入的 value 本身不共享时，线程间操作才互不影响。
常见场景包括用户上下文、链路追踪 traceId、数据库连接、事务上下文等。使用线程池时必须注意清理，避免线程复用导致数据串号和内存泄漏。

#### 常见追问

- `ThreadLocalMap` 的 key 为什么是弱引用？
  因为 `ThreadLocalMap` 属于线程，线程通常生命周期较长；如果 key 是强引用，即使业务不再持有 `ThreadLocal`，它也会被 `ThreadLocalMap` 一直引用，无法回收。
  key 使用弱引用后，`ThreadLocal` 对象可以被 GC 回收，避免 key 本身长期泄漏。
  但 value 仍是强引用，key 被回收后会形成 `key=null` 的 Entry，在线程存活期间 value 仍可能泄漏。
  所以我会在使用完后主动调用 `ThreadLocal.remove()`，尤其在线程池场景。
- 为什么线程池中使用后要 `remove()`？
  线程池会复用工作线程，`ThreadLocalMap` 会跟随线程长期存在。
  如果不调用 `remove()`，当前请求存入的对象可能一直被该线程持有，导致内存泄漏。
  同时，下一个任务复用该线程时，还可能读到上一个任务遗留的数据，造成上下文串扰。
  因此在 `finally` 中调用 `remove()`，确保无论任务是否异常都能清理。
- `InheritableThreadLocal` 解决什么问题？
  用于让子线程创建时继承父线程的上下文，解决普通 `ThreadLocal` 不能在线程创建时自动传递值的问题，常用于传递用户信息、链路追踪 ID、租户标识等。
  它只在创建子线程时进行一次值传递。默认情况下，子线程继承的是同一个 `value` 引用，不是深拷贝；因此父线程后续重新 `set()` 新对象，子线程通常不可见，但如果 `value` 是可变对象，父线程修改其内部状态，子线程仍可能观察到变化。
  可以重写 `childValue()` 定制子线程继承的值，例如复制对象。
  线程池工作线程可能在首次提交任务时创建，也可能被预启动；无论何种方式，`InheritableThreadLocal` 都只在线程创建时继承一次，后续复用的每个任务不会自动重新继承上下文。应通过任务包装、显式参数传递或专用上下文传递工具处理。

### atomic 原理是什么？

JUC 的原子类主要基于 `volatile`、CAS 和底层 CPU 原子指令实现。`volatile` 保证共享变量的可见性和一定的内存有序性；CAS 通过比较内存值与预期值是否相等，相等则原子更新，否则更新失败。
以 `AtomicInteger.compareAndSet(expect, update)` 为例，单次 CAS 失败只会返回 `false`，不会自动重试。是否重试由外层算法决定；`getAndIncrement()` 这类方法只承诺原子的读改写语义，底层可能使用 CAS 重试或其他原子 CPU 指令，调用方不应依赖具体实现。
CAS 相比锁开销更低，但存在 ABA 问题、高竞争下自旋消耗 CPU，以及通常只能保证单个变量或单个内存位置原子更新等限制。JDK 8 中底层主要通过 `Unsafe` 调用 CAS；较新 JDK 也提供了 `VarHandle` 作为更标准的底层访问机制。

#### 常见追问

- CAS 有什么缺点？
  1. ABA 问题：值从 A 变为 B 又变回 A，CAS 只比较当前值时会认为值未变化。可通过版本号或 `AtomicStampedReference` 处理。
  2. 高竞争下 CPU 开销较大：单次 `compareAndSet()` 失败只返回 `false`；但使用 CAS 循环实现的原子读改写操作，在竞争激烈时可能频繁重试并消耗 CPU。
  3. 通常只能保证单个变量或单个内存位置的原子更新：多个变量需要保持一致性时，单独使用 CAS 不够，通常需要锁、不可变对象整体替换或更高层并发结构。
  4. 实现复杂度较高：需要处理 CAS 失败、重试策略、ABA、内存语义等问题，可读性通常不如锁。
- ABA 问题怎么解决？
  一般通过增加版本号解决：每次修改值时，同时递增版本号，CAS 比较“值 + 版本号”。
  即使值最终又从 B 变回 A，版本号也已经变化，CAS 就会失败。
  Java 中可以使用 `AtomicStampedReference`；如果只需标记是否变化，也可以使用 `AtomicMarkableReference`。
- `AtomicInteger` 和 `LongAdder` 有什么区别？
  `AtomicInteger` 基于单个值通过 CAS 更新，适合低并发、需要获取精确实时值的场景。
  `LongAdder` 将计数分散到多个 Cell 中，累加时降低 CAS 冲突，适合高并发统计场景。
  `LongAdder.sum()` 需要汇总各个 Cell，结果在并发更新期间不保证严格实时一致；`AtomicInteger.get()` 是当前精确值。
  `LongAdder` 不适合依赖旧值进行条件更新的场景，例如 `compareAndSet`，因为它不提供这类语义。

### AQS 是什么？

AQS 是 JUC 同步器的基础框架，核心是一个 `volatile int state` 和一个 CLH 等待队列。`ReentrantLock`、`Semaphore`、`CountDownLatch` 等都基于 AQS 实现。

#### 常见追问

- AQS 独占模式和共享模式有什么区别？
  独占模式下，同一时刻只允许一个线程获取同步状态，例如 `ReentrantLock`。获取失败的线程进入同步队列等待，当前线程释放后通常唤醒后继节点竞争锁。
  共享模式下，同一时刻允许多个线程获取同步状态，例如 `Semaphore`、`CountDownLatch`、`ReentrantReadWriteLock` 的读锁。线程获取共享资源后，如果剩余资源仍可用，会继续传播唤醒后续共享节点。
  **核心区别**是：独占模式强调“一个线程持有”，共享模式强调“多个线程可同时持有，以及资源是否还能继续分配”。
- `state` 在不同同步器里表示什么？
  `state` 是 AQS 中的 `volatile int` 同步状态字段，具体含义由不同同步器自行定义。
  在 `ReentrantLock` 中，`state` 表示独占锁重入次数，首次加锁为 1，重入递增，释放递减至 0。
  在 `Semaphore` 中，`state` 表示当前可用许可证数量。
  在 `CountDownLatch` 中，`state` 表示尚未完成的计数，减到 0 后等待线程可以继续执行。
  在读写锁中，通常用 `state` 的不同位分别记录读锁持有次数和写锁重入次数。
- AQS 为什么使用队列？
  为了管理获取同步状态失败的线程。线程获取锁失败后入队并 `park` 阻塞，避免持续自旋占用 CPU。
  同步队列以 FIFO 倾向组织等待与唤醒，释放同步状态时通常唤醒合适的后继节点。实际获取顺序和是否公平由具体同步器的获取策略决定；非公平同步器可能让新到线程抢在已排队线程之前获取同步状态。
  队列节点还记录线程的等待状态，便于处理取消、中断、超时等场景。
  本质上，队列把“竞争失败的线程”集中管理，实现高效的阻塞、唤醒和调度。
