# JVM 高频题

本页整理 JVM 面试高频问题，覆盖 JVM 组成、运行时数据区、类加载、垃圾回收、对象晋升和调优工具。回答时要能画出内存结构，并结合线上排查说明工具和参数。

## JVM 组成

### JDK、JRE、JVM 有什么关系？

JVM 是 Java 程序运行的虚拟机，负责执行字节码、管理内存和垃圾回收；
JRE 是 Java 运行环境，包含 JVM 和运行时类库；
JDK 是 Java 开发工具包，包含 JRE 以及编译、调试、打包等开发工具。
开发 Java 程序需要 JDK，运行 Java 程序通常只需要 JRE 或对应运行时环境。

#### 常见追问

- 只安装 JRE 能不能编译 Java 程序？
  只安装 JRE 一般不能编译 Java 程序。 JRE 主要是 Java 运行环境，包含 JVM 和运行 Java 程序需要的核心类库，作用是运行已经编译好的 .class 或 jar。
- JDK 9 之后 JRE 目录为什么不明显了？
  JDK 9 之后因为模块化和 jlink（可以按需裁剪模块，生成一个定制化的运行时环境） 的出现，JRE 不再以以前那种单独目录形式存在，JDK 本身就包含运行 Java 程序需要的能力。
- JVM 是否只支持 Java 语言？
  JVM 不只支持 Java 语言。JVM 只识别字节码，也就是 .class 文件，只要一门语言能被编译成符合 JVM 规范的字节码，就可以运行在 JVM 上。
  常见的 JVM 语言有 Kotlin、Scala、Groovy、Clojure，包括一些 JRuby、Jython 这类实现。

### JVM 主要由哪些部分组成？

JVM 主要由类加载子系统、运行时数据区、执行引擎和本地方法接口组成。
类加载器负责加载 class；
运行时数据区负责存储对象、栈帧、类元信息等；
执行引擎负责解释执行或 JIT 编译；
本地方法接口用于调用底层 native 方法。

#### 常见追问

- 类加载器有哪些？
  **启动类加载器** Bootstrap ClassLoader，它负责加载 Java 核心类库，比如 java.lang.*、java.util.* 这些，通常是由 JVM 底层实现的，不是普通 Java 类。
  **扩展类加载器** Extension ClassLoader，在 JDK 8 及以前主要负责加载 jre/lib/ext 目录下的类；不过 JDK 9 之后因为模块化，扩展机制被移除了，对应变成了 Platform ClassLoader。
  **应用程序类加载器** Application ClassLoader，也叫系统类加载器，负责加载 classpath 下的业务类和第三方依赖，这是我们平时开发最常接触的。
  **自定义类加载器**，继承 ClassLoader 实现自己的加载逻辑，比如热部署、插件化、隔离不同应用依赖等场景。
- 本地方法栈存什么？
  本地方法栈主要是为 JVM 调用 Native 方法服务的，也就是用 native 关键字修饰、底层一般由 C/C++ 实现的方法。
  里面保存的是 Native 方法执行时需要的信息，比如本地方法的调用栈帧、局部变量、参数、返回值相关信息，以及调用本地库时的一些运行状态。

### 什么是 JIT？

JIT 是即时编译器，会在程序运行过程中把热点字节码编译成本地机器码，提高执行效率。解释执行启动快但长期运行性能较低，JIT 可以根据运行时统计信息进行方法内联、逃逸分析、锁消除等优化。服务端应用通常运行越久，热点代码越容易被优化。

#### 常见追问

- 什么是热点代码？
  热点代码就是程序运行过程中被频繁执行的代码。一种是被多次调用的方法，另一种是被多次执行的循环体。
- JIT 和 javac 有什么区别？
  javac 负责让 Java 代码变成 JVM 能识别的字节码，JIT 负责让热点字节码在运行时跑得更快。
- 为什么 Java 有预热过程？
  主要是因为 JVM 一开始执行的是字节码解释执行，性能不是最优。
   JVM 从解释执行逐步过渡到热点代码 JIT 编译优化，再加上应用资源初始化共同导致的。

### JVM 内存结构是什么？

JVM 运行时内存主要包括堆、虚拟机栈、本地方法栈、程序计数器和方法区 / 元空间。
堆是对象分配和 GC 的主要区域；
栈保存方法调用栈帧；
程序计数器记录当前线程执行位置；
元空间存放类元信息。

![JVM 运行时数据区](/images/interview/jvm/jvm-runtime-data-areas.webp)

上图展示了 JVM 运行时数据区的核心组成。面试回答时要能区分线程共享区域和线程私有区域：堆和方法区 / 元空间通常是线程共享的，虚拟机栈、本地方法栈和程序计数器是线程私有的。

#### 常见追问

- 哪些区域线程共享？哪些线程私有？
  JVM 运行时数据区里，线程共享的主要是堆和方法区。
  线程私有的主要是虚拟机栈、本地方法栈和程序计数器。
- OOM（内存溢出） 可能发生在哪些区域？
  堆、方法区/元空间、虚拟机栈、本地方法栈、直接内存都可能发生 OOM，程序计数器是唯一一个规范里没有规定 OOM 的区域。
- 栈溢出和堆溢出有什么区别？
  第一，发生区域不同，栈溢出在虚拟机栈，堆溢出在堆；
  第二，原因不同，栈溢出通常是调用太深，堆溢出通常是对象太多或内存泄漏；
  第三，影响范围不同，栈是线程私有的，通常影响当前线程，堆是共享的，严重时会影响整个应用。

### 堆和栈有什么区别？

堆是线程共享区域，主要存放对象实例和数组，是 GC 管理的核心区域；
虚拟机栈是线程私有区域，存放方法调用栈帧，包括局部变量表、操作数栈、动态链接和返回地址。
堆溢出通常是对象太多或泄漏，栈溢出通常是调用层级过深或栈帧过大。

#### 常见追问

- 局部变量一定在栈上吗？
  不一定。局部变量的引用或基本类型值通常在栈帧里，但局部变量指向的对象不一定在栈上，通常在堆上；经过 JVM 优化后，也可能不实际分配到堆上。
- 对象一定在堆上吗？
  不一定，但通常可以这么理解：Java 对象大多数情况下是分配在堆上的，因为堆是 JVM 管理对象实例的主要区域。
  不过从 JVM 优化角度看，如果 JIT 通过逃逸分析发现一个对象不会逃逸出方法或线程，就可能进行标量替换，把对象拆成几个字段，放在栈帧或寄存器里，甚至对象本身都不真正分配出来。
- 什么是逃逸分析？
  逃逸分析是 JVM 的一种优化手段，主要是分析一个对象的作用范围，判断它会不会逃出当前方法或者当前线程。
  一个对象只在方法内部使用，没有被返回、没有赋值给成员变量、没有传给外部方法，那它就是没有逃逸，JIT 就可能做优化。
  常见优化有三个：
    第一是栈上分配或直接不分配到堆上；
    第二是标量替换，把对象拆成几个基本字段来处理；
    第三是锁消除，如果对象不会被其他线程访问，相关同步锁可能被去掉。

### 队列和栈有什么区别？

队列是先进先出 FIFO，常用于任务排队、消息缓冲和线程池任务队列；栈是后进先出 LIFO，常用于方法调用、表达式计算和回溯场景。JVM 虚拟机栈体现的就是方法调用的入栈和出栈过程。

#### 常见追问

- 递归为什么容易栈溢出？
  因为每调用一次方法，JVM 都会创建一个新的栈帧，里面保存局部变量、操作数栈、返回地址等信息。
  递归如果层数太深，就会不断往当前线程的虚拟机栈里压入栈帧，当栈空间不够用时，就会抛出 StackOverflowError。
  写递归时注意两个点：一是必须有明确的递归终止条件，二是要评估递归深度，如果深度很大，就考虑改成循环或者用显式栈来模拟递归。
- 阻塞队列适合什么场景？
  阻塞队列比较适合生产者-消费者场景。
- 栈和堆是数据结构还是内存区域？
  从数据结构角度看，栈是一种后进先出的数据结构，堆是一种特殊的树形结构，常见的是大顶堆、小顶堆，通常用来实现优先队列。
  从 JVM 内存角度看，栈和堆是运行时内存区域。Java 里每个线程有自己的虚拟机栈，方法调用会创建栈帧；堆是线程共享的，主要用来存放对象实例和数组，也是 GC 主要管理的区域。
  如果问 Java 内存，一般说的是内存区域；如果问算法或集合，一般说的是数据结构。两者名字一样，但含义不一样。

## 类加载

### 什么是类加载器？

类加载器负责把 class 字节码加载到 JVM，并生成对应的 Class 对象。常见类加载器包括启动类加载器、平台类加载器和应用类加载器，也可以自定义类加载器实现隔离、热部署或插件化。类加载器不仅影响类从哪里加载，也影响类的唯一性判断。

#### 常见追问

- 两个类名相同的类一定是同一个类吗？
  类的唯一性由“类的全限定名 + 类加载器”共同决定。类名相同不一定是同一个类，只有类名相同并且由同一个类加载器加载，才算是同一个类。
- 类加载器为什么需要层级结构？
  主要是为了配合双亲委派机制。本质上是为了保证类加载的安全性、一致性，同时又保留一定的扩展能力。（避免核心类被重复加载，防止用户自己写一个同名的 java.lang.String 替换 JDK 核心类，不同层级加载不同范围的类，方便实现隔离和扩展）
- 自定义类加载器适合哪些场景？
  适合需要“隔离、扩展、动态加载”的场景。自定义类加载器不是日常业务开发常用的东西，更多是在容器、框架、插件化、热部署、类隔离这些底层或架构场景里使用。

### 双亲委派模型是什么？

类加载请求会先委派给父加载器，父加载器无法加载时才由子加载器处理。这样可以避免核心类被篡改，也能避免类重复加载。

#### 常见追问

- 双亲委派如何保证核心类安全？
  可以保证核心类优先由更高层级的类加载器加载，避免核心类被伪造、篡改或重复加载，从而保证 Java 基础类库的安全性和一致性。
- 什么场景会打破双亲委派？
  主要是为了实现 SPI 扩展、类隔离、版本隔离、热部署和动态加载，但核心类一般还是不能随便打破，否则会有安全风险。
- Tomcat 为什么要自定义类加载器？
  Tomcat 自定义类加载器主要是为了类隔离、版本隔离、热部署，以及保护容器自身类库不被 Web 应用影响。

### 静态类加载和动态类加载有什么区别？

静态类加载通常指依赖在编译期已经明确，例如 new Xxx()、静态字段/方法引用；但真正加载、链接、初始化仍发生在运行期。动态类加载是运行期根据类名、配置或插件机制通过 Class.forName() / ClassLoader.loadClass() 加载类。

#### 常见追问

- `Class.forName()` 会触发类初始化吗？
  会，默认情况下会触发类初始化。Class.forName(String) 会触发初始化，Class.forName(String, false, loader) 不会，是否初始化主要看第二个 initialize 参数。
- `ClassLoader.loadClass()` 和 `Class.forName()` 有什么区别？
  Class.forName() 更偏向“加载并可初始化类”，常用于反射和 JDBC 驱动加载；ClassLoader.loadClass() 更偏向“通过指定类加载器加载类”，常用于自定义类加载器、容器和插件化场景。
- 动态类加载有哪些风险？
  动态类加载一定要控制来源、隔离 ClassLoader、避免频繁加载，并且做好卸载清理，否则容易带来安全漏洞、类冲突、内存泄漏和性能问题。

### 类加载过程是什么？

类加载过程包括加载、验证、准备、解析和初始化。
加载是读取 class 字节码并生成 Class 对象；验证保证字节码安全；准备为静态变量分配内存并设置零值；解析把符号引用转成直接引用；初始化执行静态变量赋值和静态代码块。

#### 常见追问

- 准备阶段和初始化阶段有什么区别？
  准备阶段是 JVM 分配静态变量内存并赋默认值，初始化阶段是执行程序员写的静态赋值和静态代码块。
- 静态代码块什么时候执行？
  静态代码块在类第一次主动使用、触发初始化时执行，只执行一次，执行顺序是父类静态代码块先于子类静态代码块。
- 类什么时候会被初始化？
  类在第一次被主动使用时初始化，核心就是 new 对象、访问静态成员、调用静态方法、反射初始化、初始化子类前先初始化父类。

## 对象回收与引用

### 如何判断对象是否可回收？

主流 JVM 使用可达性分析判断对象是否可回收。从 GC Roots 出发，如果对象无法通过引用链到达，就认为对象不可达，可以被回收。引用计数法简单但无法解决循环引用问题，因此 HotSpot 不依赖引用计数作为主要判断方式。

#### 常见追问

- 哪些对象可以作为 GC Roots？
  线程栈中的引用、静态变量引用、常量引用、JNI 引用、被锁持有的对象、JVM 内部引用和活跃线程对象。
- 循环引用为什么不是问题？
  循环引用对引用计数法是问题，但 HotSpot 主流 GC 使用可达性分析。从 GC Roots 出发，如果一组互相引用的对象整体不可达，即使它们互相引用，也会被判定为可回收。
- `finalize()` 为什么不推荐？
  执行依赖 GC，调用时机不确定，甚至可能永远不执行，因此不能可靠地释放资源。带有 finalize() 的对象回收成本更高，会延长对象生命周期，影响 GC 性能。同时它还可能导致对象复活、异常被吞掉、执行顺序不可控以及安全风险。
  现在推荐使用 try-with-resources、AutoCloseable 或 Cleaner 来替代 finalize()。
  Java 9 之后，Object.finalize() 已经被标记为废弃。

### Java 引用类型有哪些？

Java 引用类型包括强引用、软引用、弱引用和虚引用。
强引用只要存在就不会被回收；
软引用适合缓存，内存不足时可能被回收；
弱引用只要发生 GC 就可能被回收；
虚引用不能获取对象，主要用于跟踪对象回收和管理堆外资源。

#### 常见追问

- `WeakHashMap` 使用什么引用？
   key 使用的是弱引用，value 还是强引用。当某个 key 除了 WeakHashMap 里的弱引用之外，没有其他强引用指向它时，下一次 GC 就可能回收这个 key。key 被回收后，对应的 entry 也会在后续访问 WeakHashMap 时被清理掉。
   常用于缓存这类场景：希望 key 没有外部强引用时，缓存项可以自动被 GC 回收。
- 软引用适合做缓存吗？
  软引用可以做缓存，但不推荐作为主要缓存方案。因为指向的对象在内存充足时可以保留，内存不足、发生 GC 时可能被回收，回收时机由 JVM 和 GC 决定，不可控，可能导致缓存命中率不稳定；而且内存压力大时大量软引用被清理，也可能带来性能抖动。
  更推荐用 Caffeine、Guava Cache 这类成熟缓存，设置最大容量、过期时间、淘汰策略，这样行为更可控。
- 虚引用和 `ReferenceQueue` 有什么关系？
  虚引用必须配合ReferenceQueue使用，它本身通过 get() 永远拿不到对象，返回的一直是 null。
  虚引用负责“关联目标对象”，ReferenceQueue 负责“接收对象可回收的通知”。虚引用不配队列基本没有实际意义。

### 对象什么时候进入老年代？

对象通常先分配在 Eden 区，经历 Minor GC 后仍然存活会进入 Survivor 区，多次存活达到年龄阈值后晋升老年代。大对象也可能直接进入老年代。如果 Survivor 空间不足，部分对象也可能提前晋升。

#### 常见追问
- JVM中的堆一般有几代？
  从常见分代堆看，主要是新生代和老年代；
    新生代又分 Eden、From Survivor、To Survivor。
    JDK 8 以后类元信息主要放在 Metaspace，不再是永久代。
- Minor GC 和 Full GC 有什么区别？
  **Minor GC**：主要回收新生代，一般是 Eden 空间满了触发，回收速度相对快，停顿时间通常比较短。Minor GC 范围小、频率高、耗时短。
  **Full GC**：是回收整个堆，通常包括新生代、老年代，有时还包括元空间，触发原因可能是老年代空间不足、元空间不足、显式调用 System.gc()、分配担保失败等。Full GC 范围大、频率低但代价高，通常会带来比较明显的 Stop The World。
- 老年代满了会发生什么？
  通常会先触发一次 Major GC 或 Full GC，尝试回收老年代里的无用对象，这个过程一般会发生 STW，应用线程会暂停。
  GC 后空间还是不够，比如对象确实都还被引用，或者存在内存泄漏，那后续对象晋升或大对象分配就会失败，最终抛出 OutOfMemoryError: Java heap space。
- 为什么频繁 Full GC 很危险？
  GC 期间业务线程暂停，请求处理不了，接口延迟会明显升高，严重时会超时。
  回收范围大，成本高，如果频繁发生，说明 JVM 一直在忙着回收内存，CPU 会被 GC 占用，业务吞吐量会下降。

## 垃圾回收算法与收集器

### JVM 垃圾回收算法有哪些？

常见算法包括标记-清除、标记-整理、复制算法和分代收集。
  标记-清除容易产生碎片；
  标记-整理会移动对象减少碎片；
  复制算法适合存活对象少的新生代；
  分代收集根据对象生命周期把堆划分为新生代和老年代，采用不同回收策略。

#### 常见追问

- 新生代为什么适合复制算法？
  存活对象少、复制成本低，并且能避免碎片，提高对象分配效率。
- 老年代为什么不适合单纯复制算法？
  老年代对象存活率高。如果每次 GC 都把大量对象复制到另一块区域，复制成本会非常高，GC 时间会变长。
  老年代一般更适合标记-清除、标记-整理这类算法。
- 什么是内存碎片？
  堆里空闲内存不是一整块连续空间，而是被分散成很多小块。

### 常见垃圾回收器有哪些？

常见收集器包括 Serial、Serial Old、ParNew、Parallel     Scavenge、Parallel Old、CMS 和 G1。 ZGC、Shenandoah。
  Serial 偏单线程；
  Parallel 偏吞吐量；
  CMS 偏低停顿但会产生碎片且已废弃；
  G1 以 Region 为单位回收，兼顾吞吐和可预测停顿。

#### 常见追问

- 吞吐量优先和低延迟优先怎么选？
  能接受较长停顿、追求整体处理能力，选吞吐量优先；对响应时间敏感、不能有明显卡顿，选低延迟优先。
- CMS 为什么被废弃？
  因为碎片、退化 Full GC、维护成本和可扩展性问题，逐渐被 G1、ZGC 这些收集器替代了。
- G1 为什么适合大堆？
  因为它不是按传统的新生代、老年代整块去回收，而是把堆切成很多个 Region。
  优先挑垃圾最多、回收收益最高的 Region 来处理；以设置期望停顿时间，比如 -XX:MaxGCPauseMillis，它会根据目标停顿时间尽量选择合适数量的 Region 回收，停顿时间更可控；基于复制和整理算法，能减少内存碎片。

### CMS 垃圾回收器有什么特点？

CMS 以获取最短回收停顿为目标，主要阶段包括初始标记、并发标记、重新标记和并发清除。它使用标记-清除算法，停顿较短，但会产生内存碎片，也可能出现 Concurrent Mode Failure。CMS 在新版本 JDK 中已经被废弃。

#### 常见追问

- CMS 哪些阶段会 STW（ Stop-The-World： JVM 暂停业务线程做 GC。）？
  初始标记和重新标记。
  如果发生 Concurrent Mode Failure（ CMS 并发回收赶不上内存分配），CMS 会退化成 Full GC，那就是一次很重的 STW。
- CMS 为什么会产生碎片？
  因为它使用的是“标记-清除”算法。
- Concurrent Mode Failure 是什么？
  CMS 回收速度赶不上内存分配或对象晋升速度，老年代提前被占满，导致并发回收失败。

### G1 和 CMS 有什么区别？

CMS 以降低老年代停顿为目标，但会产生内存碎片，且已经废弃。
G1 将堆划分为多个 Region，可以按收益优先回收，并支持可预测停顿目标，是现代服务端应用的通用选择。

#### 常见追问

- G1 的 Region 有什么好处？
  把大堆拆小块，按价值回收，停顿可控，空间更灵活，也能减少碎片。
- Humongous Object 是什么？
  是 G1 里的大对象，大小超过一个 Region 容量 50% 的对象。
  G1 为了处理超大对象设计的一种特殊对象分配方式，但如果系统里频繁产生大数组、大字符串、大 byte[]，就要特别关注它对 GC 的影响。
- G1 适合所有场景吗？
  面向大堆、低停顿、可预测停顿的场景，比如几 GB 到几十 GB 堆内存的后端服务。

### 分代垃圾回收怎么工作？

新生代通常分为 Eden、From Survivor 和 To Survivor，默认比例常见为 8:1:1。对象优先分配到 Eden，Minor GC 时存活对象复制到 Survivor，年龄加 1，达到阈值后晋升老年代。
老年代对象生命周期更长，空间不足时可能触发 Major GC 或 Full GC。

#### 常见追问

- Survivor 区为什么要有两个？
  为了让新生代复制回收更高效、无碎片，同时降低对象过早晋升到老年代的概率。
- 对象年龄默认阈值是多少？
  默认晋升阈值一般是 15，也就是通过参数 -XX:MaxTenuringThreshold 控制。
  对象在 Survivor 区每熬过一次 Minor GC，年龄就加 1，达到这个阈值后就会晋升到老年代。
  对象年龄最大就是 15，因为对象头里用于记录年龄的空间只有 4 bit。
- 大对象为什么可能直接进入老年代？
   JVM 为了避免新生代复制成本高、Survivor 放不下、频繁晋升带来的额外开销。

## JVM 调优与排查

### 什么是 Java 内存泄漏？

Java 内存泄漏不是对象无法释放，而是对象已经不再被业务需要，却仍然被引用链持有，导致 GC 无法回收。常见原因包括静态集合长期持有对象、监听器未注销、ThreadLocal 未清理、缓存无淘汰策略等。排查时通常结合堆 dump、GC 日志和引用链分析。

#### 常见追问

- Java 有 GC 为什么还会内存泄漏？
   Java 有 GC 只能解决“不可达对象”的自动回收，解决不了“还可达但已经不用了”的对象。
- ThreadLocal 为什么可能导致泄漏？
  泄漏主要跟 ThreadLocalMap 的结构有关。
  每个线程内部都有一个 ThreadLocalMap，key 是 ThreadLocal 的弱引用，value 是我们存进去的对象的强引用。如果 ThreadLocal 对象本身没有外部强引用了，GC 后 key 会被回收，变成 null；但 value 还被 ThreadLocalMap 强引用着。在线程池场景下，线程是长期存活的，所以这个 ThreadLocalMap 也长期存在，value 就可能一直释放不了，造成内存泄漏。
  所以使用 ThreadLocal，尤其在线程池里，一般要在 finally 里调用 remove()，避免 value 残留在线程上。
- 如何通过 MAT 分析泄漏对象？
  先看谁占内存大，再看谁持有它，最后通过 GC Roots 引用链定位是哪段代码没有释放引用。

### Full GC 频繁怎么排查？

先确认 GC 日志和监控指标，判断是老年代压力、内存泄漏、大对象分配还是元空间问题。然后结合 `jstat`、堆转储、MAT / JProfiler 分析对象占用和引用链。

#### 常见追问

- CPU 飙高怎么定位？
  先定位高 CPU 进程，再定位高 CPU 线程，再把线程 ID 对应到 jstack 的 nid，最后根据线程栈判断是业务死循环、频繁 GC、锁竞争还是第三方组件问题。
- OOM 后现场如何保留？
  提前配好 heap dump、GC 日志、错误日志，必要时加 OOM 回调脚本，这样 OOM 后才能根据现场分析，而不是只能猜。
- 为什么生产要开启 HeapDumpOnOutOfMemoryError？
  可以把 OOM 现场留下来，方便后续做根因分析。

### JVM 调优工具有哪些？

常用工具包括 `jps`、`jstack`、`jmap`、`jstat`、`jcmd`、JConsole、VisualVM、MAT、Arthas 和商业分析工具。面试中可以按问题分类回答：
  线程问题看 `jstack`，
  内存问题看 `jmap` 和 MAT，
  GC 趋势看 `jstat` 和 GC 日志。

#### 常见追问

- 如何排查线程死锁？
  先看现象，比如接口一直不返回、线程池任务堆积、CPU 不一定高，但吞吐明显下降。
  然后先拿到 Java 进程 pid，用 jstack 连续抓几次线程栈，如果存在 Java 层面的死锁，jstack 通常会直接在末尾打印 Found one Java-level deadlock，并列出哪些线程互相持有哪些锁、等待哪些锁。
  重点看线程状态是 BLOCKED 的线程，看它们卡在哪个 synchronized 或 Lock 上，再看 waiting to lock 和 locked 这两类信息，判断是不是 A 线程持有 lock1 等 lock2，B 线程持有 lock2 等 lock1。
  线上允许，也会用 Arthas，直接找出阻塞其他线程的线程，对排查锁竞争和死锁比较方便。
  如果是 ReentrantLock 这类锁，jstack -l 能看到 ownable synchronizers，有助于判断锁被哪个线程持有。
  定位到代码后，常见解决方式是统一加锁顺序，避免嵌套锁，减少锁粒度，或者用 tryLock 加超时兜底，避免无限等待。
  另外连续抓 2 到 3 次栈，如果线程一直卡在同一批锁等待关系上，基本就能确认不是短暂锁竞争，而是真死锁。
- 如何分析堆 dump？
  一般会用 Eclipse MAT 或者 VisualVM、JProfiler 来分析 heap dump，线上生成的 hprof 文件先下载到本地，注意文件可能很大，本地 MAT 的内存也要调大。分析时我不会一上来就看对象数量，而是先看 Leak Suspects Report，快速判断有没有明显泄漏嫌疑。然后重点看 Dominator Tree，按 Retained Heap 排序，因为 Retained Heap 表示这个对象被回收后能释放多少内存，更适合定位真正占内存的根对象。接着会看大对象是谁，比如大的 HashMap、ArrayList、byte[]、char[]、缓存对象、业务 DTO 集合等，再通过 Path To GC Roots 看它为什么没被回收，是被静态变量、单例、线程、ThreadLocal、缓存、连接会话还是类加载器引用住了。如果是集合类占用大，我会继续看里面存的 key/value 类型和数量，判断是不是缓存无上限、查询一次加载太多数据、消息堆积或者 ThreadLocal 没 remove。还会结合 GC 日志和 OOM 时间点判断是内存泄漏还是瞬时大对象，比如如果老年代持续上涨不回落，通常偏泄漏；如果某个请求或任务突然申请大数组，可能是瞬时内存峰值。最后会把 dump 里的引用链对应到代码，比如哪个静态 Map、哪个本地缓存、哪个线程池任务队列持有对象，然后再改代码，比如加缓存上限和过期策略、分页处理、及时关闭资源、ThreadLocal remove，或者调整 JVM 内存参数。
- Arthas 常用命令有哪些？
  **dashboard**：看 JVM 实时概况，比如线程、内存、GC、运行时信息，适合刚连上去先看整体状态。
  **thread**：看线程情况，比如 thread 查看线程列表，thread -n 5 看 CPU 最高的几个线程，thread 线程ID 看具体栈，thread -b 查找阻塞其他线程的线程。
  **jvm**：查看 JVM 参数、内存区域、GC、线程、系统属性等信息。
  **sysprop / sysenv**：查看系统属性和环境变量，比如确认启动参数、配置路径、环境变量是否符合预期。
  **sc**：搜索 JVM 已加载的类，比如 sc -d com.xxx.UserService 看类是哪个 classloader 加载的、字段和类信息。
  **sm**：查看类的方法，比如 sm com.xxx.UserService 可以看这个类有哪些方法。
  **jad**：反编译线上实际加载的类，确认运行的代码版本是不是预期版本。
  **watch**：观察方法入参、返回值、异常，比如 watch com.xxx.UserService getUser '{params, returnObj, throwExp}' -x 3。
  **trace**：看方法内部调用耗时，适合定位接口慢在哪个子调用上，比如 trace com.xxx.UserService getUser。
  **stack**：查看某个方法是被谁调用的，适合排查调用链来源，比如 stack com.xxx.UserService getUser。
  **monitor**：统计方法调用次数、成功失败次数和平均耗时，比如 monitor -c 5 com.xxx.UserService getUser。
  **tt**：记录方法调用现场，可以之后 replay 或查看入参返回值，适合排查偶发问题，但生产要谨慎控制范围。
  **profiler**：做火焰图分析 CPU 热点，比如 profiler start，过一会 profiler stop。
  **heapdump**：在线导出堆 dump，用于后续 MAT 分析。
  **ognl**：执行 OGNL 表达式，读取 Spring Bean 或静态变量，生产要非常谨慎。
  **classloader**：查看类加载器信息，有时排查类冲突会用。
  **vmoption**：查看或修改 JVM 参数，比如临时打开 PrintGC 相关参数或者查看 HeapDumpOnOutOfMemoryError。
  **logger**：查看和修改日志级别，线上临时把某个包调成 DEBUG 比较方便。
  *实际排查时一般是先 dashboard、thread、jvm 看整体状态，再根据问题类型选择 trace、watch、stack、monitor；如果是 CPU 高就 thread -n 配合 profiler，如果是内存问题就 heapdump，如果是代码版本或类冲突就 sc、jad、classloader。*

### 常见 JVM 参数有哪些？

常见参数包括 
`-Xms` 初始堆大小、
`-Xmx` 最大堆大小、
`-Xss` 线程栈大小、
`-XX:NewRatio` 新生代与老年代比例、
`-XX:SurvivorRatio` Eden 与 Survivor 比例、
`-XX:+UseG1GC` 指定 G1、
`-XX:+PrintGCDetails` 或新版 `-Xlog:gc*` 打印 GC 日志。

#### 常见追问

- `-Xms` 和 `-Xmx` 为什么建议设置一致？
  为了避免 JVM 在运行过程中动态扩容或收缩堆带来的性能抖动。
- JDK 9 之后 GC 日志参数有什么变化？
  JDK 9 之后 GC 日志改成了统一日志框架，也就是 -Xlog，以前 JDK 8 常用的 -XX:+PrintGCDetails、-XX:+PrintGCDateStamps、-Xloggc:/path/gc.log 这类参数就不推荐用了，部分参数会被废弃或者提示 warning。
- 如何选择垃圾回收器？
  看业务对吞吐、延迟、堆大小和 JDK 版本的要求。
  一般服务选 G1，吞吐优先选 Parallel，低延迟大堆选 ZGC，小内存简单程序选 Serial。
